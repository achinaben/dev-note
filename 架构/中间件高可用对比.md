## 分布式中间件高可用与写入机制对比

> 目标：用同一套问题（分片/主从/日志/刷盘/复制/确认）看懂 Kafka、RocketMQ、MongoDB、Elasticsearch 的“高可用与写入可靠性”差异。  
> 使用方式：先看第 6 章“通用分析模板”与第 7 章“一页纸速记”，面试/复盘都够用；需要细节再回看各组件章节。

### 一、Kafka

#### 1. 集群结构

- **分区级主从**：每个 `partition` 有 1 个 `leader`，若干 `follower`
- **没有全局主节点**：按分区选 leader，而不是整个集群只有一个 master

#### 2. 分区与副本

- `topic → partitions`
- 每个 partition 有 `replication factor`（如 3 副本）
- 副本分布在不同 broker 节点上，避免单机故障

#### 3. 写入与确认（Producer）

- `acks=0`：发送即成功，不等任何结果（最快，最不安全）
- `acks=1`：leader 写成功即返回（默认，能容忍少量数据丢失）
- `acks=all/-1`：**leader + 所有 ISR 副本写成功才返回**（最高可靠）

#### 4. 消费与偏移量（Offset）

- offset 记录消费进度，可以自动提交，也可以**手动提交**
- 常见做法：**先处理消息，确认没问题后再手动提交 offset**，保证“至少一次”

#### 5. 高可靠典型配置

- 副本数 `replication.factor ≥ 3`
- Producer：`acks=all`，broker 设置合理的 `min.insync.replicas`
- Consumer：关闭自动提交，改为**手动提交 offset**

### 二、RocketMQ

#### 1. 集群结构

- **Broker 主从（Master/Slave）**
- NameServer 提供路由和元数据，不存业务消息

#### 2. 分区方式

- `topic → 多个 Message Queue`
- 可类比为“分区”，分布在不同 broker 上，实现负载均衡和容灾

#### 3. 刷盘策略

- **同步刷盘**：
  - 写入内存 + 写磁盘（fsync）成功才返回
  - 延迟较大，可靠性高
- **异步刷盘**：
  - 写入内存即返回，由后台线程批量刷盘
  - 性能好，可靠性略差

#### 4. 主从复制策略

- **同步复制**：
  - Master 写成功后，还要等 Slave 写成功才返回
- **异步复制**：
  - Master 写成功即返回，Slave 异步跟进

#### 5. 高可靠典型配置

- Broker 主从部署
- 开启 **同步刷盘 + 同步复制**
- 以牺牲部分性能换取尽可能不丢消息

### 三、MongoDB（Replica Set）

#### 1. 集群结构

- **Replica Set**：1 个 Primary + 多个 Secondary
- 建议节点数 **≥3**，保证有多数派用于选举和写入确认

#### 2. 写入持久化路径

- **Journal（预写日志）**
  - 写数据前先写入 journal（顺序写，易刷盘）
  - 配置 `j: true` 时，必须写 journal + fsync 成功才算成功
- **Checkpoint / 快照**
  - 后台周期性执行 checkpoint，把内存脏页刷到数据文件
  - 即使宕机，也能从最新 checkpoint + journal 回放恢复

#### 3. 写关注（Write Concern）

- `w:1`：Primary 写成功即返回
- `w:majority`：集群**多数节点写成功**才返回
- `j:true`：必须写入 journal 并刷盘

#### 4. 高可靠典型配置

- Replica Set 成员数 **≥3**
- 使用：
  - `writeConcern: { w: "majority", j: true }`
- 兼顾：
  - 不丢数据
  - 宕机可自动选主
  - 崩溃可通过 journal 恢复

### 四、Elasticsearch（ES）

#### 1. 集群结构

- 存在 **master 节点**：
  - 负责集群元数据管理、索引/分片分配等
- 数据层面：
  - 每个 index 拆成多个 **primary shard**
  - 每个 primary shard 有若干 **replica shard**

#### 2. 分片与副本

- `index → primary shards + replica shards`
- replica shards 分布在不同节点
- 确保单节点宕机时，数据依然有副本可用

#### 3. 写入流程（简化）

1. 客户端请求 → 协调节点
2. 协调节点将请求路由到目标 shard 的 **primary**
3. primary 写入内存 buffer + translog
4. primary 将操作转发给所有 replica shard
5. primary 与 replica 成功后，向客户端返回成功（细节可配置）

#### 4. 刷盘与日志

- 写入先记录到 **translog（事务日志）** 和内存
- translog 按策略 fsync 到磁盘（保证崩溃可恢复）
- 后台再将内存数据刷新为 segment 文件供查询
- 客户端视角：更像是“同步写日志 + 同步复制，数据文件异步整理”

#### 5. 高可靠典型配置

- 为每个 index 设置：
  - `number_of_shards` 合理
  - `number_of_replicas ≥ 1`
- 确保：
  - primary 与 replica 分布在不同节点
- 合理配置：
  - translog fsync 策略
  - `refresh_interval` 等参数

### 五、通用分析模板：分布式系统高可用的 6 个关键问题

#### 1. 分片 / 分区（Sharding / Partition）

- 问自己：**数据/请求是如何被“切块”的？**
  - Kafka：`topic → partitions`
  - RocketMQ：`topic → message queues`
  - ES：`index → primary shards`
  - MongoDB 分片集群：`collection → chunks`
- 作用：决定负载如何分摊，单机故障影响的范围有多大。

#### 2. 主从结构（Leader/Follower 或 Master/Slave）

- 问自己：**在哪个层级存在“主从/主备”？**
  - 集群层面是否有 master 节点？
  - 分片/分区层面是否有 leader/follower？
- 例子：
  - Kafka：分区级 leader/follower，无全局主
  - RocketMQ：Broker Master/Slave
  - MongoDB：Replica Set Primary/Secondary，多数派选举
  - ES：集群 master + 每个 shard 的 primary/replica

#### 3. 日志机制（预写日志 / 事务日志）

- 问自己：**写数据前是否先写日志？**
  - WAL / Journal / Translog
  - 日志写成功是否是“成功”的关键步骤？
- 例子：
  - Kafka：分区日志天然类似 WAL
  - RocketMQ：commitlog 等顺序写文件
  - MongoDB：Journal
  - ES：Translog

#### 4. 刷盘策略（Flush / Fsync）

- 问自己：**写入数据时，是同步刷盘还是异步刷盘？**
  - 同步刷盘：写入必须 fsync 才返回
  - 异步刷盘：先返回成功，再由后台批量刷
- 影响：性能 vs 数据在崩溃场景中的安全性。

#### 5. 副本复制（Replication）

- 问自己：**数据有几个副本？复制方式是同步还是异步？**
  - 副本数量（replication factor）
  - primary 到 replica 的复制是否阻塞客户端返回
- 例子：
  - Kafka：leader → follower，同步复制，`replication.factor` 控制副本数
  - RocketMQ：Master → Slave，可同步/异步复制
  - MongoDB：Primary → Secondary，多节点复制
  - ES：primary → replica，同步复制

#### 6. 写入确认等级（Write Acknowledgement / Write Concern）

- 问自己：**客户端收到“成功”，说明什么已经完成？**
  - 只写入主节点？
  - 是否写入预写日志？
  - 是否写入多个副本？
- 例子：
  - Kafka：`acks=0/1/all`
  - RocketMQ：是否等刷盘、是否等 Slave 复制成功
  - MongoDB：`w:1` / `w:majority`，以及 `j:true`
  - ES：primary + replica 写入成功再返回（基础行为）

### 六、一页纸速记总结

- **Kafka**
  - 分区级 leader/follower，无全局主
  - topic → partitions，多副本在不同 broker
  - 高可靠：副本≥3 + `acks=all` + 手动提交 offset

- **RocketMQ**
  - Broker Master/Slave 主从架构
  - topic → 多个 queue
  - 高可靠：同步刷盘 + 同步复制

- **MongoDB**
  - Replica Set：Primary + Secondary，多数派选举
  - Journal（预写日志）+ Checkpoint 刷盘
  - 高可靠：`writeConcern: { w: "majority", j: true }`，节点数≥3

- **Elasticsearch**
  - 集群有 master；index 拆成 primary/replica shards
  - 写入：先 primary，再 replica，同步复制；日志与 segment 刷盘相对解耦
  - 高可靠：每个 shard 至少 1 个 replica，分布在不同节点
