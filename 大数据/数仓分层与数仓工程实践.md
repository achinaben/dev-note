# 数仓分层与数仓工程实践

本文档整理数仓分层的本质、分层原因、不同规模企业的分层策略，以及从规划到落地的数仓工程方法与保障体系。

> **阅读提示：** 下文中的 **DWT / DWA** 在不同公司可能都叫「宽表层」；**ADS / APP / DM** 可能都叫「应用层/集市层」。以**职责**（谁产出、谁消费、粒度如何）为准，缩写仅作对照。

---

## 一、数仓分层的本质

数仓分层的本质是通过**划分数据处理阶段**，实现数据**解耦、复用和简化管理**，使数据架构更清晰、维护成本更低。

---

## 二、数仓必须分层的三个核心原因

### 2.1 解耦数据处理，隔离变化影响

业务系统（如 ERP、CRM）的数据格式或逻辑可能频繁变动。分层后：

- **ODS 层**：原始数据直接同步、少加工。
- **下游模型层（如 DWD、DWS）**：只需调整对应适配逻辑，无需全链路修改。

从而避免「牵一发动全身」。

### 2.2 提高数据复用性，降低重复开发

将通用的中间数据（如用户基础信息、订单明细汇总）沉淀在中间层（DWS / DWA 等），不同业务需求（报表、分析、机器学习）可直接复用，无需重复从原始数据开始计算，节省算力与开发时间。

### 2.3 简化问题定位，提升运维效率

分层后数据流向清晰（典型路径：**ODS → DWD → DWS → ADS**），出现质量问题时可快速定位层级：

- 原始数据错误 → 重点查 ODS 同步逻辑。
- 指标计算错误 → 重点查 DWS / 汇总层逻辑。

无需在海量数据中盲目排查。

### 2.4 与「主题域 / 指标体系」的关系（补充）

分层解决的是**纵向**加工阶段；**主题域**（如交易、会员、供应链）解决的是**横向**业务切分。常见做法是：**同一主题域内**从 ODS 一路建到 DWS，**跨域**在 ADS 或应用层做指标拼接。  
**指标体系**（原子指标、派生指标、复合指标）多在 **DWS / 指标层**固化口径，在 **ADS** 做面向报表或 API 的裁剪，避免「同一 GMV 在十张报表里十个算法」。

## 三、「可以不分层」的特殊场景

并非所有数据仓库都必须严格分层，以下情况可简化或不分层：

| 场景 | 说明 |
|------|------|
| **数据规模极小** | 仅几张表、数据量在 GB 级以内，业务逻辑简单（如小型创业公司的基础报表）。 |
| **需求高度单一** | 仅服务 1～2 个固定场景（如只做月度销售统计），无需复用数据。 |
| **临时数据需求** | 为特定项目搭建的短期存储，项目结束后停用，无需长期维护。 |

## 四、不同规模企业的分层方案对比

| 维度 | 小型企业（约 100 人以内，GB 级） | 中型企业（约 100～1000 人，TB 级） | 大型企业（1000 人以上，PB 级） |
|------|----------------------------------|-------------------------------------|--------------------------------|
| **分层策略** | 简化：**ODS + ADS**，无中间层 | 标准：**ODS + DWD + ADS**，可选简单 **DWS** | 完整：**ODS + DWD + DWS + DWA + ADS**，部分加 **MID**（机器学习数据）等 |
| **核心目标** | 快速上线、基础报表，降低开发与维护门槛 | 平衡效率与成本，多业务线复用，减少重复开发 | 强解耦、高复用，支撑实时分析、AI，保障质量与稳定性 |
| **工具选择** | 轻量：MySQL、Excel、FineBI（基础版）等 | 混合：Hive/Spark、ClickHouse、Tableau 等 | 企业级：Flink、Hadoop、Impala、DataWorks 等 |
| **维护成本** | 极低，1～2 人可兼顾 | 中等，约 3～5 人团队 | 高，常需 10 人以上（含治理、BI） |
| **适用场景** | 简单销售报表、考勤等固定需求；无复杂复用 | 多业务线独立分析；灵活报表 | 跨部门数据打通；实时推荐、风控；强治理与合规 |

**复查说明（合理性）：** 上表按「人数 + 数据量级」是经验区间，不是硬阈值；**工具列**仅举例，实际常出现「小公司也用云上托管数仓」或「大企业仍保留 Hive」——以数据量、查询 SLA、团队技能为准。

## 五、数仓工程方法论框架

工业界主流思路以**分层建模**为核心，常见框架对比如下：

| 方法论 | 核心思想 | 分层逻辑（示意） | 适用场景 | 特点 |
|--------|----------|-------------------|----------|------|
| **Inmon** | 自上而下：先建数仓再建集市 | ODS → EDW → 数据集市（DM） | 大型、多业务线、强一致性（如金融/集团） | 整合度高、一致性强；周期长、成本高 |
| **Kimball** | 自下而上：先集市再整合 | ODS → 维度/事实（DWD）→ 集市（DWS/DM） | 中小、业务聚焦、快速见效（如电商/互联网） | 周期短、贴合业务；跨集市一致性需额外保障 |
| **Data Vault** | 中心表 + 卫星表 + 链接表，易扩展 | ODS → HUB → SAT → LINK → 信息集市 | 超大型、高频变更、海量数据（如零售/物流） | 灵活、易扩展；模型复杂、学习成本高 |
| **混合（Inmon+Kimball）** | 核心层保一致性，集市层敏捷 | ODS → EDW → DWD → DWS → DM | 多数中型互联网/制造业 | 平衡长期稳定与短期响应，落地最广 |

> **说明：** 各公司层名可能不同（如 ADS / DM / APP、DWT / 宽表），以职责与血缘为准，不必拘泥于缩写。

## 六、数仓工程全流程（混合架构示例）

整体遵循：**需求分析 → 模型设计 → 开发实现 → 测试上线 → 运维优化**。

### 6.1 阶段 1：需求分析

| 方向 | 要点 |
|------|------|
| **业务调研** | 访谈业务方，明确核心需求；梳理业务流程与 KPI/OKR（如 DAU、GMV、复购率）。 |
| **数据调研** | 梳理数据源与《数据源清单》；评估完整性、准确性、时效性。 |
| **输出** | 《数仓需求规格说明书》：业务范围、指标定义（口径、维度）、时效（T+1 / 实时等）。 |

### 6.2 阶段 2：模型设计

**分层职责与粒度（示例，可与公司规范对齐）：**

| 分层 | 英文全称 | 核心职责 | 数据粒度 | 典型存储（离线 / 实时） |
|------|----------|----------|----------|-------------------------|
| **ODS** | Operational Data Store | 同步源数据，尽量保持原貌（仅格式转换） | 与源一致 | HDFS/Hive；Kafka |
| **DWD** | Data Warehouse Detail | 清洗 + 维度建模（事实表/维度表） | 最细粒度 | Hive/Iceberg；Kudu/HBase 等 |
| **DWS** | Data Warehouse Summary | 按主题汇总（用户-日、商品-日等） | 主题汇总 | Hive/Iceberg；Redis/Kudu 等 |
| **DWT** | Data Warehouse Wide Table | 实体宽表（如用户 360、商品宽表）；部分公司写作 **DWA** | 实体级 | Hive/Hudi；HBase/ES 等 |
| **ADS** | Application Data Service | 面向应用/报表/API 的**最外层**数据服务层，常由 DWS/DWT 二次加工 | 按报表或接口粒度 | Hive/MySQL；Redis/API 等 |
| **DM** | Data Mart | 数据集市：可与 ADS 同层或作为 ADS 的子集（部门级集市） | 按场景 | Hive/MySQL；Redis/ES 等 |

**与全文路径的对齐：** 常见离线链路写作 **ODS → DWD → DWS → DWT（可选）→ ADS/DM**。DWT 不是必建层——若 DWS 已能满足宽表需求，可合并省略。

**Kimball 维度建模要点（多在 DWD）：**

- **维度表**：分析角度（用户、商品、时间、渠道等），相对稳、低基数。
- **事实表**：业务事件（下单、支付、点击），高基数；可分为事务型、周期快照、累积快照等。

**设计输出物：**《分层设计文档》《数据模型设计文档》《ER 图》等。

### 6.3 阶段 3：开发实现

#### 离线数仓（T+1 / T+N）

| 环节 | 要点 |
|------|------|
| **ODS** | 工具：Sqoop、DataX、Flume 等；策略：全量 / 增量 / 全量+增量合并；按 `dt` 等分区输出。 |
| **DWD** | 清洗（去重、补全、格式统一、过滤测试数据等）；Hive/Spark SQL 建维度表与事实表。 |
| **DWS/DWT/DM** | 主题聚合、宽表整合；合理分区。 |
| **调度** | Airflow、Azkaban、DataWorks 等；配置依赖顺序、执行时间、失败重试与告警。 |

#### 实时数仓（秒 / 分钟级）

| 环节 | 要点 |
|------|------|
| **ODS** | Kafka 接入；Debezium CDC；业务直写事件 JSON；按域划分 Topic。 |
| **DWD** | Flink 清洗、解析 JSON；维度关联（如 Redis 存维度快照）。 |
| **DWS** | 滚动/滑动窗口聚合；结果入 Redis、Kudu 等。 |
| **DM** | 写入 ES、MySQL 等供查询与大屏。 |

### 6.4 阶段 4：测试上线

- **数据测试**：准确性（与源、与上层核对）、完整性（字段、分区）、一致性（离线 vs 实时允许小幅误差需约定）。
- **性能测试**：离线任务时长、查询耗时；实时端到端延迟与吞吐。
- **上线**：灰度业务域与用户，准备回滚与切换方案。

**测试补充（可写入用例清单）：**

| 类型 | 示例 |
|------|------|
| **边界** | 分区空跑、源表零行、极大 Key 倾斜 |
| **回溯** | 补数后 DWD/DWS 与 ADS 是否可重算对齐 |
| **口径** | 与财务/运营 Excel 或 BI 看板抽样对数 |

### 6.5 阶段 5：运维优化

- **监控**：任务成败、数据量波动、查询超时；Prometheus+Grafana、ELK、告警等。
- **质量**：规则校验（非空、范围、分区完整）；定期质量报告。
- **性能**：模型合并、分区/分桶、计算引擎与并行度、冷热分层与压缩格式。
- **迭代**：新指标走完整流程；废弃表与任务及时下线。

## 七、工程保障体系

| 体系 | 目标 | 关键措施 |
|------|------|----------|
| **数据质量** | 准确、完整、一致、及时 | 接入校验、ETL 内嵌规则、自动巡检、责任边界（源 vs 数仓）。 |
| **元数据** | 可知、可管、可用 | 技术元数据（表结构、任务依赖）+ 业务元数据（口径、负责人）；Atlas 等工具；血缘与检索。 |
| **安全** | 不泄露、不篡改、可追溯 | 最小权限、脱敏、审计日志、加密存储。 |
| **成本与生命周期** | 可计量、可下线 | 表/任务分级（核心/一般/临时）、冷数据归档、无效任务治理。 |

**湖仓与离线数仓（一句话）：** 若采用 **数据湖 + 表格式**（Iceberg/Hudi/Delta），分层逻辑不变，多的是**统一元数据、ACID、时间旅行**等能力，不替代「ODS/DWD/…」的职责划分。

## 八、常见问题与思路

| 问题 | 解决思路（摘要） |
|------|------------------|
| **数据倾斜** | 离线：重分区、过滤异常；实时：热点 Key 拆分等。 |
| **任务慢** | SQL 优化、资源扩容、分桶减少扫描。 |
| **离线与实时不一致** | 统一时间口径、次日离线校正、一致性校验任务。 |
| **模型难扩展** | 维度建模、预留扩展字段、按域拆分降低耦合。 |

### 8.1 常见误区（复查）

| 误区 | 更稳妥的做法 |
|------|----------------|
| **层数越多越好** | 层数多会增加调度链路与排查成本；以**复用与职责清晰**为度，能合并的宽表层不要硬拆。 |
| **ODS 做大量业务计算** | ODS 宜**贴近源、少加工**，复杂清洗放在 DWD，否则源变更时影响面过大。 |
| **ADS 直接读 ODS** | 短期省事，长期口径混乱；至少经 DWD 统一清洗与主键对齐。 |
| **只建离线、忽视实时口径** | 若业务已用实时看板，需在文档中约定**离线为准还是实时为准**及 T+1 校正机制。 |

## 九、小结

- 数仓工程的核心不是堆砌组件，而是**以业务为导向**，通过**分层建模、规范开发、测试与持续运维**沉淀高价值数据资产。
- **新手建议**：从 Kimball + 混合架构思路入手，先落地单一业务域（如交易域），再扩展。
- **企业需平衡**：短期响应与长期一致性；避免小公司过度设计（如盲目上 Data Vault），也避免无分层导致混乱。
