## 1. 手册目标与边界

### 1.1 目标

- **统一语言**：把采集、事件口径、分层、指标、告警等概念统一到一套团队可执行的规范。
- **指导选型**：从数据量级、团队技能、成本与 SLA 出发，给出组件组合与适用场景。
- **可工程化落地**：提供可直接用于评审/排坑的关键约束与检查清单（规则上限、去重幂等、告警降噪、治理边界等）。

### 1.2 边界（你需要先知道的事实）

- 本手册强调“**监控/告警**”与“**在线限流**”不同：  
  - **在线限流**在网关/入口做毫秒级决策（失败要可控，直接 429/降级）。  
  - **监控/告警**允许秒级/分钟级延迟，更强调趋势与可解释性。  
- 本手册给的是“推荐可落地方案”，并不假设你一定要上全家桶；**层数与组件数量不是 KPI**。

## 2. 一张图理解端到端链路

一个常见、可扩展的端到端链路可以写成：

1. **采集层**：客户端 SDK / 前端埋点 / 服务端日志 / 业务埋点  
2. **接入层**：Kafka（统一入口、削峰、解耦），CDC（Debezium/Canal），离线同步（DataX/Sqoop）  
3. **计算层**：实时 Flink（清洗/聚合/窗口/状态），离线 Spark/Hive（T+1/补数/重算）  
4. **存储层**：  
   - 分析（OLAP）：ClickHouse  
   - 湖仓底座（冷数据/长期归档）：对象存储/HDFS + 湖表格式（Iceberg/Hudi/Delta）  
   - 检索（全文/日志）：Elasticsearch  
5. **数仓分层**：ODS → DWD → DWS →（可选 DWT 宽表）→ ADS/DM  
6. **服务层**：报表/看板/对外 API（数据服务）  
7. **治理与保障**：质量、元数据/血缘、安全、成本与生命周期

## 3. 用户行为数据采集（事件模型与合规）

### 3.1 常见采集方式

| 方式 | 主要数据来源 | 典型粒度 | 成本与特点 |
|------|--------------|----------|------------|
| 服务器日志采集 | Web/网关访问日志 | 请求级 | 无需改业务代码，粒度偏粗（URL、状态码、时间等） |
| 客户端采集 | 端上行为与上下文 | 视实现而定 | 可拿到设备/网络/交互上下文，常与 SDK/埋点结合 |
| 代码埋点 | 业务代码显式上报 | 事件级（可很细） | 语义最准，但需规范，否则口径混乱 |
| 可视化埋点 | SDK + 圈选配置 | 控件/页面级 | 运营自助，减少发版依赖；依赖 UI 稳定性 |
| 全埋点 | SDK 自动采集 | 广覆盖 | 覆盖广但噪声大，数据量与合规压力更高 |

**常见落地组合：** 全埋点/可视化补齐“页面与点击”，代码埋点兜底“关键业务事件”（注册、下单、支付、退款等）。

### 3.1.1 三类“埋点方式”的边界（写清能减少争议）

| 方式 | 优点 | 典型风险/限制 |
|------|------|---------------|
| 全埋点 | 覆盖广、先采后用 | 数据量大、噪声多；合规压力更大；对“业务语义事件”（如下单成功）仍需代码埋点兜底 |
| 可视化埋点 | 运营可自助配置、减少发版依赖 | 依赖 UI 结构稳定；控件复用/动态列表容易圈错；事件语义容易不一致 |
| 代码埋点 | 语义最准、可关联业务状态 | 开发成本高；若缺规范，容易“各写各的”导致口径混乱 |

### 3.2 统一事件（Event）模型：所有采集最终都要对齐它

一个可用事件通常回答 4 个问题：

| 问题 | 常见字段（示例） | 说明 |
|------|------------------|------|
| 谁做的 | `user_id`、`device_id`、`session_id` | 未登录用设备标识；登录后用用户标识并保留映射关系 |
| 做了什么 | `event_name`、`event_type` | 如 `page_view`、`click`、`add_to_cart`、`pay_success` |
| 什么时候/在哪 | `event_time`、`timezone`、`page_url`、`referrer` | `event_time` 建议毫秒时间戳；保留来源页便于漏斗/路径 |
| 带了什么信息 | `properties` | 商品 id、金额、渠道、AB 分组等业务属性 |

**口径稳定性建议：**

- **事件名要稳定**：上线后尽量不改名；要改就做“新旧事件映射表”，避免历史口径断裂。
- **属性分层管理**：公共属性（设备、网络、App 版本）与业务属性（订单、商品）分开治理。

### 3.3 会话、去重与幂等（不做会让数据“虚胖”）

- **会话切分**：常见规则为无操作超过 \(30\) 分钟视为新会话；App 退后台/杀进程也可触发会话结束。
- **客户端上报去重**：事件携带 `event_id`（UUID/雪花等），服务端按 `event_id` 去重；弱网重试/断点续传必做。

更具体的“容易出问题”的场景与建议：

| 场景 | 问题 | 建议做法 |
|------|------|----------|
| 弱网重试/断点续传 | 同一事件可能上报多次 | 事件带 `event_id`，服务端按 `event_id` 去重 |
| App 崩溃/重启 | “启动事件”重复 | 客户端生成启动批次 id，或服务端做时间窗口内合并 |
| 浏览器回退/刷新 | PV/曝光重复 | 增加 `page_instance_id`（一次页面实例）区分“同 URL 不同实例” |

### 3.4 用户标识（ID）体系与渠道归因（必须先定口径）

常见 ID：

| ID | 何时有 | 用途 |
|----|--------|------|
| `device_id` | 初次启动就有 | 未登录用户分析、去重、反作弊 |
| `user_id` | 登录后才有 | 用户级留存、付费、分群 |
| `session_id` | 每次会话 | 漏斗/路径分析 |

渠道/广告归因（简化版建议）：事件里至少带上 `utm_source / utm_medium / utm_campaign`（或自定义渠道字段），并提前约定“首次触达/最近一次触达”的归因口径，否则转化率会算出多个版本。

### 3.4 隐私合规与数据安全（必须写进方案/评审）

最低限度要约定：

- **最小化采集**：不采与目标无关的敏感信息。
- **敏感字段处理**：手机号/邮箱脱敏或哈希；密码、身份证号等原则上不入行为库。
- **用户同意**：SDK 初始化与上报受“用户授权/隐私开关”控制（未授权不采集或仅采严格必要字段）。
- **保留期限**：明细保留多久、何时归档或汇总（与成本治理绑定）。

### 3.5 采集选型建议（可直接用于评审）

1. **先定目标**：你们要做“产品分析（漏斗/留存）”、还是“风控反作弊”、还是“投放归因”，目标不同，采集粒度与字段完全不同。  
2. **关键链路必须代码埋点**：注册/下单/支付/退款等建议服务端埋点为准；前端埋点作为补充。  
3. **先把事件规范写出来**：事件命名规范、公共属性、ID 体系、去重规则、合规边界写清后再谈 SDK/平台选型。

## 4. 数据接入与异构数据源

### 4.1 推荐接入总线：Kafka

Kafka 典型价值：

- **高吞吐、低延迟**：适合行为事件、网关日志、业务事件流。
- **解耦**：上游只管写，下游可多路消费（实时计算、离线落盘、监控告警、回放补算）。
- **削峰**：突发流量通过队列缓冲，降低下游被打爆概率。

### 4.2 常见补充组件（按场景选）

- **CDC**：Canal / Debezium（适合从 MySQL/PostgreSQL 等抓取变更，形成变更流）
- **离线同步**：Sqoop / DataX（库表级批量同步，典型用于 ODS）
- **Pulsar**：与 Kafka 定位相近，多租户与分层存储更强，但生态一般弱于 Kafka（谨慎评估团队经验）

## 5. 存储层选型（OLAP/湖仓/检索）

### 5.1 常见组件对比（工程视角）

| 组件 | 优势 | 劣势/注意点 | 适用场景 |
|------|------|-------------|----------|
| ClickHouse | 列式、查询快、压缩高 | 写入/复杂事务不是强项；需要良好分区/分片设计 | OLAP 分析、实时报表、明细与汇总 |
| HDFS | 分布式、可靠、成本低 | 交互查询差，更多配合计算引擎 | 冷数据、离线数仓底座 |
| 对象存储（S3/OSS） | 成本低、扩展好 | 需配合引擎做“表/计算”（不是数据库） | 冷归档、湖仓底座、备份 |
| Elasticsearch | 检索强、近实时 | 运维成本与治理要求高（索引、分片、冷热等） | 全文搜索、日志检索、告警查询 |
| MongoDB | 文档存储灵活 | 分析型聚合非强项 | 半结构化业务数据、文档类日志 |

### 5.2 推荐组合（把“复杂度”写清楚）

#### 方案一：实时 + 分析一体（偏 OLAP，链路短）

```
Kafka + Flink + ClickHouse
```

- 适合：更偏“实时 + 分析”的团队，希望栈简单、上手快、报表体验好
- 关键前提：要做好 ClickHouse 的分区/分片、冷热分层、写入限速、去重口径与治理

#### 方案二：传统离线数仓（强批处理、生态成熟）

```
Kafka + Flink + HDFS + Hive + Spark
```

- 适合：批处理占比高、成本敏感、有 Hadoop 经验团队

#### 方案三：搜索 + 分析并重（检索与指标分工明确）

```
Kafka + Flink + Elasticsearch + ClickHouse
```

- 适合：日志平台、监控告警、既要全文检索又要多维分析

## 6. 计算引擎选型（Flink/Spark/交互查询）

### 6.1 计算引擎对比（结论导向）

| 组件 | 优势 | 劣势/注意点 | 适用场景 |
|------|------|-------------|----------|
| Flink | 低延迟、状态/窗口成熟、Exactly-Once 语义完善 | 学习成本高；状态与 checkpoint 需要治理 | 实时流处理、实时聚合、实时数仓与监控告警 |
| Spark | 生态广、批处理强、ML 丰富 | 微批延迟通常高于纯流；资源开销更大 | 离线 ETL、复杂计算、机器学习 |
| Hive（+ 执行引擎） | SQL 友好、与湖仓/离线生态契合 | 交互性能依赖引擎/文件格式/分区设计 | 传统数仓、离线调度 |
| Presto/Trino | 跨源即席查询 | 大查询内存与治理要求高 | 联邦查询、即席分析 |
| ClickHouse（计算+存储） | 即席聚合快 | 复杂 DAG/多阶段 ETL 非主战场 | 指标分析、物化汇总、实时看板查询 |

### 6.2 实时工程的两条红线（写进评审能避大坑）

- **窗口越大、滑动越小** → 同一事件会被分配到更多窗口，计算放大（成本与延迟都会爆）
- **key 越细（user+ip+uri+method）** → 状态数量暴涨，RocksDB state 变大，checkpoint 变慢

经验建议：

- **先粗后细**：先按 `uri+method` 做接口级趋势告警，再按 `user/ip` 做定位告警（两条流/两套规则）
- **配置要有上限**：如 `windowSeconds` 最大 10 分钟、`slideSeconds` 最小 5 秒
- **默认降级策略要明确**：规则系统抖动时到底“默认规则”还是“跳过告警”，必须写清

## 7. 数仓分层与工程实践（ODS→DWD→DWS→ADS）

### 7.1 分层的本质与价值

分层的本质是把数据加工拆成阶段，带来三件事：

- **解耦**：源系统变化，尽量只影响 ODS/DWD 适配，不波及全链路
- **复用**：通用明细/汇总沉淀为中间层资产，减少重复开发与算力浪费
- **易排障**：数据流向清晰（ODS→DWD→DWS→ADS），质量问题更容易定位

补充理解：

- **分层**解决“纵向加工阶段”，**主题域**解决“横向业务切分”（交易、会员、供应链等）
- **指标体系**建议在 DWS/指标层固化口径，在 ADS 做面向应用裁剪，避免“一份 GMV 十个算法”

### 7.2 分层职责（建议的标准口径）

| 分层 | 核心职责 | 数据粒度 | 常见存储（离线/实时） |
|------|----------|----------|------------------------|
| ODS（贴源） | 同步源数据，尽量保持原貌（仅格式转换） | 与源一致 | HDFS/对象存储/湖表；Kafka |
| DWD（明细） | 清洗、标准化、主键对齐；维度建模（事实/维度） | 最细粒度 | Hive/Iceberg；也可入 ClickHouse 明细 |
| DWS（汇总/主题） | 按主题域聚合与复用（用户-日、商品-日等） | 主题汇总 | Hive/Iceberg；也可入 ClickHouse 汇总 |
| DWT/DWA（可选宽表） | 实体宽表（用户 360、商品宽表等） | 实体级 | Hive/湖表；或宽表入 OLAP |
| ADS/DM（应用/集市） | 面向报表/API 的最终数据服务层 | 按应用口径 | ClickHouse/ES/MySQL/Redis/API |

> DWT 不是必建层：如果 DWS 已能满足宽表与复用，就合并省略，避免链路变长。

### 7.3 不同规模的分层策略（只做“必要分层”）

| 规模/数据量（经验） | 推荐分层 |
|--------------------|----------|
| 小（GB 级、需求单一） | ODS + ADS（先跑通与稳定口径） |
| 中（TB 级、多业务线复用） | ODS + DWD + ADS（按需补 DWS） |
| 大（PB 级、强治理与实时） | ODS + DWD + DWS +（可选 DWT）+ ADS/DM（配套治理体系） |

### 7.4 工程全流程（从需求到运维）

1. **需求分析**：明确业务范围、指标口径、时效（T+1/实时）、维度与消费方式  
2. **模型设计**：主题域拆分 + 分层职责 + 维度/事实建模（主要在 DWD）  
3. **开发实现**：离线（Spark/Hive）与实时（Flink）分别实现；调度依赖、重试与告警配置  
4. **测试上线**：数据准确性/完整性/一致性，性能与延迟；灰度与回滚方案  
5. **运维优化**：任务成功率、数据量波动、查询 SLA；持续做模型与成本治理

## 8. API 访问监控：Redis 规则 + Flink 动态窗口（告警系统）

### 8.1 先澄清：这是“监控/告警”，不是“在线限流”

- **在线限流**：网关/入口做实时拦截（稳定优先，失败可控，直接 429/降级）
- **监控/告警**：允许延迟，关注异常趋势与可解释性（谁触发、持续多久、用的哪版规则）

本章方案：Redis 存规则，Flink 做窗口统计，超过阈值输出告警（写 ES / 调告警服务）。

### 8.2 Redis 规则设计（建议字段）

规则以 JSON 存在 Redis（如 key：`api_rule:{ruleId}`），字段建议至少包含：

| 字段 | 含义 |
|------|------|
| `ruleId` | 规则唯一标识 |
| `uri` | 接口路径 |
| `method` | HTTP 方法 |
| `windowSeconds` | 统计窗口长度（秒） |
| `maxAccessCount` | 窗口内最大访问次数 |
| `enabled` | 是否启用 |

为了更“可用”，建议补充：

| 字段 | 含义 | 为什么需要 |
|------|------|------------|
| `ruleScope` | 粒度：`USER`/`IP`/`APP`/`URI` | 决定 `keyBy` 维度，维度不一致会误伤/漏报 |
| `matchType` | 精确/前缀/正则 | URI 常带路径参数，需要模式匹配 |
| `slideSeconds` | 滑动步长 | window 与 slide 成对配置，避免计算开销失控 |
| `minSample` | 最小样本数 | 小流量接口避免误告警 |
| `severity` | 告警级别 | 路由到不同通知渠道 |
| `dedupSeconds` | 去重窗口 | 避免刷屏 |
| `cooldownSeconds` | 冷却时间 | 降低噪声 |
| `ruleVersion` | 规则版本 | 告警可追溯、支持回滚与灰度 |

#### 8.2.1 规则配置示例（便于你直接试跑）

```plaintext
# /api/login 接口：1 分钟内最多 5 次请求
SET api_rule:login "{\"ruleId\":\"login\",\"uri\":\"/api/login\",\"method\":\"POST\",\"windowSeconds\":60,\"maxAccessCount\":5,\"enabled\":true}"

# /api/payment 接口：5 分钟内最多 10 次请求
SET api_rule:payment "{\"ruleId\":\"payment\",\"uri\":\"/api/payment\",\"method\":\"POST\",\"windowSeconds\":300,\"maxAccessCount\":10,\"enabled\":true}"
```

### 8.3 口径设计：你到底在“数什么”

监控常见 3 个口径（建议至少选一个并写清）：

| 口径 | 描述 | 常用场景 |
|------|------|----------|
| 访问次数/QPS | 窗口内请求总数 | 防刷、接口被打爆趋势 |
| 错误率 | \(5xx\)/总请求 | 线上故障、依赖异常 |
| 延迟分位 | P95/P99 | 性能退化、慢查询 |

建议在日志解析阶段补齐 `statusCode`、`latencyMs`，同一条 key 输出：`count`、`errorCount`、`p95Latency`（RED：Rate/Errors/Duration）。

### 8.4 工程关键点（比“写个窗口”更重要）

- **keyBy 维度必须与规则粒度一致**：例如规则是按 IP，就不要 keyBy userId；规则按 URI，就不要把 userId 拼进去。
- **窗口与滑动的成本边界**：窗口大 + 滑动小会导致计算放大；写进规则上限。
- **规则系统故障兜底**：Redis 取规则失败时，是“默认规则”还是“跳过告警”，要明确并在告警中标记。
- **告警去重与冷却**：同一接口持续异常，不要每个窗口都发一条。
- **动态配置生效**：定时拉取（简单稳定）或订阅通知（生效快但链路复杂）；变更要有版本号与灰度策略。

> 重要提示：Flink 自定义动态窗口分配需要谨慎核对 API 版本与生命周期；一些示例实现会出现生命周期方法不匹配、`keyBy` 键未实现 `equals/hashCode`、`AggregateFunction` 返回 `null` 导致下游异常等工程问题。建议以“规则维度 + 窗口策略 + 状态成本 + 告警降噪”为核心设计，而不是拘泥于某段示例代码。

### 8.5 动态配置：Redis 规则如何安全生效（别把自己“配挂了”）

#### 8.5.1 刷新策略

| 策略 | 做法 | 适合 |
|------|------|------|
| 定时拉取 | 每 N 秒按 key 拉取/扫描 | 简单稳定，延迟可控 |
| 订阅通知 | 配置平台更新后发 Redis PubSub / MQ | 生效快，但链路复杂 |

#### 8.5.2 一致性与回滚

- 规则变更建议带 **版本号**（`ruleVersion`），并在告警里写清“本次使用的规则版本”，便于追溯。  
- 新规则上线建议支持 **灰度**（按 URI 或按小流量集群先启用），避免一次配置误伤全量。  

### 8.6 数据流小结（把系统拆清楚，排障更快）

1. **配置层**：Redis 存 `api_rule:*`，定义每接口窗口与阈值（含粒度、滑动、去重等）。  
2. **采集层**：网关/服务日志进 Kafka（建议包含 `uri/method/userId/ip/statusCode/latencyMs/eventTime`）。  
3. **计算层**：Flink 按规则粒度 `keyBy`，开窗聚合，超过阈值输出 `Alert`。  
4. **输出层**：告警写 ES/时序库并调用告警服务；同时沉淀报表与事后分析。

## 9. 工程保障体系（质量/元数据/安全/成本）

| 体系 | 目标 | 关键措施 |
|------|------|----------|
| 数据质量 | 准确、完整、一致、及时 | 接入校验、ETL 内嵌规则、自动巡检、责任边界（源 vs 数仓） |
| 元数据与血缘 | 可知、可管、可追溯 | 技术元数据（表结构、任务依赖）+ 业务元数据（口径、负责人）；血缘与检索 |
| 安全 | 不泄露、不篡改、可审计 | 最小权限、脱敏、审计日志、加密存储 |
| 成本与生命周期 | 可计量、可下线 | 表/任务分级（核心/一般/临时）、冷数据归档、无效任务治理 |

一句话理解湖仓：用了 Iceberg/Hudi/Delta 之后，分层逻辑仍然存在；新增的是 ACID、时间旅行、统一元数据等能力，并不替代 ODS/DWD/… 的职责划分。

## 10. 落地检查清单（上线前复查）

### 10.1 采集与事件

- 事件模型是否统一（谁/做了什么/何时何地/属性）  
- 事件名是否稳定，是否有“新旧映射”机制  
- 是否有 `event_id` 幂等去重；弱网重试是否会导致重复  
- 合规边界是否写清（最小化采集、授权开关、敏感字段处理、保留期限）

### 10.2 接入与数据链路

- Kafka topic 是否按业务域拆分，是否明确 schema/版本兼容策略  
- CDC/离线同步是否明确增量/全量策略、分区字段（如 `dt`）  
- 是否有回放/补数能力（能回溯、能重算、能对齐）

### 10.3 数仓分层与口径

- 是否明确分层职责（ODS 贴源少加工；复杂清洗放 DWD）  
- 指标口径是否在可复用层固化（避免 ADS 直接读 ODS）  
- 离线与实时口径不一致时，是否写清“以谁为准”与 T+1 校正机制

### 10.4 实时作业与告警

- keyBy 维度是否与规则粒度一致  
- 窗口大小与滑动步长是否有上限，状态规模是否可控  
- 规则系统故障兜底策略是否明确（默认/跳过），是否有版本号与灰度  
- 告警是否去重、是否有冷却，是否能解释（规则版本、阈值、持续时间、触发维度）
- **时间语义**：若使用事件时间，是否有 watermark；若日志时间不可信，是否明确改为处理时间并写清口径  
- **告警输出健壮性**：是否避免“空告警/刷屏”；是否对持续异常做合并与降噪（去重窗口 + 冷却时间）  

### 10.5 运维与治理

- 任务成功率与延迟是否可观测（监控/告警）  
- 数据量波动与质量规则是否自动化（巡检与日报）  
- 成本是否可度量（存储、扫描、state、checkpoint）与可治理（冷热/归档/下线）
