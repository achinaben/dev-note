## 一、先对齐：MQ 常解决的六类问题

可以把 MQ 想成系统之间的 **邮局**：投递方不必等收信方当场处理完，信件在邮局里排队、可持久化、可多个人各取一份。

| 类型 | 一句话 | 典型业务画面 |
| --- | --- | --- |
| **异步** | 主链路不等慢活 | 下单成功先返回，发券、发短信后台做 |
| **解耦** | 发布方不关心有几个订阅方 | 订单支付后发「已支付」事件，积分/库存/营销各订阅 |
| **削峰** | 高峰先入队，下游按能力吃 | 大促、秒杀时保护数据库与核心接口 |
| **多订阅 / 广播** | 同一条事实，多方各自处理 | 用户注册 → 风控、推荐、运营统计 |
| **日志与事件流** | 业务事实变成可回放的数据流 | 埋点、审计、实时数仓、流计算 |
| **最终一致性辅助** | 「晚一点但必须对上账」 | 常配合 **Outbox / 本地消息表 + 幂等消费** |

更细的模型对比（点对点 / 发布订阅、削峰本质等）见《消息队列》前几节。

## 二、高频场景一览表

下表用于 **「我们这类需求，MQ 通常扮演什么角色」** 的快速对齐；落地时几乎都要叠加 **幂等、监控、堆积与降级策略**。

| 场景 | 业务例子 | MQ 常扮演的角色 | 设计时优先想 |
| --- | --- | --- | --- |
| **非核心链路后置** | 发邮件/短信、生成 PDF、推送第三方 | 异步 | 失败重试、死信、用户是否感知延迟 |
| **领域事件驱动** | 订单状态变更、用户资料更新 | 解耦 + 多订阅 | 事件版本、幂等键、顺序是否需要 |
| **高峰写压力** | 秒杀、抢购、批量导入 | 削峰 + 缓冲 | 队列上限、超时、降级、热点 key |
| **定时/延时业务** | 未支付关单、超时取消、提醒 | 延时消息或等价方案 | 精度、堆积、各产品实现差异 |
| **跨系统对账** | 支付结果同步多个子系统 | 可靠投递 + 最终一致 | Outbox、回查、补偿任务 |
| **行为与审计** | 埋点、操作日志、合规留痕 | 日志型管道 | 体量、留存、下游回放需求 |
| **后台任务队列** | 图片处理、报表生成、批量打款 | 点对点/工作队列 | 优先级、公平性、任务超时 |
| **多端通知** | App 推送、站内信、Webhook | 广播或扇出 | 送达语义、重复推送体验 |
| **数据管道入口** | 异步入湖、CDC 下游触发 | 高吞吐日志型 | 分区、顺序、Exactly-once 边界 |

## 三、典型场景怎么设计（要点与误区）

### 3.1 异步通知（邮件、短信、消息推送）

- **适合的原因**：主接口要快、稳定；第三方通道慢或不稳定不应拖垮下单/支付。
- **常见做法**：业务成功后发一条「待发送通知」类消息；消费者调用第三方，失败走重试/死信。
- **误区**：以为「进了 MQ 就一定发出」——仍需 **投递确认、消费 ACK、监控告警**；用户可能收到重复，产品上要可接受或做去重展示。

### 3.2 跨系统联动（订单支付 → 积分 / 库存 / 营销）

- **适合的原因**：新增一个订阅方时，不必改订单核心服务的同步调用链。
- **常见做法**：发布 **领域事件**（含业务 ID、版本、发生时间）；各订阅方独立消费、各自幂等。
- **误区**：把「强一致」全押在 MQ 上——MQ 更适合 **最终一致**；强一致场景要单独方案（同步事务、TCC 等，见《分布式事务》）。

### 3.3 削峰填谷（大促、突发流量）

- **适合的原因**：用排队换取 **时间**，让数据库和核心服务在可控速率下工作。
- **常见做法**：网关或应用层限流 + 入队；消费者并发与批量写库可配置。
- **误区**：只削峰不设 **队列上限与降级**——极端堆积会拖垮磁盘或拉长延迟；要准备 **拒绝策略、降级开关、人工介入**。

### 3.4 延时任务（关单、超时处理）

- **适合的原因**：不必为每个订单挂定时器；集中由 MQ 或调度系统触发。
- **常见做法**：延时消息、或调度扫描 + 发消息；各产品能力不同，需对照《mq功能》与官方说明。
- **误区**：忽略 **精度与堆积**——量大时延时精度可能下降；关单与支付回调并发时要靠 **状态机 + 幂等** 防错。

### 3.5 最终一致性（本地事务与发消息「尽量一起成功」）

- **适合的原因**：避免「库已提交但消息没发出」或相反。
- **常见做法**：**事务消息**（如 RocketMQ 半消息 + 回查）或 **Outbox 表**：本地事务内写业务 + 写 outbox，异步线程扫表发 MQ。
- **误区**：把 **MQ 事务消息** 当成 **分布式数据库 ACID**——语义边界要读透；消费侧仍按 **至少一次** 做幂等。

### 3.6 顺序消息（同一订单、同一用户的多条变更）

- **适合的原因**：乱序会导致状态错误（先收到「已退款」再收到「已支付」等）。
- **常见做法**：同一业务键路由到 **单分区 / 单队列 / MessageGroup**；消费者侧避免多线程抢同一键。
- **误区**：追求 **全局全序**——成本高且很少必要；通常 **分区内 / 分组内有序** 即可。一条失败可能 **阻塞同队列后续**，要提前设计跳过、隔离或重试策略。

### 3.7 行为流、埋点、审计

- **适合的原因**：高并发写、可丢失少量与「绝不能丢」要分层；下游可能是数仓、风控、推荐。
- **常见做法**：高吞吐管道（常讨论 Kafka 等）+ 分区；与业务库解耦。
- **误区**：审计类与统计类混在一个 topic 不设 **留存与合规策略**——要分清 **合规必达** 与 **统计可丢采样** 的边界。

## 四、不太适合用 MQ 的情况

| 情况 | 为什么 | 可考虑的替代 |
| --- | --- | --- |
| **强一致、立即读到最新** | MQ 天然异步，有可见延迟 | 同步 RPC、同库事务、读己之写设计 |
| **极低延迟且路径极短** | 多一跳网络与序列化 | 进程内调用、共享内存、同机优化 |
| **业务量极小、团队无运维经验** | 复杂度高、故障面变大 | 数据库轮询、简单任务表、成熟 SaaS |
| **需要复杂查询消息内容** | MQ 不是数据库 | 业务库 + CDC 或搜索引擎 |
| **「只消费一次」硬要求** | 端到端很难 cheap 地保证 | 按至少一次设计 + **幂等** |

## 五、场景 → 产品倾向（粗指引）

与《mq选型》一致取向：**粗粒度**，不确定时做 POC 与压测。

| 场景倾向 | 常讨论的方向 | 备注 |
| --- | --- | --- |
| 行为日志、多消费者回放、流处理生态 | Kafka | 分区与消费者 rebalance 纳入设计 |
| 订单域、延时关单、事务消息、Tag | RocketMQ | 仍要消费幂等与运维规范 |
| 复杂路由、传统任务队列、低延时 AMQP | RabbitMQ | 持久化、镜像、内存策略要配置 |
| 设备上报 | MQTT 接入，平台侧再接 Kafka/RocketMQ 等 | 见《mq选型》 |

## 六、上线前自检清单

- [ ] **语义**：生产与消费是否按 **至少一次** 设计？幂等键、去重或状态机是否具备？
- [ ] **堆积**：峰值下队列长度、磁盘、延迟是否可接受？有无 **上限与降级**？
- [ ] **顺序**：是否真的需要？若需要，范围是 **全局** 还是 **同一业务键**？
- [ ] **延时**：延时/定时是否满足业务精度？失败重试会不会 **重复触发**？
- [ ] **观测**：生产速率、消费滞后、重试/死信、错误日志是否有 **告警**？
- [ ] **安全**：Topic/权限/VPC 是否按环境隔离？
