## 一、MQ 在业务里常扮演的角色

可以把 MQ 理解成系统之间的 **「邮局」**：一方把「信件」（消息）投进邮局，另一方按自己的节奏取走处理；双方不必同时在线，也不必互相知道对方地址细节。

| 角色 / 价值 | 通俗说法 | 典型例子 |
| --- | --- | --- |
| **异步** | 主流程不等慢活干完再走 | 下单成功后发邮件、发短信，不卡支付接口返回 |
| **解耦** | A 改接口不必牵动所有调用方 | 订单服务发「订单已支付」事件，积分、库存、营销各订阅自己的主题 |
| **削峰** | 高峰先排队，下游按能力吃 | 大促时请求先入队，后台慢慢写库，避免数据库被打挂 |
| **缓冲与可恢复** | 消息先存着，宕机后可继续处理 | Broker 持久化 + 消费位点/ACK，重启后继续消费 |
| **多订阅 / 广播** | 同一件事通知多个系统各干各的 | 一条用户注册事件，同时触发风控、推荐、运营统计 |
| **日志与事件流** | 把业务事实变成可回放的数据流 | 行为日志、审计、实时数仓、与 Flink 等流计算对接 |
| **最终一致性辅助** | 跨系统「晚一点但必须对上账」 | 常配合 **Outbox / 本地消息表 + 幂等消费**；详见《消息队列》《分布式事务》 |

更细的削峰本质、推/拉模型、点对点与发布订阅对比，见《消息队列》第一至四节等。

## 二、能力维度速查（功能 → 设计时要想到什么）

下面按 **「产品功能」** 列维度；落地时几乎都要叠加 **业务幂等** 与 **监控告警**。

| 能力 | 解决什么问题 | 设计时注意 |
| --- | --- | --- |
| **可靠投递（生产侧）** | 尽量不让消息「发了却没落稳」 | 同步/异步发送、确认（confirm）、重试与超时；与刷盘、复制一起看作「多大概率不丢」 |
| **可靠消费（消费侧）** | 处理失败怎么办、是否重试 | ACK/NACK、重试队列、死信；**至少一次** 很常见 → **幂等** |
| **顺序消息** | 同一订单的多条变更必须按序处理 | 通常只在 **单分区 / 单队列 / 同 MessageGroup** 等范围内有序；全局严格顺序成本高 |
| **延时 / 定时消息** | 30 分钟未支付关单、定时提醒 | 各产品实现不同（原生、插件、TTL+死信）；**精度与堆积** 要测 |
| **事务消息（MQ 语义）** | 「本地事务成功」与「消息发出」尽量一致 | **不是** 数据库跨库 ACID；RocketMQ 半消息+回查、Kafka EOS 等语义边界要读透 |
| **过滤与路由** | 消费者只关心部分消息，或复杂投递规则 | Tag、Key、Exchange 类型、Kafka 分区策略等 |
| **回溯 / 重放** | 补数据、审计、新消费者从头读 | 日志型（Kafka、RocketMQ 位点）更强；经典队列型更偏「还在队列里的才能消费」 |
| **吞吐与延迟** | 扛多大峰值、接口要多快 | 分区数、批量、预取、网络与磁盘；**数字以压测为准** |
| **安全与隔离** | 多租户、权限 | ACL、vhost、VPC、云上托管策略 |

顺序、延时、事务三类在《消息队列》第十节有概览；消费侧模式与幂等见第十一节。

## 三、Kafka / RocketMQ / RabbitMQ 功能对照（简表）

与《mq 选型》第三节一致取向：**粗量级**，便于「这张表在聊功能时拿出来对齐」。

| 功能向维度 | Kafka | RocketMQ | RabbitMQ |
| --- | --- | --- | --- |
| **核心模型** | Topic + Partition（日志） | Topic + Queue（Broker 上分片） | Exchange + Queue + Binding |
| **吞吐** | 常见讨论很高（视分区与磁盘） | 常见十万级讨论多 | 万级～十万级常见，极高需调优 |
| **延迟** | ms 级 | ms 级 | 同机房等场景可 μs～ms 级 |
| **顺序** | 同一 key → 同一分区内有序 | 顺序主题 / MessageGroup 等 | 单队列保序相对直观 |
| **延时消息** | 非主打，常外置 | 原生能力较强 | 插件 / TTL + 死信等组合 |
| **事务消息** | 生产者事务 / EOS（语义有限） | 半消息 + 回查，贴近业务叙事 | AMQP 事务窄；可靠多靠 confirm + 业务幂等 |
| **路由 / 过滤** | topic + key（分区） | Tag 等 | routing key + 多种交换机 |
| **消费位点 / 语义** | offset，可回溯 | 位点，可回溯 | ACK/NACK，再投递 |
| **典型强项** | 日志流、多消费者回放、流处理生态 | 国内业务消息、顺序/延时/事务消息套件 | 灵活路由、AMQP 生态、传统企服 |

## 四、按「我要解决 X」快速对应

| 你的目标 | 可优先看的方向 | 补充 |
| --- | --- | --- |
| 全站日志、行为流、实时入湖、Flink | **Kafka** | 分区规划与消费者 rebalance 要纳入设计 |
| 订单状态、延时关单、事务消息、Tag 过滤 | **RocketMQ** | 仍要配合消费幂等与运维规范 |
| 任务队列、复杂路由、MQTT/STOMP、低延时 | **RabbitMQ** | 镜像队列、持久化与内存策略要配好 |
| 设备上报（物联网） | **MQTT Broker** 接入，平台侧再接 Kafka/RocketMQ 等 | 见《mq 选型》第七节 |
| 只想要「简单异步 + 云托管」 | 跟 **云平台默认**（如阿里云 RocketMQ、AWS MSK） | 运维成本常优先于个人偏好 |

仍不确定时，用《mq 选型》第四节 **决策检查清单** + POC 压测收敛。

## 五、功能背后的共同前提

1. **端到端很少保证「只消费一次」**：按 **至少一次** 设计，用 **幂等键、去重表、状态机** 兜底。
2. **MQ 的「事务」≠ 分布式数据库事务**：跨库强一致仍要靠业务方案与补偿。
3. **顺序、延时、堆积** 是一枚硬币的两面：顺序消费失败可能 **阻塞同队列后续消息**，要提前设计重试与降级。
