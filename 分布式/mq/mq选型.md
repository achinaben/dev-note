# 消息中间件选型（MQ 选型）

**你在做的事**：在 Kafka、RocketMQ、RabbitMQ（以及少量遗留的 ActiveMQ 等）里，按业务目标选一款或组合使用，而不是凭印象「哪个火用哪个」；**物联网**场景下往往还要先选 **MQTT 接入层**，再与本文三类 MQ 组合。

## 一、先定三类目标

选型时先问清楚优先级（往往只能突出其中一两项）：


| 优先级类型         | 典型问法                                  | 更常倾向              |
| ------------- | ------------------------------------- | ----------------- |
| **吞吐与扩展**     | 每秒多少条、是否要长期当「日志流 / 数据管道」、是否要接 Flink 等 | **Kafka** 讨论得多    |
| **业务消息与功能套件** | 顺序、延时、事务消息、Tag 过滤、国内生态与云产品            | **RocketMQ** 讨论得多 |
| **路由灵活与低延时**  | 复杂路由、多协议、传统企服、队列语义要「好懂」               | **RabbitMQ** 讨论得多 |


**举例**：要做「全站行为日志 + 实时数仓入湖」，通常先看 **Kafka**；要做「订单状态变更 + 延时关单 + 事务消息」，国内团队常看 **RocketMQ**；要做「多种交换机模型 + AMQP 生态 + 低延时任务队列」，常看 **RabbitMQ**。

## 二、按场景快速指向

下列为**经验向**指向，落地前仍需结合版本与压测。


| 场景                                             | 优先考虑                                                      | 简要理由                                       |
| ---------------------------------------------- | --------------------------------------------------------- | ------------------------------------------ |
| 日志采集、大数据管道、流处理、事件溯源（append-only）               | **Kafka**                                                 | 分区日志模型与生态成熟，吞吐与回放能力强                       |
| 微服务业务消息、顺序消息、延时、事务消息、过滤标签                      | **RocketMQ**                                              | 功能面贴近国内常见业务诉求，云厂商支持多                       |
| 任务队列、复杂路由（topic/direct/fanout）、多协议（MQTT/STOMP） | **RabbitMQ**                                              | Exchange–Queue–Binding 模型灵活，AMQP 成熟        |
| 遗留 Java EE / 老系统 JMS 集成                        | **ActiveMQ** 等                                            | 新系统单独选型一般不再首选                              |
| 只要「简单异步 + 一种云托管」                               | 跟随 **基础设施**：阿里云偏 RocketMQ、AWS 偏 MSK（Kafka）等               | 运维成本常比纯技术偏好更优先                             |
| **物联网**：设备上报/下发、弱网、海量连接                        | **MQTT Broker** 接入，平台侧再桥接 **Kafka / RocketMQ / RabbitMQ** | 设备侧不宜以「直连 Kafka」为主方案；详见 [七、物联网场景](#七物联网场景) |


**跨系统最终一致性**（业务库与消息一致）：多数团队用 **Outbox / 本地消息表 + 幂等消费**，不完全依赖某一种 MQ 的「事务消息」包办；详见《消息队列》与《分布式事务》相关笔记。

## 三、Kafka / RocketMQ / RabbitMQ 对照（选型向）

### 3.1 一页总表（粗量级，以压测为准）


| 维度               | Kafka                        | RocketMQ                        | RabbitMQ                           |
| ---------------- | ---------------------------- | ------------------------------- | ---------------------------------- |
| **一级模型**         | Topic + **Partition**（日志）    | Topic + **Queue**（Broker 上分片）   | **Exchange** + **Queue** + Binding |
| **吞吐印象**         | 常见讨论可达很高（依赖分区与磁盘）            | 常见 **十万级** 量级讨论多                | **万级～十万级** 常见，极高吞吐需调优              |
| **延时印象**         | **ms 级**                     | **ms 级**                        | **μs～ms 级** 场景多（同机房等）              |
| **消费位点**         | **offset**，可回溯               | **位点**，可回溯                      | 队列语义，**ACK/NACK** 与再投递             |
| **顺序**           | **同一 key → 同一分区** 内有序        | 顺序主题 / MessageGroup 等           | **单队列**保序相对直观                      |
| **延时消息**         | 非主打，常外置方案                    | **原生能力较强**                      | 插件 / TTL+死信等组合                     |
| **事务消息**         | 生产者事务 / EOS（语义有限，≠ 数据库事务）    | **半消息 + 回查** 等，贴近业务「发消息与本地事务」叙事 | AMQP 事务窄；可靠多靠 **confirm + 业务幂等**   |
| **路由**           | 主要靠 **topic + key（分区）**      | **Tag** 等过滤                     | **routing key** + 多种交换机类型          |
| **协调组件**         | ZK 或 **KRaft**               | **NameServer**                  | 无独立注册中心式「NameServer」               |
| **典型短板（选型时要接受）** | 运维与分区规划、消费者 rebalance 带来重复消费 | 主从写入语义、概念比「只有一个队列」多             | 极高吞吐与超大规模下成本与调优                    |


### 3.2 消费语义（选型必想）

三者端到端都**不要轻易假设「只消费一次」**：

- **Kafka**：至少一次常见；**rebalance** 等可能带来少量重复 → 下游 **幂等**。
- **RocketMQ**：重试、位点提交策略下同样按 **至少一次** 设计较稳妥。
- **RabbitMQ**：手动 ACK、断连、NACK 重回队等会导致 **再投**。

《消息队列》第十一节对幂等与应答有更细展开。

## 四、决策检查清单

按顺序勾选，帮助把讨论从「喜好」拉回「约束」。

1. **数据像不像「日志」**：要长期保留、多订阅者各自回放、与流计算强绑定？→ 偏向 **Kafka**。
2. **是否要原生延时 / 顺序 / 事务消息（产品级）**：强依赖且希望少造轮子？→ 多看 **RocketMQ**（仍要理解语义边界）。
3. **是否要极灵活路由与多种消息模式**：点对点、广播、主题、头路由混用？→ 多看 **RabbitMQ**。
4. **团队与运维**：现有监控、云平台托管、熟悉语言客户端、是否有专人运维 **Kafka 集群**？
5. **峰值与成本**：分区数、副本、镜像队列、跨可用区等都会换算成 **钱与人力**。
6. **合规与多租户**：隔离模型（vhost、ACL、VPC）是否在候选产品里好落地？

## 五、常见误区

- **误区 1**：「MQ 的事务消息 = 数据库 ACID 跨库事务」—— 多数场景只是**有限语义**或**最终一致**手段，复杂一致仍要靠业务设计与补偿。
- **误区 2**：「选了 Kafka 就不用管顺序」—— 顺序通常只在 **分区内 / 单队列** 成立；全局严格有序成本高。
- **误区 3**：「RabbitMQ 吞吐量一定差」—— 对很多企服业务量 **足够**；瓶颈常在业务与数据库而非 broker。
- **误区 4**：「只比 TPS 数字」—— **延迟分布、故障行为、运维工具链、客户端成熟度** 往往决定能不能长期用下去。

## 六、仍不确定时怎么收敛

1. 用 **真实消息大小、峰值 QPS、保留时长、消费者数量** 做 **POC 压测**（含故障注入：杀节点、断网、延迟）。
2. 固定 **1～2 个必测用例**：如顺序 key 场景、失败重试 + 死信、延时精度、回放与对账。
3. 与《消息队列》**第十二节** 粗表交叉核对；细节回三份专项笔记修订。

## 七、物联网场景

物联网消息选型建议按 **两层** 拆开，避免把「MQTT」和「Kafka/RocketMQ/RabbitMQ」混成一个问题。

### 7.1 两层架构（常见做法）


| 层级             | 解决什么问题                    | 常见选型                                                                                     |
| -------------- | ------------------------- | ---------------------------------------------------------------------------------------- |
| **设备接入层**      | 终端省电、弱网、TLS、主题订阅、QoS、海量连接 | **MQTT 协议 + MQTT Broker**（如 MosquEMQX、HiveMQ、VerneMQitto、 等）                             |
| **平台 / 数据中台层** | 高吞吐管道、多下游、回放、与微服务/数仓对齐    | 仍按本文 **一至六节** 在 **Kafka / RocketMQ / RabbitMQ** 中选；常见组合是 **Broker → 桥接 / 规则引擎 → Kafka，** |


**说明**：RabbitMQ 可通过 **MQTT 插件** 兼做接入层，后台消费侧继续用 AMQP，适合已深度使用 RabbitMQ 的团队。

### 7.2 MQTT Broker 快速指向（接入层）


| 更在意                                | 可优先考虑                                   |
| ---------------------------------- | --------------------------------------- |
| 轻量、学习成本、中小规模                       | **Eclipse Mosquitto**                   |
| 大规模连接、规则引擎、企业功能与商业支持               | **EMQX**、**HiveMQ** 等                   |
| 与现有 RabbitMQ 统一运维、设备 MQTT + 后台队列一体 | **RabbitMQ + MQTT 插件**                  |
| 少自建、要合规与云集成                        | 各云 **物联网平台 / IoT 套件**（底层多为 MQTT + 托管管道） |


落地前建议明确：**峰值连接数、消息 QPS、QoS 等级、是否要持久化离线消息、证书与多租户、与目标 MQ 的官方/成熟桥接方案**。

### 7.3 与 Kafka / RocketMQ / RabbitMQ 的衔接

- **偏时序与海量上报、多消费者各自消费、接 Flink 等**：平台侧仍倾向 **Kafka**，由 Broker 或流处理写入。
- **偏国内业务中台、顺序、延时、事务消息**：平台侧可看 **RocketMQ**。
- **已以 Rabbit 为中心、路由复杂**：可 **MQTT 插件接入 + AMQP 消费**，或桥接到现有队列。

消费语义与 **幂等** 要求与 **3.2 节** 一致：设备重连、QoS 重传、桥接重试均可能带来 **至少一次**，业务侧按幂等设计。

### 7.4 物联网补充检查清单

1. **接入与平台是否分离**：是否允许「仅 Broker、无第二段 MQ」就能满足？若要多下游解耦，通常需要第二段。
2. **消息是否需长期留存与回放**：需要则平台侧更偏向 **Kafka** 一类日志型模型。
3. **下行指令实时性与到达语义**：QoS0/1/2 与业务补偿策略是否在需求里写清？
4. **安全**：设备证书、Broker ACL、到内网 MQ 的链路是否分区隔离。
