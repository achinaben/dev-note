## 一、协议与特性

### 1.1 协议与模型

- **发布-订阅**；也支持 **点对点**、**一对多** 等用法（通过 Topic、消费者组组合实现）。
- **主从 Broker 集群**、**持久化消息**、**顺序消息**、**消息过滤（Tag）**、**事务消息**、**回溯消费**。
- 特点概括：**高性能、高可靠、低延迟**，适合 **大规模分布式系统**。

### 1.2 特性与场景

在阿里集团等环境中常用于：**交易、充值、流计算、消息推送、日志流式处理、Binlog 分发** 等。

---

## 二、组件总览

| 组件 | 作用（一句话） |
| --- | --- |
| **Producer** | 生产消息并投递到 Broker（经 **Proxy** 或直接连 Broker，视部署而定）。 |
| **Proxy** | **5.x / 云原生** 常见接入层：协议适配、权限与消息管理；经 NameServer 发现 Broker，再转发存取。 |
| **NameServer** | 轻量路由注册中心：Broker、Topic、队列等路由信息。 |
| **Broker** | 存消息、设位点；通常主从部署。 |
| **Consumer** | 拉取/消费消息，必须归属某个 **消费者组**。 |

## 三、生产者（Producer）

### 3.1 消息投递语义（可靠投递）

- **至少一次（At least once）**：客户端会 **重试发送**，直到 **Broker 至少成功接收一次**（仍要结合消费侧幂等与去重理解端到端语义）。
- 与部分 MQ 不同：**不支持按 Key 做负载均衡发送**；**顺序消息**等场景用 **MessageGroup**、**同一队列** 等机制保证顺序（见 [七、消息](#七消息message) 与 [八、主题](#八主题topic与标签tag)）。

### 3.2 发送方式

| 方式 | 行为 | 说明 |
| --- | --- | --- |
| **同步发送** | 发送后 **阻塞等待** Broker 响应 | 最直观，适合要立刻知道成败的场景。 |
| **异步发送** | **立即返回** 不阻塞线程；Broker 响应到达后 **回调** | 高吞吐时常用，在回调里处理成功/失败。 |
| **单向发送（Oneway）** | **不等待** 响应、**无回调** | 只负责「发出去」，**感知不到** 最终是否成功；追求极限延迟/吞吐时可选，**强可靠场景慎用**。 |

### 3.3 请求超时

- **等待响应或回调** 相关的超时，多为 **客户端本地行为**。
- **默认**：约 **3000 ms**；**建议** 不要超过 **30000 ms**（具体参数名以所用 SDK 为准）。

### 3.4 发送重试

- 发送失败时，客户端通常按策略 **自动重试**（次数、间隔与 **同步/异步** 路径以 **官方 SDK 文档** 为准）。

## 四、名称服务器（NameServer）

- 每个 **NameServer** 保存 **Broker 集群的注册信息、路由信息、队列信息**。
- 职责包括：**Broker 动态注册与发现**、**Topic 路由注册**（可理解为 **Broker + Topic + Tag** 相关的路由视图）。
- **Broker 向每一台 NameServer 注册** 自己的路由信息，因此 **每个 NameServer 实例上都有一份完整路由**（彼此无强一致主从，设计偏简单、可水平扩展）。
- **Producer / Consumer**（或经 **Proxy**）通过 NameServer 获知整个集群路由，查到各 **Topic** 对应的 **Broker** 列表，再完成投递与消费。

## 五、Broker 与分片、队列

### 5.1 Broker 职责

- 接收生产者消息，为消息设置 **偏移量（Offset）**。
- **集群部署**，常见 **Master / Slave**：
  - Slave **定时从 Master 同步**（同步或异步刷盘视配置）。
  - **Master 宕机** 时，Slave 通常 **可提供消费**，但 **不能继续写入**（主写能力受影响，具体行为以版本与部署模式为准）。

### 5.2 分片（Sharding）

- 每个 **Broker** 可存 **多个 Topic** 的消息；同一 **Topic** 的消息也可 **分散在不同 Broker** 上。
- **分片更多是拓扑/规划上的概念**，一般 **不必像指定分区号那样手动指定**；实际落地常通过 **Topic 在多 Broker 上的队列分布** 体现。

### 5.3 队列（与 Topic 的关系）

- **Topic 在一个 Broker 上的分片**，再 **等分为若干份**，其中 **一份** 即一个 **队列（MessageQueue）**。
- **队列** 是 **负载均衡时资源分配的基本单元**。
- 不同 Topic **队列数量可不同**，一般在 **创建 Topic 时** 由用户/运维指定。

### 5.4 刷盘模式

| 模式 | 行为 | 可靠性倾向 |
| --- | --- | --- |
| **同步刷盘** | 消息 **写入页缓存后**，再 **刷到磁盘** 才向生产者 **确认** | 单机尽量不丢（仍要结合复制、电源等看整体 RPO） |
| **异步刷盘** | 先 **写入页缓存并确认**，再按 **周期或阈值** 批量刷盘 | 吞吐更好；掉电等极端情况可能丢 **尚未落盘** 的数据 |

### 5.5 复制模式（主从）

| 模式 | 行为 | 可靠性倾向 |
| --- | --- | --- |
| **同步复制（同步双写）** | **Slave 成功复制后**，Master 才 **应答生产者** | 机架级容灾更好；延迟略增 |
| **异步复制** | **Master 成功即应答**，向 Slave **异步复制** | 吞吐好；Master 宕机瞬间可能 **未复制到从** 的数据丢失 |

**组合理解**：生产上常同时权衡 **刷盘** 与 **复制**（例如同步复制 + 异步刷盘，或双同步换更强一致与更高延迟），以官方推荐与压测为准。

### 5.6 动态伸缩能力

- **扩展容量 / 吞吐** 的常见手段：
  - **增大 Topic 队列数**：多在 **创建主题时** 设定（`mqadmin updateTopic` 的 `-q` 等，以版本为准）；已有 Topic 上调整队列需查当前版本是否支持及影响。
  - **增加 Broker 数**：新 Broker **启动并向 NameServer 注册** 后进入集群路由。
- **新增一台 Broker 时要注意**：
  1. **启动 Broker**，使其 **注册到 NameServer** 的 Broker 列表。
  2. 默认往往 **只自动创建内部系统 Topic / 系统消费者组**；若希望新节点上也承载 **业务 Topic 与消费者组路由**，通常需用 **管理工具 / 命令行** 从 **已有 Broker 复制或同步** 相应元数据与队列分布（以你所用版本文档为准）。

### 5.7 消息存储文件结构（Broker 本地）

消息默认落在 **Broker 本地磁盘**；**存储根目录** 由配置 **`storePathRootDir`**（及衍生路径配置）决定。常见顶层目录含义：

| 目录 / 文件 | 作用 |
| --- | --- |
| **commitlog** | **消息物理文件**：所有消息 **顺序追加** 写入的统一日志（与 Topic 解耦的集中写）。 |
| **consumequeue** | **逻辑队列索引**：面向消费的 **队列索引**，体量相对小，便于 **尽量常驻内存**，提升消费定位效率；**本身也会持久化**，只存索引不存全量消息，**省磁盘**。 |
| **index**（**indexFile**） | **消息索引**：主要维护 **Message Key ↔ CommitLog 物理偏移** 等，用于 **按 Key 检索**、加速查询。 |

### 5.8 消息过期与清理

- **保留时长** 配置并不能 **单独保证** 消息一定存满那么久：底层仍是 **本机磁盘**。
- **磁盘空间不足** 时，为保证服务稳定，Broker 可能 **提前强制清理** 数据，导致 **实际保存时间 < 配置的保存时长**。容量规划要与 **保留策略、流量** 一起看。

### 5.9 PageCache（页缓存）

- **操作系统 PageCache** 会缓存文件读写，**加速磁盘 I/O**；写入路径往往是：**先进入 PageCache**，再 **异步或显式刷盘** 到物理磁盘。
- **MySQL、Redis（持久化路径）、RocketMQ** 等都离不开这一机制；因此「**写入成功**」在工程上要分清是 **进缓存** 还是 **已落盘**，与 **刷盘模式、复制模式** 一起理解端到端持久性。

## 六、队列（MessageQueue）

- **消息传输与存储的实际单元**，可类比其他 MQ 里的 **分区（Partition）**。
- RocketMQ 用 **流式、可视为无限延伸的队列结构** 存消息；**同一队列内** 消息 **按到达顺序** 存储，具有 **顺序存储** 特征。
- **队列命名（版本差异）**：
  - **服务端 5.x**：队列名为集群分配的 **全局唯一字符串**，与物理节点 **解耦**（常称 **QueueName** 等概念，以官方文档为准）。

## 七、消息（Message）

### 7.1 最小传输单元与持久化

- **Message** 是最小传输单元。
- 默认 **持久化**：消息写入服务端存储文件，保证 **可回溯** 与故障 **可恢复**。

### 7.2 Message ID 与位点

- **Message ID**：业务上可用来标识消息；**重试时 Message ID 不变**（便于排查与去重策略设计，具体仍以客户端与服务端行为为准）。
- **消息位点（Offset）**：消息按到达顺序落在某 **Topic** 的多个队列中；每条消息在队列内有唯一 **Long 型坐标**，即 **消息位点**。  
  **主题 + 队列 + 位点** 可定位一条消息。

### 7.3 消费位点（ConsumerOffset）

- RocketMQ 按 **消费者分组** 维护消费进度：某组消费某队列时，**已消费到的最新一条消息的位点**，即 **消费位点**。

### 7.4 删除与滚动

- 存储空间有限，服务端会 **滚动删除** 最早的消息，因此 **最小/最大消费位点** 会随时间 **整体前移、递增变化**（概念上持续滚动）。

### 7.5 消息索引（MessageKey）

- 由 **生产者** 设置；用于区分业务、**按 Key 查询** 等（与「是否按 Key 负载均衡发送」是不同维度）。

### 7.6 消息类型

| 类型 | 说明 |
| --- | --- |
| **Normal** | 普通消息；常用于 **微服务解耦、事件驱动、数据集成**。 |
| **FIFO** | 顺序消息；用 **MessageGroup** 标记一组消息的先后顺序，**同组内** 严格按发送顺序投递。 |
| **Delay** | 定时/延时；到达延时后才对消费者可见。 |
| **Transaction** | 事务消息；保障 **本地事务与发消息** 的一致性（**4.3.0+** 已支持分布式事务消息能力）。 |

### 7.7 顺序消息的补充说明

- **不能** 依赖 **Key 负载均衡到多队列** 来做全局顺序；要保证顺序，需让相关消息进入 **同一队列**（例如同一 **MessageGroup** 对应固定队列）。
- **顺序消费重试**：某条消息消费失败进入重试时，为保证 **先后顺序**，**后续消息会卡住**，需等 **前一条** 处理完成（成功或最终跳过策略）后才会继续 —— 设计业务时要预估 **重试堆积** 风险。

### 7.8 时间与大小限制（常见默认值，以版本为准）

- **定时时间**、**消息发送时间**：由消息属性承载（具体字段名见 SDK）。
- **定时/延时**：定时时间多为 **毫秒级 Unix 时间戳**；**默认定时跨度上限** 常见为 **24 小时**（以版本为准）。
- **消息体大小**：
  - 普通、顺序消息：**约 4 MB**；
  - 事务、定时/延时消息：**约 64 KB**。

## 八、主题（Topic）与标签（Tag）

- **Topic**：消息的 **一级分类**；内部由 **多个队列** 组成，**水平扩展** 主要靠 Topic 内 **队列数与 Broker 分布**。
- **Tag**：**二级分类**；消费者可按 Tag **过滤**，**过滤在服务端完成**，减少无效网络传输；便于同一 Topic 下扩展不同消费逻辑。

### 8.1 RocketMQ 5.x：消息类型与 Topic 绑定（需预先建主题）

在 **5.x** 中，**普通 / 延时 / 顺序 / 事务** 等能力对应不同的 **`MessageType`**，需要在 **创建 Topic 时** 通过属性指定；**发送端 Topic 的 message.type 必须与消息类型一致**，否则受限。

**mqadmin 通用习惯**

- 进入安装包 **`bin`** 目录；**Linux / macOS** 多为 `./mqadmin ...`，**Windows** 多为 **`mqadmin.cmd`**。
- 几乎所有子命令都要带 **`-n <NameServer 地址>`**，格式 **`ip:port`**。
- 消息类型属性示例：**`-a +message.type=NORMAL`**（`DELAY` / `FIFO` / `TRANSACTION` 同理）。
- 队列数等业务参数可按命令支持项配置，例如部分版本支持 **`-q <queue_num>`** 指定队列数量（以 `mqadmin help updateTopic` 为准）。

**按类型创建 Topic 示例（请替换集群名、地址、Topic 名）**

| 消息类型 | 使用限制摘要 | 创建示例 |
| --- | --- | --- |
| **Normal** | 仅能在 **`MessageType=Normal`** 的 Topic 上发普通消息 | `mqadmin.cmd updateTopic -n 127.0.0.1:9876 -t TestTopic -c DefaultCluster -a +message.type=NORMAL` |
| **Delay** | 仅 **`message.type=DELAY`** 的主题；**定时状态可持久化**，故障重启后仍按原时间投递，可能 **整体晚一点** 触发 | `mqadmin updateTopic -n <nameserver> -t <topic> -c <cluster> -a +message.type=DELAY` |
| **FIFO** | 仅 **`FIFO`** 主题；常与 **单队列顺序、MessageGroup** 配合 | `mqadmin updateTopic -c DefaultCluster -t FIFOTopic -o true -n 127.0.0.1:9876 -a +message.type=FIFO` |
| **Transaction** | 仅 **`TRANSACTION`** 主题 | `mqadmin updateTopic -n localhost:9876 -t transaction_topic -c DefaultCluster -a +message.type=TRANSACTION` |

**普通消息应用场景（复习）**

- **微服务解耦、事件驱动、数据集成**：需要 **可靠传输**，对 **处理时机、全局顺序** 一般无苛刻要求时，用 Normal Topic 最合适。

## 九、订阅关系（Subscription）与消息过滤

Apache RocketMQ 的 **领域模型** 是 **发布-订阅**：**每个主题的队列** 都可以被 **多个消费者分组** 订阅；各组有 **独立的消费位点** 与 **订阅视图**。

### 9.1 订阅关系是什么

- **发布订阅模型** 下，**过滤、重试、消费进度** 等规则的配置载体。
- 以 **消费组** 粒度管理：组内统一 **过滤表达式、重试、进度恢复** 等行为。
- 除 **过滤表达式** 外，多数订阅信息 **持久化**；服务端重启或连接断开后 **订阅关系仍保留**（表达式是否持久化以版本与配置为准）。

### 9.2 消息过滤：三端分工

| 角色 | 做什么 |
| --- | --- |
| **生产者** | 发消息时预先设置 **Tag、用户属性** 等，供后续过滤。 |
| **消费者** | 在初始化及消费过程中，通过 **订阅接口** 向服务端注册：要订哪个 **Topic**、**过滤表达式** 是什么。 |
| **服务端** | 消费者 **拉取 / 推送** 时做 **动态过滤**：按消费者上报的表达式匹配，只投递 **符合条件** 的消息。 |

### 9.3 Tag 标签过滤（字符串匹配）

- **单 Tag**：表达式就是目标 Tag 字符串 → 只收该 Tag 的消息。
- **多 Tag（或）**：`Tag1||Tag2||Tag3` → 命中任一即投递。
- **全量**：表达式为 **`*`** → 该 Topic 下消息 **全部** 投递给该订阅（仍受 Topic、组隔离约束）。

### 9.4 SQL92 属性过滤

- 在 SQL 表达式里，**Tag 对应属性名** 为 **`TAGS`**（以官方文档为准）。
- 使用 **SQL92 风格** 作为过滤表达式，例如：
  - 订阅：`FilterExpression("Region IS NOT NULL AND Region='Hangzhou'", FilterExpressionType.SQL92)`
  - 订阅全部：`FilterExpression("True", FilterExpressionType.SQL92)`
- **Broker 必须开启属性过滤**，否则应用可能 **启动失败**。在 **`broker.conf`** 中增加并 **重启 Broker**：

```properties
enablePropertyFilter = true
```

```bash
nohup ./mqbroker -n localhost:9876 -c ../conf/broker.conf >../mqbroker.log 2>&1 &
```

### 9.5 示例：下单 `Trade_Topic` 与多下游

- **若各下游要处理的是「同一条业务语义、同一份内容」**：**发一条消息** 即可；**同一 Topic 下多个消费者组** 会 **各自消费一份**（组间像广播订阅）。
- **若各下游只要自己的那一类通知**：可 **发多条带不同 Tag 的定向消息**，例如 `log`（日志组）、`msg`（短信组）、`jifen`（积分）、`pay`（支付）、`wuliu`（物流）等，各组订阅 **自己的 Tag**，减少无效消费。

## 十、消费者组与消费者

### 10.1 消费者组（Consumer Group）

- **多个消费行为一致的消费者** 的 **负载均衡分组**。
- **默认约每 20s** 做一次 **重新负载均衡**（Rebalance）。
- **消费时必须指定消费者组**：同一 Topic 下可有 **多个消费者组**（组间彼此独立进度）。

| 拓扑含义 | 说明 |
| --- | --- |
| **点对点（组内）** | 常指 **集群模式** 下，**组内** 多实例 **分摊** 队列，**每条消息在组内只被消费一次**。 |
| **发布订阅（组间）** | **多个消费者组** 订同一 Topic 时，**每组各消费一份**，适合多业务线。 |

### 10.2 消费模式：集群（Clustering）与广播（Broadcasting）

通过 **`setMessageModel()`** 传入 **`MessageModel.CLUSTERING`** 或 **`MessageModel.BROADCAST`**；**未指定时默认为集群消费**。

| 对比项 | **集群消费（默认）** | **广播消费** |
| --- | --- | --- |
| **实例关系** | 同组多实例 **协作分摊** | 同组内多实例 **各自都收全量**（语义以 SDK 为准） |
| **消息次数** | 组内 **每条消息 1 次** | 组内 **每实例各消费一次**（总消费次数 > 1） |
| **重试** | **支持** | **不支持** |
| **典型场景** | **微服务多实例** 水平扩展 | **物联网**、需 **每个客户端都收到** 的通知 |
| **停机期间消息** | 客户端停止期间发到服务端的消息 **仍会按位点继续消费** | 停机期间的消息可能被 **跳过** |
| **顺序消息** | 按队列顺序策略消费 | **不支持顺序消息** |
| **重置消费位点** | 支持（以控制台/版本为准） | **不支持** |
| **控制台** | 可查询堆积、订阅关系等 | **无统一消费进度**：堆积查询、堆积报警、订阅关系查询等常 **不可用** |

### 10.3 消费者类型：Push、Simple、Pull

#### PushConsumer（推模式，SDK 托管）

- **高度封装**：业务只写 **消费监听器**；**拉取、位点提交、消费重试** 多由 **SDK 完成**（监听器返回结果后由 SDK 驱动位点提交与重试）。
- 监听器返回值：**`ConsumeResult.SUCCESS`** / **`ConsumeResult.FAILURE`**；**未捕获异常** 等会走 **重试**。
- **限制**：偏 **同步消费** 模型；存在 **全局消费超时** 等约束（见 SDK）。

#### SimpleConsumer（简单消费者，手动 ACK）

- **原子接口型**：**拉取、ACK、重试** 由业务 **主动调用**。
- 处理完成后需 **主动 ACK** 提交消费结果。
- **优势**：可 **异步消费**、**自定义超时**；**最大不可见时间** 为 **必填**，建议范围约 **10 秒 ～ 12 小时**；支持 **批量**（如单次拉取默认批次常见 **32 条**，以 SDK 为准）；可 **自行调节拉取频率**。
- **流控（参数名以版本为准）**：本地缓存超过阈值会降低拉取频率，例如：
  - `pullThresholdForQueue`（默认约 **1000** 条）
  - `pullThresholdSizeForQueue`（默认约 **100MB**）
  - `consumeConcurrentlyMaxSpan`（默认约 **2000**）

#### PullConsumer

- **仅建议在流处理框架集成** 等场景使用；一般业务优先 **Push / Simple**。

### 10.4 一致性、可靠性与幂等

- **至少一次** 投递下，消费端应 **幂等**（去重键、业务状态机、版本号等）。
- **广播** 场景下 **无重试、无位点** 等特性，更要在业务侧做 **防护**。

### 10.5 消费重试与死信队列（DLQ）

**触发重试的常见原因**

- 返回 **失败**、**抛未预期异常**。
- **处理超时**（含 PushConsumer **队列内排队超时** 等，以 SDK 为准）。

**重试行为**

- Broker 会在 **同消费组内** 做 **重新分配 / 延迟重投**；**默认最大重试次数** 常见为 **16 次**（以版本与配置为准）。

**死信队列（DLQ）**

- **超过重试上限** 仍失败的消息会进入 **死信队列**；**过期仍未消费** 的消息也可按策略进入 DLQ（以产品能力为准）。
- **命名**：与 **Group** 绑定，常见形如 **`%DLQ%<ConsumerGroup>`**（不同版本/控制台展示可能略有差异）。
- **特性摘要**：**一个 DLQ 对应一个 Group**，不是对应单个消费者实例；**若从未产生死信**，可能 **不会预先创建** 对应 DLQ Topic；该 Group 的 DLQ 可包含 **来自多个业务 Topic** 的死信。
- **保存时间**：死信数据常见 **约 3 天** 后 **自动删除**（以集群配置为准）。
- **处理建议**：起 **专门消费者** 写入日志/数据库；排查修复后可在 **控制台** 对单条死信 **重发**，让业务重新消费。

### 10.6 消费进度：位点、结果、初值与重置

- **消费位点（ConsumerOffset）**：消息 **不会** 因「消费成功」立刻从磁盘删除；服务端按 **消费组 + 队列** 记录 **已消费到的最新位点**（与 [七、7.3](#73-消费位点consumeroffset) 一致）。
- **ConsumeResult**：Push 模型下监听器返回值，表示 **本次处理成功或失败**，驱动 **是否提交位点 / 是否重试**。
- **5.x SDK 首次消费**：消费组 **首次** 启动时，位点初值常以 **服务端保存为准**，常见行为是 **从最新一条开始消费**（以官方说明为准）。

**重置消费位点**

- 在 **消息仍保留** 的时间范围内，把某组的进度 **拨到指定时刻或指定位点**，用于：**补历史**、**快速跳过堆积**、**业务回溯纠错**。
- **限制**：一般只能重置 **对消费者已可见** 的消息；**定时中、重试等待中** 等状态 **不可** 用重置解决。

### 10.7 消息轨迹（Message Trace）

- 从 **Producer → Broker → Consumer** 的 **时间、节点** 等信息组成的 **链路**，用于 **排障**。
- 默认常 **开启**；轨迹 Topic 常见为 **`RMQ_SYS_TRACE_TOPIC`**（以版本为准）。

### 10.8 回溯消费

- **已成功消费** 的消息因业务需要 **再消费一遍**：依赖 Broker **仍保留** 消息，并按 **时间（可精确到毫秒）** 或位点 **回退进度**。
- **必须** 配合消费端 **幂等**，否则重复数据会造成重复记账等问题。

### 10.9 消费提速与批量

**提高并行度**

- **增加 Topic 的 MessageQueue 数量**（建 Topic / 扩容时规划）。
- 若 **消费者实例数 < 队列数**，在同一 **ConsumerGroup** 下 **增加 Consumer 实例** 可提高并行度。
- **应急**：写 **临时分发程序** 从旧 Topic **快速读出** 写到 **新 Topic**（新 Topic **队列数放大**），再在 **新 Topic** 上起更多消费者并行处理。
- **单实例线程数**：5.x **PushConsumer** 可用 **`PushConsumerBuilder.setConsumptionThreadCount()`**；老版本 Push 常用 **`consumeThreadMin` / `consumeThreadMax`**；**SimpleConsumer** 可由业务线程池自行并发，注意 **线程安全**。

**批量消费**

- 推荐 5.x **SimpleConsumer**：每次调用指定 **批次大小**，一次拉取多条，减少 RTT。

## 十一、事务消息（两阶段、半消息与反查）

**事务消息**把 **本地数据库事务** 与 **发消息** 绑在一起，用接近 **两阶段提交（2PC）** 的思路，追求 **分布式场景下的最终一致**。可理解为：在不少业务里 **替代「本地消息表 + 定时补偿」或「最大努力通知」** 的一种内置方案（是否采用仍要看团队熟悉度与运维成本）。

### 11.1 经典两难（为什么要事务消息）

| 顺序 | 风险 |
| --- | --- |
| **先发消息再落库** | 消息已发出，本地事务失败 → **消息无法自动回滚**，下游可能读到不该生效的数据。 |
| **先落库再发消息** | 事务已提交，发消息失败 → 需 **本地消息表、定时扫描重发** 等补偿。 |

**事务消息**的目标：在 **RocketMQ 能力范围内**，让 **「库事务成功」与「消息最终可投递」** 对齐到可接受的 **最终一致**。

### 11.2 两阶段流程（概念）

1. **预备消息（半消息）**：生产者先向 Broker 发 **半消息**；Broker 写入 **事务专用存储**（内部主题 **`RMQ_SYS_TRANS_HALF_TOPIC`**、`queueId` 固定为 **0** 等实现细节），此时对 **业务消费者不可见**。
2. **执行本地事务**：例如扣款、写订单等。
3. **二次确认**：生产者根据本地事务结果向 Broker 发 **Commit** 或 **Rollback**。
   - **Commit**：半消息 **转为对目标 Topic 消费者可见**（实现上可理解为从半消息队列 **投递到业务 Topic 队列**）。
   - **Rollback**：**终止** 该事务消息流程，半消息被丢弃/回滚。
4. **发送结束请求**：部分实现里，半消息阶段结束请求可为 **Oneway**（发后即忘）；**仍以所用版本/SDK 说明为准**。

### 11.3 消息回查（Check）

当 Broker **迟迟收不到明确的 Commit/Rollback** 时，会 **回查** 生产者，由 **事务检查器（TransactionChecker）** 根据 **业务数据真实状态** 再决定结果。

**常见触发**

- 本地事务执行中 **长时间无返回**（如网络抖动），先前只能报 **Unknown**。
- 本地已决定 **Commit** 但 **二次确认请求** 因宕机等 **未到达** Broker，半消息一直挂着。

**Broker 侧定时扫描**

- 例如 **`TransactionalMessageCheckService`** 周期性扫描未完成半消息（**默认间隔** 常见为 **约 60 秒**；**上限** 不超过 **约 1 小时**，以官方为准）。
- 对每条未完成半消息，向 **对应 Producer** 发起 **回查请求**。

**生产者侧检查器**

- 通常按 **业务主键**（如订单号）查库：**已提交则 Commit，应回滚则 Rollback**。
- **进行中的事务**：回查时若事务仍在执行，应继续返回 **Unknown**，**不要** 急于 Commit/Rollback（避免误判）。
- **超时强制回滚**：超过 **半消息最大存活时间**（常见默认约 **4 小时**，**一般不可自定义**）仍无明确结果，半消息可能被 **强制回滚**。

### 11.4 事务状态（TransactionResolution）

服务端根据生产者返回的 **事务状态** 决定半消息 **提交投递** 还是 **回滚**，通常包括：**提交、回滚、未决（Unknown）**。

### 11.5 生命周期（简表）

| 阶段 | 含义 |
| --- | --- |
| **初始化** | 半消息在客户端构造完成，尚未发送。 |
| **事务待提交** | 半消息已到服务端，**单独存放**，等对本地事务结果；**对消费者不可见**。 |
| **消息回滚** | 明确回滚，流程结束。 |
| **提交待消费** | 明确提交，消息进入 **普通存储/业务 Topic**，消费者可见。 |
| **消费中** | 消费者拉取并处理；超时或失败会触发 **消费重试**（重试消息在实现上常视为 **新的投递尝试**，原消息生命周期阶段已结束）。 |
| **消费提交** | 消费者上报结果；服务端 **逻辑标记** 已处理；物理文件仍可能保留至过期或容量策略删除。 |
| **消息删除** | 按保留策略 **滚动删除** 最老数据。 |

### 11.6 使用限制（复习）

- 事务消息仅适用于 **`MessageType=Transaction`** 的 Topic（见 [第八节 8.1](#八主题topic与标签tag)）。

## 十二、参考图与架构要点

下图汇总：**总体拓扑（含 Proxy）**、**点对点模型**、**Broker 主从与刷盘**、**多消费者组 + Tag 与实践建议**。

![RocketMQ 架构与消费模型示意图](image/rocketmq/image.png)

### 12.1 总体拓扑（左上）

- **生产者 / 消费者**：多语言客户端，可走 **Remoting** 或 **gRPC**。
- **Proxy**：位于客户端与 **Broker 集群** 之间；**计算层** 常含 **协议适配、权限管理、消息管理**；向 **NameServer 集群** 做 **服务发现**，再与 Broker 交互完成存储与读取。
- **NameServer**：Broker **注册** 路由；Proxy / 客户端 **查询** 路由。
- **典型流向**：生产者 → Proxy → Broker；消费者经 Proxy 消费（**直连 Broker 的老架构/模式仍可能存在**，以实际部署为准）。

### 12.2 点对点模型（右上）

- 图示强调：**点对点** 对应 **一个 Topic 只有一个队列**、且 **只有一个消费者组**。
- **组内** 多个消费者 **分摊** 队列中的消息：**每条消息只被一个消费者处理**（负载均衡）。

### 12.3 Broker 高可用与持久化（左下）

- **Master** 承担读写；**Slave** 同步数据；Master 故障时 **Slave 可参与选主**（能否自动切换、切换后是否可写依版本与部署模式而定）。
- **同步复制 / 同步双写**：Master 在 **从库确认复制** 后再给生产者成功应答，可靠性更高。
- **异步刷盘**：写入可能先经 **页缓存**，与真正 **落盘** 之间存在时间差；是否「绝对不丢」取决于 **复制 + 刷盘** 组合策略。

### 12.4 广播订阅、Tag 与消费实践（右下）

- **同一 Topic、多个消费者组**：每组维护 **独立消费位点**，效果接近 **广播给不同业务线**。
- **Tag 过滤示例**：消费者组 A 订阅 `Topic A + Tag a`，组 B 订阅 `Topic A + Tag b`；**同一组内** 各消费者的 **Tag 订阅规则应一致**，组内 **并行消费**。
- **实践建议（图上常见总结）**：
  - 消费逻辑尽量 **幂等**；
  - 避免 **短时间大批量空拉**（浪费资源）；
  - **业务处理成功后再提交消费位点**（慎用「自动提交」）；
  - 异常时记录失败状态，用 **定时任务或重试机制** 补消费。

### 12.5 消费位点与回溯

- 服务端为 **每个消费者组、每个队列** 维护 **消费位点**；在允许的前提下可按 **位点或时间** 做 **回溯 / 重放**。
