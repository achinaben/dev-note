# 分布式锁选型

**你在做的事**：在分布式系统里需要“互斥”时，决定是用 **Redis / ZooKeeper / 数据库 / etcd** 等哪一类锁，或是否**根本不需要分布式锁**（用 DB 条件更新、唯一约束、幂等等兜底）。

**本文目标**：读完能按场景快速缩小范围，并知道各方案的**取舍与常见坑**；具体命令、Redisson 参数、看门狗细节见《分布式锁》。

**建议搭配阅读**

- 《分布式锁》（原理、Redisson/Curator、超时与续期、高并发实践）
- 《分布式理论》（CAP 等背景，按需）

## 一、选型前先问的三个问题

1. **互斥范围有多大？**  
   同一行数据、同一用户、同一资源 ID，还是全局单点任务？范围越小，越有机会用 **DB 行锁 / 条件更新** 在事务内解决，而不必单独引一层分布式锁。

2. **持锁时间多长、QPS 多高？**  
   **短临界区 + 高并发** 往往优先考虑 **Redis 类**（配合短 `waitTime`、降级、队列）；**长任务或强一致协调** 更常考虑 **ZooKeeper/etcd** 或 **DB + 任务表**，而不是长时间占 Redis 锁。

3. **正确性靠什么兜底？**  
   分布式环境下锁都可能 **抢不到、提前过期、误释放**（见《分布式锁》中“工程现实”）。最终数据一致通常还要 **唯一约束、乐观锁版本号、幂等、状态机** 等；锁是**辅助**，不是唯一防线。

## 二、方案总览与对比表

下表是**常见实践级**对比，具体产品以官方文档与集群架构为准。

| 维度 | Redis（Redisson 等） | ZooKeeper / Curator | MySQL（行锁 / `GET_LOCK`） | etcd |
| ---- | -------------------- | ------------------- | -------------------------- | ---- |
| **典型优势** | 延迟低、吞吐高、与缓存同栈 | 临时节点 + 会话，锁释放路径相对清晰 | 与业务数据同事务、实现简单 | 强一致 KV、租约（lease） |
| **典型劣势** | 依赖 TTL/续期，时钟与实现细节易踩坑 | 写路径与选举开销，高 QPS 下压力大 | DB 易成瓶颈，长事务风险 | 运维与使用门槛，偏基础设施 |
| **并发性能** | 高 | 中低（相对 Redis） | 取决于 DB 与事务设计 | 中高（视集群与用法） |
| **故障与网络分区** | 与部署模式有关；多节点需理解复制滞后 | 多数场景依赖 ZK 的协调语义 | 随 DB 高可用方案 | 与 Raft 一致性相关 |
| **常见用途** | 热点互斥、短锁、限流旁路、定时任务抢执行 | 选主、配置、中低 QPS 协调锁 | 中小流量、与账务同事务 | K8s 生态、配置与分布式协调 |

**库内组件对照**（便于落地）：Redis 侧常见 **Redisson**；ZK 侧常见 **Curator**。更多对比见《分布式锁》「选型对比：Redisson vs Curator」。

## 三、按场景怎么选

### 1）高并发、临界区很短（毫秒～秒级）

- **优先**：**Redis 分布式锁**（`tryLock(waitTime, leaseTime)` 等，避免无限等待）。
- **同时**：用 **DB 条件更新 / 幂等** 兜底“锁过期或重复执行”。
- **避免**：仅靠锁扛所有流量；应配合 **限流、降级、MQ 排队**（见《分布式锁》「分布式锁 + 消息队列」）。

### 2）定时任务 / 批处理：多实例只跑一个

- **常用**：**Redis 锁**（实现快）或 **DB  advisory lock / 任务表抢占**（与 DB 调度一体时）。
- **注意**：任务要 **幂等**；锁 holder 崩溃后依赖 **TTL 或会话失效**，可能出现短暂双跑窗口，需业务可容忍或靠幂等消除影响。

### 3）与数据库强一致同事务（同一连接内完成）

- **优先**：**单库事务 + `SELECT ... FOR UPDATE` / 乐观锁**，在**一个事务**里完成读改写。
- **分布式锁**适合跨服务、跨库或无法放在一个事务里的互斥；不要为用锁而用锁。

### 4）协调、选主、配置、中低 QPS 的“谁来做”

- **可考虑**：**ZooKeeper / etcd**；Curator 等封装成熟。
- **不太适合**：纯缓存热点、极高 QPS 的互斥（除非架构上已接受其成本）。

### 5）多 Redis 主节点、怕单点时的“RedLock”

- **事实**：RedLock 是**一种多独立 Redis 实例**上的算法思路，争议与前提较多（时钟、故障转移、与业务语义）。
- **观点/待定**：是否采用应结合团队熟悉度、SLA 与官方/社区最新讨论；**不要默认**“多节点 Redis = 一定比单集群 Redisson 更安全”。
- **工程上更常见**：接受 **单 Redis 集群 + 短 lease + 业务幂等**，或改用 **DB/ZK/etcd** 做真正强协调。

## 四、各方案要点与边界

### Redis

- **要点**：`SET NX PX` + **value 唯一标识** + **Lua 校验再删**；续期必须校验 token（见《分布式锁》看门狗章节）。
- **边界**：主从切换、时钟漂移、长 GC 导致续期不及时等，都可能影响“锁是否严格互斥”；**业务层幂等**很重要。

### ZooKeeper

- **要点**：临时顺序节点、会话断开即释放路径相对清晰。
- **边界**：写入与通知有成本；高并发下需控制抢锁频率与 watch 风暴。

### MySQL

- **要点**：应用内 **`GET_LOCK`** 或事务内 **行锁**；与业务表同事务时一致性好理解。
- **边界**：连接池、长事务、锁等待会拖垮连接与吞吐；扩展读多写少外的极端并发需拆分或换层。

### etcd

- **要点**：租约（lease）与 CAS；适合已有 K8s/etcd 栈的团队。
- **边界**：学习成本与运维；高 QPS 场景要评估集群能力与客户端重试策略。

## 五、常见误区

- **误区 1**：认为“上了分布式锁就绝对不会重复执行”。  
  **事实**：超时、重入、重试、多实例都可能重叠执行，必须 **幂等 / 约束**。

- **误区 2**：高并发下使用 **无限等待** 抢锁。  
  **事实**：易耗尽线程、打爆 Redis；应 **短 wait + 降级/排队**（见《分布式锁》）。

- **误区 3**：能用 **一行 SQL 条件更新** 解决时仍引入 Redis 锁。  
  **事实**：更简单路径往往更稳；锁多一层就多一层故障面。

- **误区 4**：把 **长耗时业务** 整段放在分布式锁里。  
  **事实**：应缩短临界区，或 **异步化 + 队列 + 状态机**。

## 六、选型检查清单

上线或评审前可快速过一遍：

- [ ] 是否可以用 **事务内行锁 / 乐观锁 / 唯一键** 替代或补强？
- [ ] 锁的 **粒度** 是否足够细（按业务 key，避免全局大锁）？
- [ ] 是否定义了 **`waitTime` / `leaseTime`**（或等价语义），避免无限等待与无限续期？
- [ ] 解锁是否 **只删自己的 token**（Redis 类必查）？
- [ ] 锁失效后，业务是否仍 **正确**（幂等、状态机、对账）？
- [ ] 是否有 **监控**：抢锁失败率、持锁时长、Redis/DB 延迟？
- [ ] 峰值时是否有 **限流 / 队列**，避免同步链路被锁拖死？
