## 分布式锁

### 为什么需要分布式锁

在单机（同一个 JVM/进程）里，线程之间可以共享内存，因此可以用 `synchronized`、`ReentrantLock` 等本地锁来做互斥。

但在分布式系统中，不同 JVM（不同进程、不同机器）之间**线程不共享内存**，本地锁无法跨进程生效，所以需要在“系统外部”（例如 Redis、ZooKeeper、数据库等）实现一把**大家都能看到**的锁，即分布式锁。

### 分布式锁的关键特性

- **排他（互斥）性**：任意时刻，只有一个客户端能持有锁（把并发变串行）。
- **安全性**：只有加锁的客户端/线程才有解锁权限（防止“别人把你的锁解了”）。
- **阻塞特性（按业务需求）**：没拿到锁就等待，直到拿到为止。
- **非阻塞特性（按业务需求）**：没拿到锁就立刻返回失败。
- **锁失效机制**：网络中断/宕机导致无法释放锁时，锁必须能自动删除，避免死锁。
- **可重入**：同一线程可以多次获得同一把锁（方法内再调用需要同一把锁的方法时，不必重复抢锁）。

### 常见实现方式

- **Redis**
  - **Redisson**
  - Redis Cluster
  - RedLock（多 Redis 实例/多主场景的一种思路）
- **ZooKeeper**
  - **Curator**
- **MySQL 行锁**

### Redis 分布式锁的基本原理

利用 Redis key 的唯一性：

- 通过 `SET key value NX EX seconds`（或等价的 setNx + 过期）实现“只有第一个 set 成功的人获得锁”
- **释放锁**时需要确保“只能删除自己加的锁”（通常用 value 保存唯一标识，配合 Lua 脚本原子校验+删除）

> 你原文的核心点：能 `setNx` 成功的客户端才获取到锁。

### 应用场景

#### 1）并发下的 CAS（Compare And Swap）风格更新

- **数据库的 CAS**：基于“读取值 + 条件更新”
  - 例：`update ... set stock = stock - 1 where id=? and stock > 0`
- **Redis 的 CAS**：基于“读取值 + 条件更新”（可配合 Lua 保证原子性）

#### 2）分布式定时任务：多实例只需要一个执行

**问题**

- 网关通常按 URL 转发；但定时任务不走 URL，不会被网关“识别/控制”，多实例部署时可能多台同时跑同一个任务。

**解决**

- 在定时任务代码的最前面加一段“抢分布式锁”的逻辑：
  - 抢到锁：执行任务
  - 没抢到：直接返回/跳过（或按需等待重试）

#### 3）分布式锁 + 消息队列：为什么常被当成「优先组合」（≠ 所有场景唯一最优）

很多团队会把 **分布式锁** 和 **MQ** 一起用，不是偶然：两者解决的问题刚好互补。

| 角色 | 典型作用 |
| ---- | -------- |
| **分布式锁** | 同步链路里 **短临界区**：把「必须互斥」的一段（查+改、占坑、去重）压在很短的 `waitTime + leaseTime` 内，避免长时间占锁拖垮线程与 Redis。 |
| **MQ** | **削峰、排队、异步最终一致**：抢锁失败或接口要快速返回时，把「必须办成」的事变成消息，由 **少量 consumer** 按队列/分片 **串行或限速** 处理；再配合 **幂等、死信、重试** 兜住可靠性。 |

**为什么说它像「最佳实践模板」**：高并发下若只靠锁，容易出现线程堆积、Redis QPS 飙高；若只靠 MQ 不设好分区键/幂等，又可能乱序或重复消费把数据写坏。**入口短锁控互斥 + MQ 扛流量与解耦**，和上文「抢锁超时 → 排队化」是同一条思路的完整版。

**什么时候不必强行「锁 + MQ」**（避免教条）：

- **单写路径已足够**：例如仅靠 **DB 条件更新 / 乐观锁版本号** + **幂等** 就能保证正确，且 QPS 可控，可以不引入 MQ。
- **MQ 自身就能串行化**：例如按 **业务主键做分区 / 单分区顺序消费**，consumer 内逻辑简单且幂等完备，有时可 **少用或不用** 分布式锁（仍要防重复消息）。
- **强一致短事务**：极小范围内用 **数据库事务 + 行锁** 即可，不必再叠 Redis 锁与 MQ。

**搭配阅读**：《消息队列》（投递语义、顺序、幂等与消费失败处理）。

**扩展单页方案**（请求侧抢锁，要对齐 **100% 上锁 · 100% 受理**、排队/重试与验收指标；零节备注写清「是否真能叫两个 100%」）：**分布式锁-请求侧上锁与受理方案**（文中「100%」指 **带前提的工程口径**，非数学绝对。）

### 选型对比：Redisson vs Curator


| 维度      | Redisson（Redis）    | Curator（ZooKeeper）              |
| ------- | ------------------ | ------------------------------- |
| 并发性能    | 支持高并发、效率高、性能开销低    | 不支持高并发、效率低、性能开销高（需要频繁磁盘读写等开销）   |
| 安全性/可靠性 | 相对较低：常依赖“过期释放”避免死锁 | 相对较高：可通过节点/会话机制，监听到领导节点宕机后更快释放锁 |
| 适用场景    | 高并发场景优先            | 可靠性要求极高且并发不高的场景                 |


### 实践建议（你原文的注意事项落地）

- **处理超时时间（业务执行超时）**：避免拿到锁后业务卡死导致长时间占锁
  - 建议记录“处理超时事件”，必要时人工介入排查/补偿
- **等待超时时间（抢锁等待超时）**：避免过多线程一直阻塞
  - 可结合“定时重试/退避重试”策略

#### 看门狗续期 vs 过期释放锁（是否冲突？）

**不冲突**，它们是配合关系，但有一个前提：**续期只能由“当前持锁者”完成**。

- **过期释放锁（TTL）**：兜底机制。进程宕机/网络中断/忘记解锁时，靠 TTL 自动释放，防止死锁。
- **看门狗续期（自动续期）**：防止“业务执行时间 > 初始 TTL”时锁提前过期被别人抢走；只要持锁者还活着且仍在执行，就持续把 TTL 往后延。

**常见坑（看起来像冲突，本质是实现不安全）**

- **续期不校验持锁者身份**：如果仅对 key 直接 `EXPIRE`，当锁已过期并被别人抢到后，老持锁者仍可能把“新持锁者的锁”续长，造成严重问题。
  - 正确做法：锁 value 存唯一 `token`，续期/解锁都必须先校验 `value == token`，并且要原子执行（通常用 Lua）。
- **业务卡死但进程未挂**：看门狗可能持续续期导致锁长期不释放。
  - 工程兜底：设置**最大持锁时间上限**、业务超时熔断、监控告警与人工介入流程。

#### Redisson：Watch Dog、等待时间、重试、无限等待（四个容易混的点）

Redisson 里这几个概念分别解决“不同阶段”的问题：**拿到锁之后怎么不误过期**，以及**没拿到锁时最多等多久、怎么等**。


| 特性                   | 作用阶段           | 解决的问题            | 典型用法/参数                               |
| -------------------- | -------------- | ---------------- | ------------------------------------- |
| 看门狗（Watch Dog）       | **持锁阶段**（已获取锁） | 自动续期，避免业务没做完锁就过期 | `lock.lock()`（无参）或 `leaseTime = -1`   |
| 抢锁等待时间（waitTime）     | **抢锁阶段**（未获取锁） | 最多等多久，超时就放弃      | `tryLock(waitTime, ...)` 的 `waitTime` |
| 重试机制（retry interval） | **抢锁阶段**（未获取锁） | 等待期间按什么频率重试抢锁    | `LockOptions` / 配置项                   |
| 无限等待                 | **抢锁阶段**（未获取锁） | 一直等到拿到锁（或被中断）    | `lock.lock()` 或 `waitTime = -1`       |


##### 1）看门狗（Watch Dog）：持锁阶段的“自动续命”

- **触发条件**：使用 `lock.lock()`（无参）或显式 `leaseTime = -1`。
- **核心行为**（默认值是“典型情况”，可配置）：
  - 初始锁过期时间默认约 **30s**
  - 持锁线程会按一定周期（常见是 lockWatchdogTimeout/3）自动续期，把 TTL 往后延
  - **持锁线程崩溃**时，看门狗停止，锁会在 TTL 到期后自动释放（兜底）

##### 2）抢锁等待时间（waitTime）：控制“最多等多久”

- 使用 `tryLock(waitTime, leaseTime, unit)` 时：
  - 在 **waitTime** 内没抢到锁就返回 `false`（方便做降级/快速失败）
  - 一旦你显式传了 **leaseTime**（非 -1），一般就意味着“锁持有时间可控”，**不会走看门狗自动续期**

##### 3）重试机制：控制“等待期间怎么抢”

- 默认会在等待时间内周期性重试（间隔可配置）。
- 高并发时，重试太频繁会导致 Redis 压力上升；可以适当**调大重试间隔**。

##### 4）无限等待：能用，但要非常谨慎

- **风险**：高并发下，大量线程长期阻塞等待锁，容易把线程池耗尽，进而把服务拖垮。

#### 常见组合与适用场景（背诵版）


| 组合                  | 适用业务                 | 示例                                  |
| ------------------- | -------------------- | ----------------------------------- |
| 看门狗 + 无限等待          | 业务时长不确定，但必须执行        | `lock.lock()`                       |
| 无看门狗 + 有限等待 +（可控重试） | 允许失败降级，高并发更常用        | `tryLock(waitTime, leaseTime, ...)` |
| 无看门狗 + 无限等待         | 业务时长确定且必须执行，但不希望自动续期 | `tryLock(-1, leaseTime, ...)`       |


#### 高并发场景：为什么更建议“显式设置 waitTime + leaseTime”

你这段结论非常关键，可以总结成三类风险：

- **线程耗尽风险**：无参 `lock()` 会让抢锁线程无限等待，高并发下线程大量堆积。
- **锁持有时间不可控风险**：依赖看门狗续期会引入不确定性（如 GC 长时间停顿、网络抖动导致续期失败），可能出现锁提前过期，进而产生并发安全问题（例如超卖/重复执行）。
- **Redis 压力风险**：大量线程等待+频繁重试，会显著增加 Redis QPS，极端情况下拖垮 Redis。

#### 推荐实践（高并发模板）

- **waitTime（抢锁等待上限）**：通常建议在 **500ms~3s** 之间（看业务可接受延迟与降级策略）。
- **leaseTime（持锁上限）**：建议按“**业务预估耗时的 2~3 倍**”设置，留缓冲，但不要无限。
- **重试间隔**：必要时调大（例如 200ms/500ms），降低 Redis 压力。
- **兜底**：限流/排队削峰（令牌桶、队列）、监控锁等待时间与失败率、告警与人工介入流程。

#### 你提供的真实代码片段：关键点提醒

你这段代码里使用了 `lock.tryLock(1, TimeUnit.SECONDS)`（只传了等待时间），通常意味着：

- **最多等 1 秒**拿锁，拿不到就跳过（这是好习惯：避免无限等待）
- 但因为没有显式传 `leaseTime`，在 Redisson 语义上更接近“拿到锁后由看门狗续期”（持锁时间不由你固定上限）

如果这是**高并发核心链路**（比如扣库存、创建订单等），更稳的写法通常是显式传入 `leaseTime`，让锁“最长持有多久”可控：`tryLock(waitTime, leaseTime, unit)`，并确保 `leaseTime > 实际业务耗时`。

#### 业务中：持锁超时 vs 抢锁超时怎么处理（落地版）

把“超时”分成两类看，会清晰很多：

##### 1）持锁超时（业务执行超时 / leaseTime 到期）

**目标**：既要避免死锁（永远不释放），也要避免锁提前过期导致并发安全问题。

- **推荐策略（优先级从高到低）**
  - **显式设置 `leaseTime`（推荐）**：让锁最长持有时间可控，到期自动释放（通常不依赖看门狗）。
    - 经验：`leaseTime ≈ 业务 P99 耗时 × 2~3`（留缓冲，但不要无限大）
  - **必须用看门狗时，加“最大业务时长上限”**：超过阈值就中断/终止/拆分任务，否则可能一直续期导致长期占锁。
  - **把任务拆短**：把长事务拆成多个小步骤（每步都能在可控时间内完成），减少“锁要持有很久”的需求。
- **超时发生后的处理（非常关键）**
  - **可观测**：记录并上报超时事件（锁 key、业务主键、开始/结束时间、耗时、线程/实例、是否续期、是否成功解锁）。
  - **可补偿**：进入补偿流程（补偿表/延迟队列/MQ），必要时人工介入。
  - **可重复执行（幂等）**：即使锁因为到期被别人拿到导致“重复执行”，也不会把数据做坏（唯一约束/状态机/去重表等）。

##### 2）抢锁超时（waitTime 到期仍未获取锁）

**目标**：避免线程无限等待耗尽线程池；拿不到锁时要有“退路”。

- **推荐策略**
  - **快速失败 + 降级（高并发最常用）**：`tryLock(waitTime, ...)` 超时就返回失败，走降级逻辑（“系统繁忙请稍后再试”、返回缓存、只读模式等）。
  - **排队化（削峰填谷）**：抢不到锁就把请求入队（MQ/本地队列），后台少量 worker 串行/限速处理，避免同步链路被锁拖死。
  - **有限重试（带退避）**：只重试少量次数，间隔递增（例如 100ms→300ms→800ms），防止所有请求同时重试打爆 Redis。

##### 3）一句话决策口诀

- **必须成功的任务**：允许等，但要有**最大等待**、最好**排队**，并保证可观测与补偿。
- **允许失败的请求（高并发热点）**：**短 waitTime + 快速失败/降级**，把系统吞吐守住。

##### 4）抢锁 / 持锁是否“一定允许失败”？业务有没有“不允许失败”？

**工程现实（事实）**：在分布式环境下，**没有**“永不失败”的锁抽象。

- **抢锁**：竞争、Redis/网络抖动、`waitTime` 到期等，都可能导致**拿不到锁**（`tryLock` 返回 `false` 是正常、要显式处理的结局）。
- **持锁**：TTL 到期、续期失败、进程崩溃导致看门狗停止、实现不安全导致“误续别人的锁”等，都可能导致**锁提前失效**或**并发重叠**；因此业务不能假设“只要曾经抢到就永远安全”。

**业务语义（常见实践）**：产品说的“不允许失败”，通常指**对用户的最终结果必须办成**（或账务必须一致），而不是指 **`tryLock` 永远不能为 false**。

| 业务诉求 | 含义 | 常见落地（与口诀对应） |
| -------- | ---- | ---------------------- |
| 允许失败（高并发读多写少、可重试提示） | 拿不到锁可以接受 | 短 `waitTime` + 快速失败/降级/缓存 |
| 必须成功（结算、批处理、关键入账） | 最终要执行且数据不能乱 | **排队**（MQ/任务表）、**更长但有上限的等待**、**可观测 + 补偿**；关键正确性往往还依赖 **DB 条件更新 / 唯一约束 / 幂等**（见《接口幂等性处理》），而不是单靠锁“包打天下” |

**一句话**：锁层**允许**抢不到、持不住；业务层若“不允许失败”，用**队列与上限等待 + 幂等与约束**兜住最终结果。

#### Redisson：能否用“中间件配置”自动重试？

先把“自动重试”分清楚：**Redisson 里至少有两类重试**，很多人会混在一起理解。

##### 1）抢锁阶段的重试（业务想要的：拿不到锁就再抢）

**结论**：通常**不能只靠一份 Redisson 全局配置**就把所有业务的“抢锁等待多久、怎么重试”统一好；更常见、也更推荐的做法是：

- 在代码里使用 `tryLock(waitTime, ...)` 或 `LockOptions` 来控制等待时间与重试节奏
- 但为了做到“像中间件配置一样统一”，可以在项目里做一层**统一封装**，把默认参数从配置文件读取

**推荐落地（最实用）**

- 封装一个 `DistributedLockService` / `LockTemplate`
  - 默认 `waitTime / leaseTime / retryInterval` 由 `application.yml` 配置
  - 业务方只调用一行：`executeWithLock(key, () -> { ... })`
- 这样既能统一策略（像中间件配置），又能按场景覆盖默认值（少数特殊业务单独传参）

##### 2）Redis 命令层的重试（Redisson Client 的网络/超时重试）

**结论**：这类重试是可以在 Redisson `Config` 里统一配置的（例如重试次数、重试间隔等），但它解决的是：

- Redis 网络抖动、超时导致的**命令失败重试**

它**不等价于**“业务层抢锁没抢到，所以再抢一次”的策略。

##### 一句话记忆

- **抢锁重试**：更偏业务策略，推荐“`tryLock/LockOptions` + 统一封装 + 配置下发”
- **命令重试**：更偏客户端容错，在 Redisson `Config` 里全局配置即可
