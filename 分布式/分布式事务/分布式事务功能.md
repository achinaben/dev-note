# 分布式事务功能（能力速览）

> 本文从「能做什么、做到什么程度、各模式差异」角度归纳分布式事务相关能力，与选型决策、落地清单配套使用。

---

## 1、要解决的业务问题（功能目标）

| 目标 | 说明 |
| --- | --- |
| **跨资源原子提交** | 多个库/多个服务的写操作，对外表现为「要么全成功、要么全失败」。 |
| **写隔离与防脏写** | 并发下避免读到/写出不可接受的中间结果（强一致链路尤其重要）。 |
| **故障后可恢复** | 协调者或参与者宕机、网络抖动后，能基于日志继续提交、回滚或补偿。 |
| **异步场景最终对齐** | 允许短暂中间态时，通过消息、重试、幂等、对账把多系统数据最终拉齐。 |

---

## 2、按方案划分的核心功能

### 2.1 Seata 整体能力

- **全局协调（TC）**：`seata-server` 作为协调者，配合注册/配置中心，管理全局事务与各分支的提交、回滚。
- **多模式**：**AT、TCC、SAGA、XA**，覆盖从「低侵入强一致」到「可补偿最终一致」的链路。
- **跨库分布式事务**：把多数据源上的写操作纳入同一全局事务语义。
- **典型隔离**：AT 场景下常对标 **RC（Read Committed）** 级别思路；写隔离依赖 **全局锁 + Undo** 等机制（AT）。
- **宕机恢复**：对「提交中/回滚中」等状态，依赖事务日志与 TC 能力做自动恢复与推进（具体行为以版本与配置为准）。

### 2.2 AT 模式功能要点

- **自动回滚**：一阶段在本地事务内写业务数据并记 **Undo Log**；二阶段回滚时用快照生成**反向 SQL**。
- **写隔离（全局锁）**：本地提交前需拿到全局锁，降低并发脏写风险；代价是锁等待、超时与热点冲突。
- **适用边界**：以**关系型库 + JDBC + 本地 ACID** 为前提；**纯 HTTP/API 无库写**时 AT 无法直接覆盖，需 TCC/SAGA/MQ 等。
- **二阶段提交异步化**：全局提交可较快返回，Undo 清理等可异步批量进行（减轻同步路径压力）。

### 2.3 TCC 模式功能要点

- **业务自定义两阶段**：**Try（预留）→ Confirm（确认生效）/ Cancel（释放预留）**，不依赖数据源代理与 Undo Log。
- **覆盖纯 API**：只要能设计「预留/确认/取消」，即使没有库写也可追求强一致语义。
- **资源冻结语义**：适合库存、余额、券等「先占住再决定」的场景。
- **框架侧要求**：Confirm/Cancel **可重试** → 必须 **幂等、空回滚、防悬挂**，并建议 **落表**（`tx_id`、`tx_state`、`expire_time` 等）。

### 2.4 SAGA 模式功能要点

- **正向本地提交 + 反向补偿**：各步先各自提交；失败则按约定顺序调用**补偿**，把整体「撤回到可接受状态」。
- **长流程 / 外部系统友好**：适合无法提供 TCC 三接口的外部公司或遗留系统，偏**编排**而非单库强隔离事务。
- **能力侧**：跨库跨服务的流程一致性、失败重试与最大努力推进、补偿与事务日志持久化（实现差异大）。
- **语义边界**：全局多为**最终一致**，存在**中间态**；隔离弱于传统 ACID，依赖业务幂等、状态机与去重。

### 2.5 XA / 2PC 类功能要点

- **标准 2PC 语义**：Prepare 投票 → 全员 Yes 则 Commit，否则 Rollback；**原子性**强。
- **XA 接口**：TM 与 RM（库）之间的标准协议，工程上常通过 **JTA + XA 数据源** 或 **Seata XA 模式**接入。
- **能力**：跨库 XA（链路组件均需支持 XA）、与主流事务管理器/连接池整合（视框架而定）。
- **代价**：锁持有时间长、高并发下易积压；依赖驱动与库的 XA 支持。

### 2.6 MQ 最终一致性（非 Seata）功能要点

- **解耦与吞吐**：异步投递，核心链路不被慢下游拖死；适合通知、积分、报表、对接外部等。
- **可靠对齐「库 ⇄ 消息」**：**Outbox（同事务写业务表+事件表）**、**事务消息（半消息+反查）**、**CDC** 等形态。
- **消费侧**：**重试 + 上限 + DLQ/DLT**、**幂等去重**、**对账/补偿任务** 形成闭环。
- **语义**：**最终一致**，默认接受**中间状态**；Seata 全局事务**不会**因异步消费失败而自动回滚已提交的生产者事务。

## 3、协议层功能对照（2PC / 3PC）

| 能力 | 2PC | 3PC |
| --- | --- | --- |
| 阶段划分 | Prepare → Commit/Rollback | CanCommit → PreCommit → DoCommit（及中止路径） |
| 阻塞缓解 | 协调者故障可能长期阻塞 | 部分阶段引入超时与默认动作 |
| 原子提交 | 支持 | 支持（实现与假设更复杂） |

**记忆**：3PC 在 **PreCommit 成功后**，参与者等不到 DoCommit 时，常见约定下可**超时默认提交**，减少无限等待；更早阶段超时则多走向中止。

## 4、能力矩阵：AT vs TCC vs SAGA vs MQ

| 方案 | 一致性（对外观感） | 业务侵入 | 主要依赖/前提 | 强项 | 典型代价 |
| --- | --- | --- | --- | --- | --- |
| **AT** | 强一致（跨库写友好） | 低～中 | 代理数据源、RDBMS、Undo、全局锁 | CRUD 链路改造小 | 全局锁冲突、长事务、慢 SQL/外部调用放大问题 |
| **TCC** | 强一致（可含纯 API） | 高 | Try/Confirm/Cancel、可预留资源 | 不绑数据源代理、可控性强 | 幂等与乱序、冻结降吞吐、实现复杂 |
| **SAGA** | 最终一致（补偿） | 中～高 | 每步有补偿、允许中间态 | 长流程、外部系统多 | 补偿不可逆、隔离弱、需状态机与幂等 |
| **MQ 最终一致** | 最终一致 | 高 | Outbox/可靠投递、重试、幂等、DLQ | 高并发、解耦 | 中间态治理、工程化要求高 |

**口诀**：**能 AT 则 AT**；要**冻结/预留**用 **TCC**；**链路长或外围异步**用 **SAGA** 或 **MQ**。

## 5、横切能力（任何方案都应具备）

- **可观测**：全局 `xid` 或业务 `tx_id` 贯穿日志与链路；失败可告警、可查询。
- **超时与重试策略统一**：避免一侧已放弃、另一侧仍在执行。
- **幂等**：提交、回滚、补偿、消息消费均可能重复触发。
- **数据模型**：关键表建议具备状态机字段（如 `tx_state`、`version`、`updated_at`），便于并发控制与排障。

## 6、功能与场景的快速对应

| 场景 | 更匹配的能力组合 |
| --- | --- |
| 下单 + 扣库存 + 扣款，不允许中间态 | Seata **AT**（或资金/库存冻结用 **TCC**） |
| 券/库存「先占后付」、支付失败释放 | **TCC**（Try/Confirm/Cancel + 超时解冻） |
| 多系统长流程、外部只能普通接口 | **SAGA** 或 **MQ** + 补偿 |
| 支付成功后的短信、积分、数仓 | **本地事务 + Outbox + MQ**，消费幂等 + DLQ |
| 秒杀极高并发、可接受「预扣 + 异步」 | 往往弱化同步分布式事务，改用 **预扣 + MQ** 等 |

## 7、最小能力验收（避免「装了框架但没闭环」）

**强一致（AT/TCC/XA）**

- 全局事务内无长耗时外部调用；热点资源有冲突预案。
- 日志可定位 XID/分支失败原因。

**最终一致（MQ + Outbox）**

- 业务与事件**同一本地事务**写入；投递失败可持久化重试。
- 消费端幂等；重试有上限；失败进死信并可人工/补偿处理。
- 核心业务有**对账或扫描补齐**兜底。
