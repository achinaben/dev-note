## 一、一句话决策

- **钱、库存、核心订单状态——不能长期停在「半成功半失败」** → 优先考虑 **Seata（AT/TCC/XA）** 或等价强一致手段；代价是性能、锁与复杂度。
- **通知、积分、报表、对接外部系统、可接受秒级～分钟级对齐** → 优先考虑 **本地事务 + MQ（Outbox / 事务消息 / CDC）+ 重试 + 幂等 + 死信/对账**。
- **流程很长、外部系统多、难以 2PC/TCC** → 看 **SAGA（正向 + 补偿）**；接受中间态，靠补偿与幂等兜底。

## 二、总览：场景 → 方案速查表

| 你在解决的业务 | 更常选的主方案 | 典型模式 / 备注 |
| --- | --- | --- |
| 跨服务跨库：下单 + 扣库存 + 支付/余额，要求同步「全成或全败」 | Seata | **AT**（多为 CRUD）；热点秒杀另议 |
| 转账、退款、余额与订单状态必须同时正确 | Seata | **AT** 或 **TCC**（看是否需冻结/预留） |
| 限量秒杀：下单成功必须等于库存扣减成功，不能超卖（同步口径） | Seata | **AT**（注意热点全局锁）；或「预扣 + 异步」见下文 |
| 券/库存「先占住、支付失败再释放」 | Seata | **TCC**（Try 冻结 / Confirm / Cancel） |
| 纯 API、无库可代理，但要预留/确认/取消资源 | Seata | **TCC**（不依赖 AT 数据源代理） |
| 长流程、多外部公司/遗留系统、只有正向接口+补偿 | Seata / 自研编排 | **SAGA**（最终一致 + 补偿幂等） |
| 强依赖数据库 XA、跨库 2PC 语义 | Seata | **XA**（驱动/库需支持 XA） |
| 订单创建后的短信、积分、审计、数仓同步 | MQ 最终一致 | **Outbox**（或事务消息/CDC）+ 消费幂等 |
| 对接 ERP、物流：慢、不稳定，不能拖死下单接口 | MQ 最终一致 | 异步 + 重试 + 补偿任务 |
| 超高并发秒杀：全局锁成为瓶颈 | 慎用全链路 Seata | 「预扣库存 + 待支付 + MQ + 超时释放」等 |
| 支付成功后的发货、加积分、短信（可晚一点但必须做完） | MQ 最终一致 | **Outbox** + DLQ + **对账兜底** |

## 三、强一致 / 原子提交类场景（Seata 为主）

### 3.1 Seata AT

**适合**

- 多个微服务、多套库上的 **典型数据库写操作**（增删改查），希望 **低侵入**：用数据源代理 + UndoLog 做回滚。
- **订单创建 + 扣库存 + 支付/扣款** 这类链路：要求接口返回成功时，几方数据已对齐，不要出现「单有了库存没扣」的长期中间态。
- **内部微服务** 为主、外部慢系统未塞进全局事务时（全局事务里不要塞 HTTP 短信、物流、第三方支付长调用）。

**不适合或要慎用**

- **长事务**：慢 SQL、外部 HTTP、大段业务逻辑放在 `@GlobalTransactional` 里 → 锁持有久、冲突放大。
- **热点行 / 热点 SKU**：全局锁竞争剧烈时，AT 可能成为瓶颈，需拆分、降级或改「预扣 + 异步」路线。

### 3.2 Seata XA

**适合**

- 需要 **数据库原生 2PC/XA** 语义、相关组件链路透传 XA 的场景（性能通常弱于 AT，但语义贴近传统 XA）。

**前提**

- 数据库与驱动支持 **XA**；接受更长锁持有与协调开销。

### 3.3 Seata TCC

**适合**

- **库存、余额、优惠券** 等：需要先 **冻结/预留**，支付成功再 **确认扣减**，失败或超时则 **释放**。
- **跨服务强一致**，且存在 **纯 API** 参与方（AT 无法代理其库时），只要能设计出 Try / Confirm / Cancel。

**必须配套**

- Try/Confirm/Cancel **全幂等**、**空回滚**、**防悬挂**、冻结 **过期时间** + 定时释放。

### 3.4 SAGA（含 Seata SAGA）

**适合**

- **步骤多、耗时长** 的业务流程；参与者含 **外部公司、遗留系统**，往往无法提供完整 TCC 三接口。
- 更偏 **服务编排 / API 编排**：每步本地先提交，失败则按相反顺序 **补偿**。

**要接受**

- 全局上多为 **最终一致**，存在 **中间状态**；补偿与 **幂等、状态机** 必须设计到位。

### 3.5 与「本地 Spring 事务」的关系（订单域）

- **跨服务** 时，`@Transactional` 传播 **无法跨进程**；要「多库多服务一起提交/回滚」，需要 **全局事务（如 Seata）**，而不是仅靠 `REQUIRED` / `REQUIRES_NEW`。

## 四、最终一致类场景（MQ + Outbox 为主）

### 4.1 典型业务

- 订单核心已完成后：**短信、推送、积分、报表、审计日志**。
- **支付成功** 后的：发货单、加积分、营销通知、同步数仓（允许短暂延迟，但最终完成）。
- **跨系统异步集成**：ERP、物流、第三方回调后的内部处理；对方慢或不稳时用消息 **削峰、解耦**。

### 4.2 工程形态（选型口诀）

- **落库与发消息要对齐、进程重启还要继续发** → **Outbox（同事务写业务表 + 事件表）**、或 **事务消息（如 RocketMQ）**、或 **CDC**。
- **只关心 Broker 尽量不丢、可接受业务侧不建表** → 生产者确认 + 持久化 + 消费成功后 ACK + **消费端幂等**（视业务容忍度而定）。

### 4.3 必须具备的能力

- **重试 + 上限 + 死信（DLQ/DLT）**；**消费幂等**（`event_id` 或 `biz_key + event_type` 落库去重）；核心结果建议 **对账/补偿任务** 扫表补齐。

## 五、混合场景（核心同步 + 周边异步）

**常见模式（订单）**

1. **核心三步**（创单、扣库存、创建支付/扣款）用 **Seata AT**（或 TCC）：任一失败则 **全局回滚**。
2. **成功后** 再发 **MQ**（建议 **Outbox** 同事务写）：短信、积分、数仓等由消费者 **异步、可重试** 完成。

**边界**

- Seata **不会**因为 **MQ 消费失败** 而自动回滚生产者已提交事务；异步链靠 **幂等 + 重试 + DLQ + 对账**，不要假设「消费失败 = 上游自动回滚」。

## 六、明确不适合 / 要慎用的场景

| 情况 | 建议 |
| --- | --- |
| 把日志服务、非关键旁路强行纳入 Seata 全局事务 | 通常 **不要**；旁路用异步或最终一致 |
| 在全局事务里同步调用外部支付、物流、短信 | **避免**；易变长事务，拖垮整条链 |
| 需要「DB 与 MQ 在同一 XA 里」才算放心 | 认清多数项目 **没有** 真·跨 DB+MQ 的 XA；用 **Outbox** 或事务消息建模 |
| 发送失败靠「抛异常回滚本地库」且 MQ 常抖动 | 核心接口 **成功率/RT** 会被 MQ 绑架；高可用场景优先 **先落库再异步投递** |
| 开启消费重试但不做幂等 | **禁止**；重复消费会越补越错 |

## 七、决策 Checklist（上线前自问）

- **是否允许中间状态对外可见或可存在一段时间？**  
  - 允许 → 倾向 **MQ / SAGA**；不允许 → 倾向 **Seata AT/TCC/XA**。
- **性能是否极端敏感（如大促秒杀）？**  
  - 是 → 慎用长链路 **AT**；评估 **预扣 + 消息 + 超时释放** 或分片与限流。
- **是否跨多个不可控外部系统？**  
  - 是 → 核心同步尽量短，外部用 **MQ** 或 **SAGA 补偿**，避免同步拖死。
- **主路径是 CRUD 跨库还是资源冻结/纯 API？**  
  - CRUD → **AT**；冻结/预留 → **TCC**；长流程补偿 → **SAGA**。
- **异步链是否已具备：Outbox 或等价可靠投递、幂等、重试上限、DLQ、对账？**  
  - 缺任一 → 先补再谈「最终一致」。
