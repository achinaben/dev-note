### 0、一页速记（背这个就够用）

#### 0.1 先做选型（决策树）

- **你能不能接受“中间状态”？**
  - **不能**（钱/库存/核心订单状态）→ 选 **Seata 强一致链路**（AT/TCC/XA）
  - **能**（通知/积分/报表/对接外部系统）→ 选 **MQ 最终一致（Outbox + 重试 + 幂等）**
- **你要管的是“数据库写一致”还是“纯 API 调用一致”？**
  - **数据库写一致为主（CRUD）** → 优先 **AT**
  - **要冻结/预留资源（库存/券/余额），或纯 API 也要强一致** → **TCC**
  - **流程很长/外部系统多，允许回滚靠补偿** → **SAGA**

#### 0.2 最小落地清单（不做会翻车）

- **强一致（AT/TCC/XA）最小清单**
  - **避免长事务**：不要把慢 SQL、外部 HTTP、长耗时逻辑放进全局事务
  - **可观测**：日志必须能定位 **XID/BranchId** 与失败原因
  - **并发冲突预案**：热点行/热点 SKU 的锁冲突要有降级或拆分策略
- **最终一致（MQ + Outbox）最小清单**
  - **Outbox 事件表**：业务数据与事件在**同一个本地事务**里写入
  - **重试 + 上限 + 死信（DLQ/DLT）**：失败可重放、可人工介入
  - **幂等去重**：消费端用 `event_id` 或 `biz_key+event_type` 做唯一约束（落库去重）
  - **对账/补偿**：对核心结果（支付/发货）要能定时扫表补齐

### 1、核心区别与适用场景

| 维度 | Seata 分布式事务 | MQ 最终一致性（基于消息队列） |
| --- | --- | --- |
| 一致性保证 | 强一致性（AT/TCC）或最终一致性（SAGA） | 最终一致性（异步补偿） |
| 性能 | 较低（需全局锁、网络交互多） | 较高（异步通信，无锁竞争） |
| 业务侵入性 | 中（AT 侵入性低，TCC/SAGA 需改代码） | 高（需设计消息表、补偿/重试逻辑） |
| 适用场景 | 核心流程、实时性要求高、数据强一致 | 非核心流程、可接受短暂不一致、高并发场景 |

#### 1.1、FAQ：必须组合 Seata + MQ 还是可以二选一？

- **一句话结论**：**不强制组合**；多数情况是**二选一**（同一类一致性需求选一种主方案），或**各管一段**（同步核心用 Seata，异步周边用 MQ）。详见下文 **「6、混合使用策略」**。
- **为什么不是“成套必装”**
  - **Seata**：把多服务/多库的写操作，用全局事务协调成**尽量同时成功或同时回滚**（AT/TCC/XA 等），偏**同步、强一致**。
  - **MQ + Outbox/重试/幂等**：用消息把步骤拆开，偏**异步、最终一致**。
  - 二者解决的是**不同风格**的一致性，没有“用了 Seata 就必须再上 MQ”的产品规定。
- **什么时候更像二选一**
  - **不能容忍钱/库存/核心订单的中间态**、链路要同步走完 → 主方案常选 **Seata（或 TCC）**，这条链**未必**需要 MQ。
  - **允许短暂不一致**、通知/积分/对账等可异步 → 主方案常选 **本地事务 + MQ（Outbox）**，**未必**需要 Seata。
- **什么时候会“组合”（仍是分工，不是绑死）**
  - **同一订单链路里**：核心几步用 **Seata** 保证一起提交/回滚；成功后发 **MQ** 做短信、积分、数仓等（见 **第 6 节**）。
  - **公司级架构**：A 业务以 MQ 为主，B 业务极少数链路用 Seata——也是常见搭配。
- **类比（帮助记忆）**：Seata 像几个人**同时举手表决**“全成或全不成”；MQ 最终一致像**先办完自己的事再寄信**，对方**稍后再办**。

#### 常见误区：Seata 会因为 MQ 消费失败而回滚吗？

- **结论（直接背）**：如果你说的“消费”是 **MQ 消费者**，那 **Seata 不会因为消费失败就自动回滚生产者已提交的本地事务**。
- **原因（为什么）**：MQ 消费一般是**异步链路**，生产者侧事务往往已经提交；消费者处理也通常**不在同一个 Seata 全局事务（XID）**里，所以不存在“失败就全局回滚”的自动联动。
- **正确姿势（怎么兜底）**：MQ 场景要靠 **Outbox/可靠投递 + 消费幂等（落库去重） + 重试/上限 + 死信（DLQ）+ 补偿/对账** 形成闭环，而不是指望回滚生产者。

#### 1.2、FAQ：写库与发 MQ（消息表是否必须、重试层级、发送失败能否抛异常回滚）

> 与《消息「100% 不丢失」方案》互补：此处从 **分布式事务 / 最终一致** 视角收束决策；MQ 全链路落地与验收以前者为准。

##### Q1：实现「消息不丢」或写库与通知一致，是不是一定要用数据库消息表？

- **结论**：**不强制**。消息表（Outbox / 本地消息表）解决的是 **「本地库已提交」与「通知最终一定会发到 MQ」** 这类 **双写一致** 问题；若你的链路里没有这种绑定，就不必为「不丢」单独上表。
- **常见分流**：
  - **必须可靠对齐「落库 ⇄ 发消息」**：优先 **Outbox（同事务写业务表 + 消息表）**、或 **RocketMQ 等事务消息（半消息 + 反查）**、或 **CDC 从库表变更投递**（等价于“以库为准再发”）。
  - **只关心生产者 → Broker → 消费者尽量不丢**（埋点、日志、可丢业务）：靠 **发送确认、持久化/副本、消费成功后再 ACK、幂等** 即可，**不强制**消息表。

##### Q2：发送失败时做重试，是不是就得用数据库消息表？

- **结论**：**分两层，不要混为一谈**。
  - **生产者 SDK 内重试**（消息仍在客户端内存/缓冲）：**不需要**消息表；能缓解瞬时网络/Broker 抖动，但 **进程崩溃** 时，未确认成功的待发可能丢失。
  - **需要「重启后仍要继续发」的持久化重试**：必须把「还有待发」记在 **磁盘** 上——**数据库消息表 + 投递器** 是常见做法；**事务消息**、**CDC** 等是另一些持久化形态，**不等于**必须用一张 Outbox 表，但 **不能**只靠内存重试扛重启。

##### Q3：消息发送失败时，能否直接抛异常，触发「全局事务」回滚，让接口返回失败？

- **结论**：**可以作为一种工程选择，但通常不当成默认方案**；且要认清 **回滚范围**。
  - **若** `发 MQ` 发生在 **本地库 `COMMIT` 之前**，失败抛异常可使 **本地业务事务回滚**，从调用方看是 **整单失败**，能避免「库里有单、消息没发出」——**技术上成立**。
  - **常见误区**：多数项目 **没有** 跨 **DB + MQ** 的 XA/2PC「真·全局事务」；异常回滚的通常是 **本地数据库**，**不会**自动撤销 Broker 上已成功的写入。若出现 **发送超时但实际已成功**，回滚后用户重试可能 **重复**，必须 **幂等**。
  - **代价**：MQ 不稳或变慢时，**核心写接口成功率与 RT** 会被强烈拖累（可用性绑在 MQ 上）。要高可用、削峰时，更常见 **Outbox**（先提交库 + 表内记待发，异步投递）或 **事务消息**，避免同步发送堵死主链路。

##### Q1～Q3 对照速记表

| 诉求 | 更贴近的做法 |
| --- | --- |
| 落库与发通知最终一致、扛重启 | **Outbox / 事务消息 / CDC** |
| 仅当场多试几次、可接受崩溃丢待发 | **SDK 重试**，不必消息表 |
| 强绑定「本请求内全成或全败」、能接受依赖 MQ | **COMMIT 前同步发 MQ，失败则回滚本地库**（配幂等与超时策略） |

### 2.1、Seata 简介（是什么 / 特点 / 能力 / 依赖）

#### 1）Seata 是什么

Seata 是一款开源的分布式事务解决方案，目标是提供**高性能、简单易用**的分布式事务能力。它提供 **AT、TCC、SAGA、XA** 等事务模式，覆盖从“低侵入强一致”到“可补偿最终一致”的多类场景。

#### 2）特点（工作方式）

- **Seata Server 作为全局事务协调者（TC）**
- **通过注册中心/配置中心**与各微服务协作，协调分支事务的提交与回滚

#### 2.1）SAGA / 补偿事务（你这段笔记对应的内容）

> 你可以把 **SAGA** 理解为：**每个参与者都先把自己的本地事务提交**；如果中途某一步失败，就按“相反顺序”调用前面已成功步骤的**补偿操作（反向动作）**，把整体结果“撤回到看起来没发生过”。

**适用场景**

- **业务流程长、步骤多（长事务）**：例如下单链路里要经过很多服务/步骤，整体耗时可能很长。
- **参与者包含外部公司/遗留系统**：对方通常**无法提供 TCC 要求的 Try/Confirm/Cancel 三个接口**，这时更适合用 SAGA 的“正向 + 补偿”模型。
- **更偏 API 编排/服务编排**：核心是“调用链路”一致，而不是“同一个数据库的强隔离事务”。

**能做什么（能力点）**

- **跨库/跨服务的流程一致性**：每一步各自提交本地事务，整体靠“补偿”兜底。
- **失败重试 + 最大努力送达**：对外部调用/异步执行失败，常配合重试策略，做到“尽最大努力最终完成或最终补偿”。
- **补偿能力**：
  - **反向操作（补偿接口/反向 SQL）**
  - **更新快照/事务日志持久化**（用于回放、补偿、排障）
  - **自动生成部分补偿**（取决于实现：有的能自动生成更新快照或反向 SQL，有的需要你手写补偿逻辑）
- **持久化可扩展**：默认用关系型数据库存快照/事务日志；一些实现支持通过 **SPI** 扩展到其他存储。

**常见不支持/需要你自己兜底的点（不同实现差异很大）**

- **资源隔离/并发冲突隔离弱**：SAGA 通常不提供像数据库事务那样的强隔离，更多依赖业务侧“幂等 + 去重 + 状态机”。
- **宕机后的自动推进能力不一定完善**：有的实现能恢复“提交中/回滚中”状态并继续推进，有的需要人工介入/补偿任务兜底（要看具体框架与配置）。

**事务特性（用一句话说清楚，避免误解）**

- **不是“全局强一致 + 强隔离”**：SAGA 的一致性通常是**最终一致性**；每一步“本地事务”可以是强一致，但“全局”通常允许存在中间状态。
- **不满足传统事务隔离**：需要你在业务上处理并发与重复（例如同一订单重复执行、乱序到达等）。
- **不依赖底层数据源代理**：很多 SAGA 实现不需要像 AT 那样代理数据源，但代价是你要更认真地设计补偿与幂等。

**怎么用（落地要点）**

- **每个正向步骤都要有对应补偿步骤**：并且补偿要**幂等、可重入**（重复调用也不会越补越错）。
- **建议引入业务状态字段**：例如订单表增加 `tx_id / tx_state / tx_expire_time`（字段名按你们规范来），用于：
  - 识别同一事务
  - 控制“进行中/已完成/已取消/已过期”
  - 便于补偿任务扫描与人工排障
- **幂等与去重**：服务端要记录“已执行/已取消”的事务（你提到的 `ConcurrentHashMap`/Set 思路属于内存级去重），更稳的做法是**落库去重**（避免重启丢状态）。
- **高性能诉求时常用**：因为大多数时候不需要全局锁，但要权衡“补偿复杂度”和“中间状态可接受度”。

**框架提示（你提到的 Omega + Alpha / Seata）**

- **Omega + Alpha Server（ServiceComb Pack 等实现）**
  - **Alpha** 常作为全局事务协调者（记录事务日志、驱动补偿/推进）。
  - **Omega** 通常在业务侧负责拦截/上报/执行补偿（具体细节随实现不同）。
- **Seata SAGA**
  - Seata 的 SAGA 一般是**基于状态机引擎**来编排与推进步骤，表达能力强，但学习成本与配置复杂度也更高。

#### 3）功能/能力（你需要记住的点）

- **支持跨库分布式事务**
- **支持 RC（Read Committed）隔离级别**
- **通过 undo 快照**实现回滚（典型于 AT 模式）
- **服务宕机后可恢复**：对“提交中/回滚中”的事务可进行自动恢复与继续推进（依赖事务日志与协调器能力）

#### 4）依赖（落地前提）

- 需要额外部署 **`seata-server`**（协调分支事务、管理全局事务状态）

- **公共前提（不分发送/消费，缺一不可）**
  - **可观测与排障**：全局事务 `xid`（或业务 `tx_id`）要能贯穿日志/链路追踪；有失败告警、失败重试、人工介入手段
  - **网络与超时**：RPC/DB/消息的超时要统一设计（避免“某边已放弃，另一边还在执行”）；要能容忍抖动与重试
  - **幂等是底线**：无论是“提交/回滚/补偿/重试”都会带来重复调用，接口必须可幂等（可参考你另一篇《接口幂等性处理》）
  - **数据模型支持状态机**：关键业务表建议有 `tx_state / version(乐观锁) / updated_at` 等字段，避免并发覆盖与状态回退

- **发送侧依赖（发起方/生产者视角：先把“发出去”做可靠）**
  - **本地事务边界清晰**：只有本地事务提交成功后，才允许驱动下一步（发消息/推进分布式事务）
  - **可靠投递（典型最终一致性链路）**：若你用“本地事务 + 消息”方案，必须保证“写库成功 → 消息一定能投递”
    - 常见落地：**Outbox（事务内写业务表 + outbox 表）**，再由后台任务/CDC 可靠投递
    - 关键点：消息要有 **唯一业务键**（如 `orderId`）和 **去重键**（如 `msgId/txId`），为下游幂等提供依据
  - **重试策略**：发送失败要可重试；重试要有上限与退避；失败要进入告警/死信/人工补偿通道

- **消费侧依赖（参与方/下游视角：决定“最终能不能一致”）**
  - **消费幂等/去重**：必须能识别重复消息或重复调用（`msgId/txId + 业务键` 落库去重最稳），做到“多次执行结果一样”
  - **失败可重试且不越补越错**：处理失败要能安全重试；补偿/回滚同样要幂等、可重入（重复调用不会把状态越滚越乱）
  - **顺序与并发控制**：同一业务键（如同一订单）建议串行化处理或用版本号/状态机校验，避免乱序导致状态回退
  - **对账/补偿闭环**：要能识别“长时间未完成”的事务（超时扫描/对账），并自动或人工触发补偿/修复

### 2.2、2PC（两阶段提交）速记

2PC（Two-Phase Commit，两阶段提交协议）把一次分布式事务拆成 **2 个阶段**：**准备阶段（Prepare/Try）** 和 **提交阶段（Commit / Rollback）**，由事务管理器（TM）统一协调多个参与者（RM/各数据库节点）做到“要么都提交，要么都回滚”。

#### 1）准备阶段（Prepare / Try）

- **TM → 所有参与者**：发送 `Prepare` 请求
- **参与者本地执行事务**（但**不提交**）
- **写本地日志**：常见包括 **Undo/Redo**（用于回滚/崩溃恢复）
  - **Undo 日志**：记录修改前数据，用于回滚
  - **Redo 日志**：记录修改后数据，用于提交后落盘/恢复
- **参与者返回投票结果**：`Yes`（可以提交）或 `No`（必须回滚）

#### 2）提交阶段（Commit / Rollback）

- 如果 **所有参与者都返回 Yes**：
  - **TM → 所有参与者**：发送 `Commit`
  - **参与者提交本地事务**，并在最后释放事务期间占用的锁/资源
- 只要 **任意参与者返回 No**（或超时等异常）：
  - **TM → 所有参与者**：发送 `Rollback`
  - **参与者回滚本地事务**，并释放锁/资源

> 你可以把 2PC 记成一句话：**先“大家都准备好了吗？”（投票），再“统一提交或统一回滚”。**

### 2.3、3PC（三阶段提交）速记（在 2PC 上加一层确认 + 超时）

3PC（Three-Phase Commit，三阶段提交）可以理解为在 2PC 基础上做了两点改进：

- **增加 `CanCommit` 阶段**：先只问“能不能提交”，尽量把“会失败的事务”提前拦住，减少锁资源占用时间。
- **引入超时机制**：参与者在某些阶段**等不到协调者指令**时，会按照协议做“默认动作”，避免 2PC 里常见的“无限期阻塞”。

#### 1）三个阶段分别在做什么

- **CanCommit（询问/投票前置）**
  - 协调者询问各参与者：是否“从条件上”允许提交（通常不真正锁资源/不做重事务，只做校验）。
  - **若参与者超时没收到询问或无法确认**：一般按 **Abort（否决）** 处理，返回不允许提交。

- **PreCommit（预提交）**
  - 协调者在收到全体 `Yes` 后，发送 `PreCommit`。
  - 参与者执行本地事务到“可提交”状态（常见做法是**写日志/锁定资源/执行但不最终提交**），并回复 ACK。

- **DoCommit（最终提交）**
  - 协调者发送 `DoCommit`，参与者真正提交并释放资源（若协调者决定放弃，则发 `Abort`/回滚）。

#### 2）你提到的“超时自动 commit”到底发生在哪

你这句“短时间内没收到协调者的 commit 请求就自动 commit”需要补全前提：**不是任何时候都能自动 commit**，通常只发生在 **参与者已经进入 `PreCommit`（预提交成功）之后**：

- **场景 A：参与者处于 `PreCommit` 状态**，等待 `DoCommit` 时超时
  - 协议常见约定：参与者会**自行提交（Commit）**，以减少阻塞并提高可用性。
  - 直觉理解：既然大家都预提交了，继续卡住只会一直占资源，所以给出“超时默认提交”的路。

- **场景 B：参与者还在 `CanCommit` 阶段**（仅回答能不能提交），等待 `PreCommit` 时超时
  - 通常会走 **Abort（不提交）**。
  - 直觉理解：还没进入预提交，就缺少“大家都准备好了”的共同证据，贸然提交更危险。

> 记忆法：**只有“已经预提交（PreCommit）”才有资格在超时后“默认提交”。**

### 2.4、DTP 模型与 XA（2PC 的标准化接口）

在关系型数据库（如 Oracle、MySQL 等）里，常见的“基于数据库的 2PC”会遵循国际开放标准组织 Open Group 定义的分布式事务处理模型 **DTP（Distributed Transaction Processing Reference Model）**。

#### 1）DTP 三个角色（背会这 3 个就够了）

- **AP（Application Program）**：应用程序，发起/参与分布式事务的业务代码
- **TM（Transaction Manager）**：事务管理器，负责全局事务生命周期与协调（提交/回滚）
- **RM（Resource Manager）**：资源管理器，事务参与者，通常就是一个数据库实例（或能管理数据库事务的组件），负责分支事务

#### 2）它们怎么交互（理解“谁指挥谁”）

- **TM 向 AP 提供 API**：AP 通过 TM 发起提交/回滚
- **TM 通过 XA 接口通知 RM**：让数据库参与事务的开始、结束、Prepare、Commit、Rollback

#### 3）XA 是什么（一句话）

**XA** 是 DTP 模型里 **TM ↔ RM** 的通讯接口规范。你可以把它理解为“数据库对外提供的一套 **2PC 标准接口协议**”。基于 XA 来做 2PC，常被称为 **XA 方案**。

### 2.5、XA/2PC 的特点、能力与问题点（面试/方案常问）

#### 1）事务特性（优点）

- **强一致性 / 原子性**：两阶段提交保证“要么都提交，要么都回滚”
- **资源最后才释放**：锁等资源往往需要等到 Commit/Rollback 完成后才能释放（保证一致性）
- **宕机可恢复**：服务/进程重启后，对“提交中/回滚中”的事务可基于日志进行恢复推进（依赖具体实现）

#### 2）能力（你原文里提到的点）

- 支持数据分片后的跨库 **XA** 事务（前提是相关组件链路都支持 XA）
- 可通过 SPI 机制整合主流 XA 事务管理器（常见：Atomikos / Narayana / Bitronix）
- 同时支持 XA 与非 XA 的连接池，并提供 Spring Boot 等接入方式（按具体框架而定）

#### 3）缺点/风险（为什么高并发下要谨慎）

- **前提依赖**：本地数据库/驱动需要支持 **XA 协议**
- **性能较差**：锁资源通常要等两阶段结束才释放；参与者越多，锁持有时间越长
- **死锁与积压风险**：高并发时，“锁定资源”与“最终提交释放锁”之间的时间差被拉长，容易出现事务积压，甚至死锁或不确定性风险上升

#### 4）把两阶段再用一句话记牢

- **Prepare**：写日志（Undo/Redo 等）、锁定资源、执行但不提交
- **Commit**：真正提交、释放资源（或 Rollback 回滚）

### 2.6、工程框架与使用案例（Seata XA 与 JTA）

#### 1）常见框架/组件

- **JTA（Java 事务规范）+ Atomikos**：`spring-boot-starter-jta-atomikos`

### 2.7、Seata AT 模式原理（2PC + Undo Log + 全局锁 + 反向补偿）

你可以把 **AT 模式**理解成：在“本地数据库 ACID 事务”基础上，Seata 通过**数据源代理**、**Undo Log（回滚日志）**与**全局锁**，把跨服务/跨库的写操作组织成“可回滚、可协调”的分布式事务。

#### 1）前提条件（不满足就别选 AT）

- **关系型数据库**：必须支持本地 **ACID** 事务
- **Java + JDBC**：业务通过 JDBC 访问数据库（Seata 需要代理数据源）
- **核心限制**：AT 主要解决“跨库写入的一致性”，如果你的业务只是**纯 API 调用**（没有可被代理的数据库本地事务），AT 并不能直接保证一致（这种更常走 TCC/SAGA 或 MQ 最终一致）

#### 2）AT 的两阶段（和 2PC 很像，但二阶段提交是“异步化”的）

- **一阶段（Prepare）**：在**同一个本地事务**里同时完成两件事
  - 提交业务数据更新
  - 记录对应的 **Undo Log（回滚日志/快照）**
- **二阶段 Commit**：提交被“异步化”
  - 全局提交很快返回成功
  - 后台异步批量清理 Undo Log
- **二阶段 Rollback**：通过 Undo Log 自动生成**反向补偿 SQL**，把数据回滚到一阶段之前

#### 3）事务隔离（写隔离）是怎么做到的：全局锁

- **关键规则**：一阶段本地事务**提交前必须先拿到全局锁**
  - 拿不到全局锁 → **不能提交本地事务**（避免脏写）
  - 全局锁获取会有重试/超时上限，超出就放弃并回滚本地事务，释放本地锁

#### 4）示例：tx1/tx2 并发更新同一行，如何避免脏写

假设两个全局事务 `tx1`、`tx2` 都要更新 `a.m`，初始 `m = 1000`：

- **tx1 先开始**
  - 开启本地事务，拿到本地锁
  - 更新 `m = 1000 - 100 = 900`
  - 本地提交前拿到该行的**全局锁**
  - 本地提交释放本地锁（全局锁仍由 tx1 持有，直到全局二阶段结束）

- **tx2 后开始**
  - 开启本地事务，拿到本地锁
  - 更新 `m = 900 - 100 = 800`
  - 本地提交前尝试拿全局锁，但该行全局锁被 tx1 持有 → tx2 需要重试等待

- **tx1 进入二阶段**
  - 如果 tx1 **全局提交**：释放全局锁 → tx2 拿到全局锁并提交本地事务
  - 如果 tx1 **全局回滚**：tx1 需要重新获取该数据的本地锁，执行反向补偿回滚
    - 若此时 tx2 一直占着本地锁且在等全局锁，tx1 的回滚可能失败并持续重试
    - 直到 tx2 等锁超时放弃全局锁并回滚本地事务释放本地锁 → tx1 回滚最终成功

> 结论：因为全局锁在 tx1 结束前一直被 tx1 持有，所以不会发生“脏写”。代价是并发冲突时会有等待、超时与重试成本。

### 3、Seata 的典型适用场景

### 3.1、什么叫“强一致性”（以及怎么达到）

很多同学一上来会把“强一致性”理解成“用了事务就一定强一致”。更准确的说法是：**强一致性 = 对外表现像“只有一个副本/一个数据库”**。

#### 1）一句话定义（最好背下来）

- **强一致性（Strong Consistency / 线性一致性）**：当一次写入对外返回成功后，后续任何读取（无论读到哪个节点/副本）都必须立刻读到这次写入后的最新结果，不会出现“有人读到新值、有人还读到旧值”的窗口期。

> 直觉理解：写成功就是“全世界都承认了”，之后再读就不允许“看见过去”。

#### 2）在分布式里为什么难（你要接受它有代价）

- **网络不可靠**：消息可能丢、乱序、延迟，节点可能宕机、分区
- **多副本复制**：写入从一个节点扩散到多个节点，传播需要时间
- **所以结论**：要强一致，通常就要“等更多人确认”，带来**更高延迟、更低吞吐、可用性下降**（典型是 CAP 取舍）

#### 3）达到强一致性，工程上常见的两条路

- **路 A：共识复制（Raft/Paxos 思路）**
  - **写**：Leader 接收写入 → 复制到多数派（Quorum）→ 多数派确认后才对外返回成功
  - **读**：通常读 Leader（或读前先做一致性校验/读到最新提交点）
  - **特点**：对“一个数据的多副本一致”非常强，但跨库/跨服务还需要额外机制

- **路 B：分布式事务（2PC/XA 或类共识事务）**
  - **目标**：跨多个资源（多个库/多个服务）做到“要么都提交，要么都回滚”
  - **特点**：能提供更接近传统 ACID 的原子提交语义，但成本高、链路复杂、容易受慢 SQL/外部调用拖累

#### 4）把“强一致性”与 ACID 对上号（避免背概念背散）

你可以把 **ACID** 看成“单库事务的四个底层保证”，而分布式要做到同级别体验，通常必须引入“共识/分布式事务/额外隔离手段”：

- **A 原子性（Atomicity）**：跨多个操作要么全成功要么全失败  
  - 分布式常见抓手：**2PC/XA、Seata（AT/TCC/SAGA/XA）、基于日志/状态机的事务推进**
- **C 一致性（Consistency）**：事务前后满足约束（余额不为负、库存不超卖等）  
  - 分布式常见抓手：**全局提交顺序（共识/主从）、幂等 + 校验、约束前置**
- **I 隔离性（Isolation）**：并发事务互不干扰（不读到“中间态”）  
  - 分布式常见抓手：**全局锁/资源冻结（TCC）、MVCC/时间戳、把关键写收敛到单主串行化**
- **D 持久性（Durability）**：提交后不丢数据  
  - 分布式常见抓手：**WAL/redo 日志 + 多副本复制 + 多数派落盘确认**

> 结论：**强一致性不是“零成本的默认选项”**。你要的是“更强的正确性”，就要为它的“协调、等待、锁冲突、慢链路放大”付出成本。

Seata 适合“强一致性优先”的核心业务流程，尤其是操作必须实时生效且不允许中间状态的场景。

#### 1）订单创建全流程（下单 → 扣库存 → 支付）

- **要求**：订单、库存、支付状态必须严格一致（如“下单成功必扣库存，扣库存失败则订单必失败”）。
- **原因**：若采用 MQ 异步方案，可能出现“订单已创建但库存扣减延迟/失败”的中间状态，导致超卖或用户投诉。
- **实现**：用 Seata **AT 模式**，通过全局事务协调订单服务、库存服务、支付服务的本地事务。

#### 2）资金相关操作（转账、退款）

- **要求**：资金变动必须实时准确，不允许“一方扣钱另一方未到账”。
- **示例**：订单退款时，需同时更新订单状态、退还用户账户余额、通知支付渠道，三者必须原子性完成。

#### 3）库存与订单强绑定场景

- **示例**：限量商品秒杀，必须保证“下单成功 = 库存扣减成功”，避免超卖。
- **原因**：Seata 的全局锁可防止并发下的库存不一致。

### 3.2、订单业务中 Seata 的适配场景（为什么 Spring 事务传播不够）

当订单业务涉及**跨服务调用**（如订单服务、库存服务、支付服务分属不同微服务，且通常对应不同数据库）时，Spring 的本地事务传播机制只能保证**单个服务/单个数据源**内一致，无法保证跨服务的原子性与一致性，此时就需要 Seata 这类分布式事务框架做全局协调。

#### 1）哪些订单场景“必须用”Seata

- **典型链路**：订单服务（创建订单）→ 库存服务（扣减库存）→ 支付服务（处理支付）
- **原因**：这些操作分属不同服务与数据库，本地事务传播机制（如 `REQUIRED`）无法跨服务传播与统一提交/回滚
- **目标**：通过 Seata 协调全局事务，确保“要么全部成功，要么全部回滚”

#### 2）Seata 如何与 Spring `@Transactional` 结合使用（要点）

- **全局事务入口**：使用 Seata 的 `@GlobalTransactional` 标记全局事务发起者（通常在订单服务的应用层方法上）
- **参与者本地事务**：下游服务仍然使用 `@Transactional` 管理本地数据库操作，但其本地事务会在 Seata 的协调下参与全局事务
- **优先级理解**：在分布式场景下，**全局事务语义优先于本地传播语义**；你不能指望用 `REQUIRES_NEW` 等本地传播行为来“跨服务制造一个独立小事务”

#### 3）事务传播机制的局限性（常见误区）

- **跨服务无法共享本地事务上下文**：`REQUIRES_NEW`、`NESTED` 等传播行为只对**同一服务进程内**的事务边界有效
- **独立微服务天然独立提交**：比如“日志记录服务”如果是独立微服务，它的事务是否回滚取决于它自身逻辑；若你希望“跟主流程强一致”，就要把它纳入 Seata 全局事务（但通常不建议把日志纳入强一致）

#### 4）Seata 模式选择建议（订单域）

- **AT 模式**：适用于大多数“CRUD 型”强一致链路，业务改造相对少（但要评估全局锁与长事务）
- **TCC 模式**：适用于复杂库存/资金等需要“预留/确认/取消”的场景，需要你实现 Try/Confirm/Cancel
- **性能考量**：分布式事务会增加网络开销与锁竞争，应避免过度使用；非核心操作建议走 MQ 最终一致性（你在本文后半部分的 MQ 章节就是这套）

#### 5）示例：Seata 在订单业务中的应用（简化版）

```java
// 订单服务（全局事务发起者）
@Service
public class OrderService {
    @Autowired private OrderMapper orderMapper;
    @Autowired private InventoryFeignClient inventoryClient;
    @Autowired private PaymentFeignClient paymentClient;

    // 全局事务注解：标记为分布式事务的发起者
    @GlobalTransactional(rollbackFor = Exception.class)
    public void createOrder(Order order) {
        // 1) 本地操作：保存订单（由 Seata 代理的数据源管理本地事务）
        orderMapper.insert(order);

        // 2) 跨服务调用：扣减库存（参与全局事务）
        inventoryClient.deductStock(order.getProductId(), order.getQuantity());

        // 3) 跨服务调用：处理支付（参与全局事务）
        paymentClient.processPayment(order.getId(), order.getAmount());
    }
}

// 库存服务（参与全局事务）
@Service
public class InventoryService {
    @Autowired private InventoryMapper inventoryMapper;

    // 本地事务：由 Seata 代理，参与全局事务
    @Transactional(rollbackFor = Exception.class)
    public void deductStock(Long productId, int quantity) {
        if (!checkStockEnough(productId, quantity)) {
            throw new InsufficientStockException(); // 触发全局回滚
        }
        inventoryMapper.updateStock(productId, quantity);
    }
}
```

### 4、BASE 理论（为“最终一致性”打基础）

当你选择 “MQ 最终一致性/异步补偿” 这类方案时，背后的思想通常就是 **BASE**：

- **B（Basically Available，基本可用）**
- **S（Soft State，软状态/中间状态）**
- **E（Eventual Consistency，最终一致性）**

一句话：**在分布式系统里，为了抗故障、抗高并发，允许短时间“不那么一致”，换取系统整体“不断服务”，最终再把数据对齐。**

#### 1）B：基本可用（Basically Available）
核心意思：**核心业务必须可用**；在压力大或故障时，允许“边缘能力”暂时不可用或体验变差。

常见做法（你可以直接记成“降级三件套”）：
- **边缘业务降级**：如推荐、评论、非关键统计先关掉/延迟显示
- **核心业务放宽响应**：如变慢一点、排队、返回“处理中”
- **限流/熔断/降载**：保护核心链路不被拖垮

例子：双 11 下单时，优惠券列表/猜你喜欢可以先不展示，但“创建订单”必须能用。

#### 2）S：软状态（Soft State）
核心意思：系统允许存在**中间状态**，数据在一段时间内可能“还没同步完/还没对齐”。

你写的例子非常典型，我帮你把表达补完整：
- **读写分离的“半同步状态”**
  - 写入先落到主库
  - 从库复制有延迟
  - 这段时间里：**主库是新数据，从库可能还是旧数据**（中间状态）

- **支付过程的中间状态（支付中）**
  - 用户点“确认支付”后，订单先进入 **支付中**
  - 支付系统去请求银行/渠道
  - 银行处理成功后回调/通知支付系统
  - 支付系统更新自己状态，并通知订单系统
  - 订单系统收到“支付成功”事件后，订单变为 **已支付**

注意：既然允许中间状态，就必须配套：**超时回滚/关闭订单、重复通知处理（幂等）、补偿重试**。

#### 3）E：最终一致性（Eventual Consistency）
核心意思：在允许中间状态的前提下，经过一段时间（秒级/分钟级/小时级）通过同步、重试、补偿等机制，**数据最终会变一致**。

实现最终一致性常见“工程四件套”：
- **可靠事件（本地消息表/Outbox）**：业务落库成功后，把“要通知别人”的事件可靠记录下来
- **异步通知（MQ）**：把事件发给下游系统
- **失败自动重试 + 死信兜底（DLQ/DLT）**
- **幂等 + 去重**：同一事件重复到达也不会“多扣/多加/多改”

### 5、MQ 最终一致性的典型适用场景

MQ 最终一致性适合“性能优先、可接受短暂不一致”的非核心流程，或跨系统集成场景。

#### 1）订单后续非核心操作（通知、日志、统计）

- **示例**：订单创建后发送短信通知、更新数据报表、记录审计日志等。
- **特点**：这些操作失败不影响订单核心流程，可通过 MQ 异步重试确保最终完成。
- **实现**：订单服务本地事务提交后发送消息到 MQ；消费端异步处理通知/统计逻辑，失败则重试。

#### 2）跨系统异步集成（如 ERP、物流系统）

- **场景**：订单创建后需同步数据到外部物流系统生成物流单。
- **特点**：外部系统响应慢、不稳定；同步调用会阻塞订单流程。
- **实现**：订单提交后发送消息到 MQ；物流系统消费消息异步处理；通过定时任务补偿失败的同步。

#### 3）高并发写入场景（如秒杀、大促）

- **问题**：秒杀订单创建若用 Seata，可能因全局锁导致性能瓶颈。
- **替代方案**：采用“预扣库存 + MQ 异步确认”：
  - 先扣减库存生成待支付订单
  - 通过 MQ 异步通知支付系统
  - 超时未支付则自动释放库存

### 6、混合使用策略（订单业务最佳实践）

实际业务中常结合两者优势：**核心流程用 Seata 保证强一致**，**非核心流程用 MQ 保证最终一致**。

#### 订单创建流程

1. **核心步骤（Seata AT 模式）**  
   订单服务创建订单 → 库存服务扣减库存 → 支付服务创建支付单  
   **（三者强一致，任意失败则全局回滚）**

2. **非核心步骤（MQ 最终一致性）**  
   订单提交成功后 → 发送消息到 MQ →  
   - 消费端 1：发送短信通知（失败重试）  
   - 消费端 2：更新用户积分（失败重试）  
   - 消费端 3：同步数据到数据仓库（失败补偿）

### 7、决策 checklist

- **是否允许中间状态？**
  - 允许 → MQ 最终一致性
  - 不允许 → Seata 强一致性

- **性能要求是否极高？**
  - 是 → MQ（异步无锁）
  - 否 → 可考虑 Seata

- **是否跨多个外部系统？**
  - 是 → MQ（避免外部系统不稳定影响核心流程）
  - 否（仅内部微服务）→ 可考虑 Seata

- **业务复杂度？**
  - 简单流程（CRUD 为主）→ Seata AT 模式（侵入性低）
  - 复杂业务（需自定义补偿逻辑）→ 若允许最终一致，选 MQ；否则选 Seata TCC

### 8、自动重试：两者都能配，但方式不同

#### 1）MQ 最终一致性：消费端重试 + 失败兜底（最常用）

- **消费端自动重试（核心）**
  - **触发方式**：消费处理失败（抛异常/返回失败）→ MQ 进行重新投递（或由框架触发重试）。
  - **建议策略**：指数退避（间隔逐渐变大）+ 最大重试次数（例如 5～16 次）。
  - **必须配套**：**幂等**（同一消息重复执行不会产生重复扣减/重复加积分）。

- **死信队列 DLQ（兜底）**
  - **机制**：超过最大重试次数仍失败 → 自动进入死信队列/死信主题。
  - **处理**：人工处理 + 定时补偿任务（把死信修复后重新投递或走补偿接口）。

  - **作用（你要会讲的点）**
    - **停止无意义的无限重试**：避免同一条失败消息反复占用消费者线程/DB 连接，拖慢正常消息
    - **隔离“毒消息”**：例如参数格式错误、业务校验永远不通过的消息，不让它堵住主队列
    - **排障与审计入口**：把失败消息集中到一个地方，便于看失败原因 TopN、定位问题
    - **补偿/重放入口**：修复代码或修复数据后，可从 DLQ 手动/自动重放，或走补偿接口把状态补齐
    - **防雪崩**：下游依赖故障时，DLQ + 告警让系统“可控降级”，而不是把主链路打爆

- **生产端“投递可靠性”重试（避免消息丢）**
  - **推荐做法**：本地消息表/Outbox（业务事务提交成功 → 记录待发送消息 → 异步/定时投递 → 成功后标记已发送）。
  - **目标**：保证“业务写库成功”与“消息最终发出”一致，避免只下单不发消息/或发消息但订单没落库。

- **关键配置清单（不依赖具体 MQ 名称）**
  - **重试次数**：MaxRetry（到达后进 DLQ）
  - **重试间隔**：Backoff（固定/阶梯/指数）
  - **超时**：消费处理超时要有上限（避免长期占用）
  - **去重/幂等键**：messageId / businessKey（例如 `orderId+eventType`）
  - **可观测**：重试次数、堆积量、DLQ 数、失败原因 TopN、报警

> 说明：不同 MQ（RabbitMQ/RocketMQ/Kafka）实现重试的“按钮”不一样，但上面这套“重试 + 死信 + 幂等 + 可观测”是通用落地模型。

#### 2）Seata：能重试，但重点是“事务恢复/回滚重试/补偿幂等”

- **AT 模式（偏框架自动恢复）**
  - **常见需要重试的点**：全局锁冲突、分支注册/上报失败、回滚阶段的临时失败。
  - **你需要保证**：
    - **本地事务可回滚**（undo 日志等机制可用）
    - **避免长事务**（长时间持锁会放大冲突与重试成本）
    - **数据库与隔离级别符合 AT 要求**

- **TCC 模式（Confirm/Cancel 的重试与幂等是硬要求）**
  - **典型情况**：Confirm/Cancel 因网络/超时失败时，协调器会再次调用。
  - **你必须做到**：
    - **Confirm 幂等**、**Cancel 幂等**
    - **空回滚**（Try 没成功也可能收到 Cancel，需要能安全处理）
    - **防悬挂**（Cancel 先于 Try 到达等乱序，需要能识别并拒绝/延迟）

- **SAGA 模式（补偿步骤的重试）**
  - **机制**：每个正向步骤都要有对应的补偿步骤；失败时按已执行顺序触发补偿。
  - **你必须做到**：补偿接口幂等 + 可重入（重复补偿不产生副作用）。

- **Seata 的“兜底”思路**
  - **失败可追踪**：把全局事务 XID、分支状态、失败原因打到日志/监控
  - **可补偿/可人工介入**：对核心资金/库存类，准备人工处理入口（避免无限重试）

#### 3）一页版结论（你可以直接记住这句话）

- **MQ**：自动重试很好配，但一定要 **幂等 + DLQ + 补偿任务**，否则重复消费会把数据“越重试越错”。
- **Seata**：也会重试，但更像“事务协调/恢复”；AT 关注 **锁冲突与长事务**，TCC/SAGA 关注 **幂等、空回滚、防悬挂、可补偿**。

### 9、MQ 自动重试落地模板（压缩版：通用模型 + 各 MQ 差异点）

> 你只要记住一套通用模型：**重试（Backoff）→ 上限（MaxRetry）→ 死信（DLQ/DLT）→ 人工/补偿（Replayer）→ 对账兜底（Reconcile）**。

#### 9.1 通用模型（任何 MQ 都这么落地）

- **消息一定会重复**：默认至少一次投递 → 消费端必须 **幂等去重**（见第 10 章）
- **失败处理四件套**
  - **重试**：失败自动再投递（建议退避：1m/5m/15m/30m/1h 或指数退避）
  - **上限**：达到上限就停止“无限重试”（避免把系统拖死）
  - **死信**：进入 DLQ/DLT 后 **必须告警**，并且支持“修复后重放”
  - **对账兜底**：关键结果（发货/积分/状态推进）用定时任务扫表补齐
- **推荐默认值（可按业务调）**
  - **MaxRetry**：8（常见范围 5～16）
  - **Backoff**：阶梯或指数退避
  - **监控必看**：重试次数分布、堆积量、DLQ 数、失败原因 TopN

#### 9.2 RabbitMQ（差异点：用 TTL + DLX 做“延迟重试”）

- **队列建议**：`biz.queue` + `retry.1m/5m/15m/30m/1h` + `dlq.queue`
- **流程**：失败 → 投 `retry.*` → TTL 到期经 DLX 回 `biz.queue` → 超限进 `dlq.queue`

#### 9.3 RocketMQ（差异点：自带消费重试 + DLQ）

- **关键开关**：`maxReconsumeTimes`（超限进 DLQ）
- **DLQ 习惯命名**：`%DLQ%{consumerGroup}`（版本/客户端略有差异）
- **建议**：DLQ 堆积立刻告警 + 提供重放入口

#### 9.4 Kafka（差异点：推荐“重试主题 + 死信主题”，别原地无限重试）

- **主题建议**：`topic.biz` + `topic.retry.1m/5m/30m` + `topic.dlt`
- **流程**：失败写入 `topic.retry.*`（带 `retryCount/nextTime`）→ 到点回 `topic.biz` → 超限进 `topic.dlt`

### 10、幂等与去重（自动重试的“安全带”）

> 只要你启用了重试，就要默认“同一条消息可能会被处理多次”。幂等做不好，重试会把业务越修越错。

- **推荐幂等键（businessKey）**
  - 常用：`orderId + eventType`（例如 `12345:ORDER_CREATED`）
  - 或者：`messageId`（前提是生产端能保证稳定且全链路携带）

- **推荐落地做法（最稳）**
  - 建一张“消费去重表/处理记录表”，`businessKey` 做唯一索引
  - 消费开始前先插入/更新状态（成功/失败/处理中）
  - 同一 `businessKey` 重复到达时直接跳过或走“查询结果返回”

- **推荐状态机**
  - 订单/库存/积分这类业务，最好用“状态机 + 幂等更新”（例如只允许从 A → B，不允许重复 B → B 产生副作用）

### 11、Seata 重试“如何才算配好了”

#### 1）AT 模式（重点：减少锁冲突与长事务）

- **实操要点**
  - **缩短全局事务**：不要把慢 SQL、外部 HTTP 调用、长耗时逻辑放进全局事务
  - **热点资源拆分**：例如库存热点 SKU，评估 AT 全局锁冲突风险
  - **失败可追踪**：日志里必须能定位到 XID（全局事务 ID）与分支状态

#### 2）TCC 模式（重点：Confirm/Cancel 可被重复调用）

> 你可以把 TCC 记成“**业务自己写的 2PC**”：一阶段先**预留资源**（Try），二阶段要么**确认**（Confirm），要么**取消**（Cancel）。

##### 2.1 常见框架

- **Seata TCC**：适合“跨服务 + 强一致 + 需要预留资源”的场景（库存/资金/优惠券等）。
- **tcc-transaction**：社区常见 TCC 框架（项目名里经常写成 `tcc-transaction`，不是 `tcc-tracsaction`）。

##### 2.2 事务特性（和 AT/MQ 的差异点）

- **不依赖数据库代理/UndoLog**：TCC 不要求你代理数据源；它靠你写的 Try/Confirm/Cancel 来保证一致性。
- **不保证传统 ACID 的“隔离性”**：TCC 的“隔离”更多靠**业务侧的资源预留/冻结/锁定**来实现（例如冻结库存、冻结余额）。
- **可以用于纯 API 调用场景**：即使没有数据库写入，只要能设计“预留/确认/取消”的接口，也能做强一致。
- **强一致的前提**：Confirm/Cancel 要能在各种重试、超时、乱序下依然正确（这也是 TCC 最难的地方）。

##### 2.3 落地用法（推荐的“业务表 + 事件表/事务表”）

你的业务通常需要把“事务状态”显式落表（避免只靠内存/日志导致不可恢复）：

- **业务表字段（示例）**
  - `tx_id`：全局事务/业务事务标识（或使用 Seata 的 XID + BranchId 做映射）
  - `tx_state`：Try/Confirmed/Canceled（或 FREEZE/COMMIT/ROLLBACK 等）
  - `ts_expire_time`：冻结/预留资源的过期时间（超时后走 Cancel/补偿）
- **事件表/请求表（Outbox/RequestLog）**
  - 记录每次 Try/Confirm/Cancel 的请求参数、幂等键、处理结果、错误原因
  - 作用：**可重试、可对账、可人工介入**

##### 2.4 2PC 的“白盒写法”（Try-Confirm/Cancel）

- **一阶段 Try（Prepare）**
  - 只做“尝试执行 + 一致性检查 + 资源预留/冻结”
  - 不做最终扣减/最终生效（避免后续 Cancel 难以回滚）
- **二阶段 Confirm（Commit）**
  - 只使用 Try 阶段已预留的资源，直接让结果最终生效
  - 业界常见假设：**Try 成功 ⇒ Confirm 应尽量设计成必成功**；如果真的失败，要有自动重试与人工兜底
- **二阶段 Cancel（Rollback）**
  - 释放 Try 阶段预留/冻结的资源（库存解冻、余额解冻、原路退款等）
  - 同样要支持重试与人工兜底（尤其是外部支付/退款接口）

##### 2.5 你必须做到（硬要求）

- **Try/Confirm/Cancel 全幂等**：同一个 `tx_id` 重复调用不会产生重复扣减/重复解冻
- **空回滚（Empty Rollback）**：Try 未成功/未执行时，Cancel 也能安全返回成功
- **防悬挂（Anti-suspension）**：Cancel 先到、Try 后到时，Try 不能再把资源冻结起来（避免“冻结悬挂”）
- **超时与补偿**：冻结必须有 `expire_time`，超时后进入 Cancel/补偿流程，避免资源长期被占用

##### 2.6 Seata TCC 接口案例（你需要定义三段逻辑）

```java
public interface TccActionOne {

    @TwoPhaseBusinessAction(
        name = "DubboTccActionOne",
        commitMethod = "commit",
        rollbackMethod = "rollback"
    )
    boolean prepare(
        BusinessActionContext actionContext,
        @BusinessActionContextParameter(paramName = "a") String a
    );

    boolean commit(BusinessActionContext actionContext);

    boolean rollback(BusinessActionContext actionContext);
}
```

如果 TCC 参与者是**本地 Bean（非 RPC）**，接口上还需要 `@LocalTCC`（以 Seata 的要求为准）：

```java
@LocalTCC
public interface NormalTccAction {
    @TwoPhaseBusinessAction(
        name = "tccActionForTest",
        commitMethod = "commit",
        rollbackMethod = "rollback",
        commitArgsClasses = {BusinessActionContext.class, TccParam.class},
        rollbackArgsClasses = {BusinessActionContext.class, TccParam.class}
    )
    String prepare(
        @BusinessActionContextParameter("a") int a,
        @BusinessActionContextParameter(paramName = "b", index = 0) java.util.List b,
        @BusinessActionContextParameter(isParamInProperty = true) TccParam tccParam
    );

    boolean commit(
        BusinessActionContext actionContext,
        @BusinessActionContextParameter("tccParam") TccParam param
    );

    boolean rollback(
        BusinessActionContext actionContext,
        @BusinessActionContextParameter("tccParam") TccParam param
    );
}
```

全局事务入口一般还是用 `@GlobalTransactional` 把多个 TCC 分支串起来：

```java
@GlobalTransactional
public String doTransactionCommit() {
    tccActionOne.prepare(null, "one");
    tccActionTwo.prepare(null, "two");
    return "ok";
}
```

##### 2.7 TCC 10 行必背版（面试/考试）

1. **TCC = 业务自己写的 2PC**（Try/Confirm/Cancel）。
2. **Try**：只做检查 + 预留/冻结资源，不做最终生效。
3. **Confirm**：只用 Try 预留资源让结果最终生效，尽量设计成“必成功”。
4. **Cancel**：释放 Try 预留资源，失败要可重试 + 可人工兜底。
5. **必须幂等**：Try/Confirm/Cancel 都可能被重复调用。
6. **空回滚**：Try 没执行/没成功时，Cancel 也要能安全成功。
7. **防悬挂**：Cancel 先到、Try 后到时，Try 不能再冻结资源。
8. **要落表**：事务/冻结状态必须可恢复（`tx_id/tx_state/expire_time` + 请求/事件表）。
9. **适合场景**：库存/资金/券等“可预留资源”的强一致链路，或纯 API 场景。
10. **最大坑**：长时间冻结导致吞吐下降；必须配超时与补偿流程。

##### 2.8 一张表背会选型：AT vs TCC vs SAGA vs MQ 最终一致

| 方案 | 一致性 | 业务侵入 | 依赖/前提 | 优点 | 典型坑/代价 | 典型场景 |
| --- | --- | --- | --- | --- | --- | --- |
| **Seata AT** | 强一致（对“数据库写”友好） | 低～中 | 需要代理数据源、关系型数据库、本地 ACID、UndoLog/全局锁 | 改造小，CRUD 链路好落地 | 全局锁冲突、长事务、外部 HTTP/慢 SQL 会拖垮 | 订单-库存-账户等跨库写 |
| **Seata TCC** | 强一致（可覆盖纯 API） | 高 | 需要设计 Try/Confirm/Cancel；资源可“冻结/预留” | 不依赖数据源代理；可控性强 | 幂等/空回滚/防悬挂/超时补偿复杂，冻结降低并发 | 库存冻结、余额冻结、原路退款 |
| **Seata SAGA** | 最终一致（可补偿） | 中～高 | 每个正向动作要有补偿动作；允许中间状态 | 不需要锁住资源太久；适合长流程 | 补偿不一定能完全回滚（外部系统/不可逆） | 旅行预订、审批流、跨系统长链路 |
| **MQ 最终一致** | 最终一致 | 高 | Outbox/事件表、重试、幂等、DLQ、对账/补偿 | 吞吐高，解耦好，抗峰值 | 中间状态处理复杂；一致性要靠工程化兜住 | 通知/积分/报表/日志等非核心链路 |

##### 2.9 三句话选型口诀（背下来就够用）

1. **能用 AT 就用 AT**：都是数据库写、链路不长、能接受全局锁带来的冲突与等待。
2. **需要“冻结/预留资源”就用 TCC**：库存/资金/券，或纯 API 也要强一致（但你要写幂等/空回滚/防悬挂）。
3. **链路长/外部系统多/允许中间态就用 SAGA 或 MQ**：能补偿的长流程用 SAGA；非核心异步通知用 MQ 最终一致。

#### 3）SAGA 模式（重点：补偿可重试、可重入）

- **你必须做到**
  - 每个正向动作都有对应补偿动作
  - 补偿接口 **幂等 + 可重入**
  - 补偿失败有兜底：人工/任务补偿通道（避免无限重试）

### 12、MQ 最终一致性落地：最大努力通知（Best-Effort Notification）

> 这是一套“最终一致性”的标准工程化做法：关键事实先**可靠记录为事件**，再用**异步通知 + 自动重试 + 死信兜底 + 人工介入**，尽最大努力让各系统最终一致。

#### 1）这是什么（一句话）

- **最大努力通知**：把关键事实先可靠落库（事件/消息），然后不断尝试通知下游系统；允许短暂不一致，但要求最终对齐。
- **典型场景**：支付成功后通知订单、库存、发货、积分、短信等下游。

#### 2）落地依赖（缺一不可）

- **可靠消息**：至少一次投递；允许重复但不能无声丢失
- **中间状态**：允许“处理中/待通知/待确认”等状态存在一段时间
- **重试**：可配置次数、间隔（退避）、上限
- **死信处理（DLQ）**：超过上限进入死信，必须可追踪可处置
  - **可选**：回滚/撤销/修正消息（多数业务更常用“补偿/修正”，而不是硬回滚）
- **人工介入**：告警 + 后台页面 + 审计，支持查询、重放、标记完成

#### 3）核心特点

- **异步**
- **最终一致性**

#### 4）发送端（生产者）两种主流程

##### 4.1 先发消息再处理本地事务（有坑）

- **风险**：发消息成功但本地事务失败
  - 普通 MQ 场景下消息通常**不能回滚/撤回**，下游可能已执行，造成不一致

- **解决方向（特定场景）**：RocketMQ 事务消息
  - 通过 **半消息 + 本地事务 + 回查确认** 绑定“消息投递”与“本地事务提交”
  - 目标效果：
    - 本地数据库事务成功且消息最终成功提交到 Broker ⇒ 发送端闭环完成
    - 下游全部最终完成 ⇒ 达到最终一致性

##### 4.2 先处理事务后发消息（最通用）：Outbox 事件表 + 定时补偿

- **核心思路**：在**同一个本地数据库事务**里同时写：
  - 业务数据
  - 一条 **Outbox（消息事件表）** 记录（相当于“要发的消息”）
- **事务提交后**：后台任务/CDC/异步发送器把 Outbox 投递到 MQ

- **常见问题：纯 MQ 最终一致性，是否需要“生产事件表 + 消费事件表”？**
  - **最小可用（至少一张表）**：强烈建议有 **生产事件表（Outbox）**
    - 目的：保证“业务落库成功”与“要发的事件被可靠记录”绑定在同一个本地事务里，避免丢消息/半成功
  - **更稳妥（推荐两张表）**：在 Outbox 之外，再加 **消费事件表（Inbox/消费记录表）**
    - 目的：抵抗 MQ 的重复投递/重试/抖动，消费端用 `event_id` 或 `biz_key+event_type` 落库去重，做到“同一事件只生效一次”
  - **什么时候可以不单独建消费事件表**：若你的“业务写入”天然幂等（例如状态机 + 乐观锁 + 唯一约束），可以只用业务表承接幂等；但仍要确保重复消息不会造成重复扣款/重复加库存等副作用

- **分支处理**
  - **事务失败**：回滚，Outbox 不落库，**不发消息**
  - **事务成功但发送 MQ 失败**：进入**重试/补偿**（Outbox 的价值）
    - **需要重试**：网络问题、Broker 短暂不可用、确认超时/确认失败
    - **不需要重试**：参数错误/业务校验错误（应直接失败并告警/人工处理）

- **定时扫描补偿（常用扫描条件）**
  - 条件：`status in (NEW,FAILED)` 且 `next_retry_at <= now`
  - 成功：标记 `SENT`
  - 失败：`retry_count++`，计算下次时间（退避）；超过上限 ⇒ `DEAD/NEED_MANUAL`

#### 5）消息过期与堆积治理（线上必做）

- **有 TTL 的消息**：到期自动删除（以具体 MQ 行为为准）
- **无 TTL 导致堆积**（消费者离线/不消费）
  - **示例规则**：任务 1 天内没有连接，且消息积压已达 5 天 ⇒ 判定“过期堆积”
  - **处理策略（按重要性从低到高）**
    - 直接删除（不落库）：低价值、可丢弃
    - 落库后删除：保留审计与后续人工补偿能力
    - 转存 DLQ：可先落库，再从原队列删除

#### 6）消费端失败处理：重试、死信、人工补偿

- **常见失败**
  - ACK 失败
  - 业务处理抛异常

#### 6.1 消费压力如何反馈给生产端（让生产端自动降速）

> **一句话**：消费者扛不住时，不要只靠“堆积在 MQ 里”，还要让生产端**感知堆积/延迟**并自动降速，形成端到端闭环（否则很容易把 Broker/DB/下游依赖打爆）。

- **最推荐（通用）：生产端根据堆积/延迟动态限流**
  - **信号来源**：队列堆积、端到端延迟、重试队列堆积、DLQ 增速（本质都是“消费能力不足”的外显）
  - **生产端动作**：发送前加一个**动态令牌桶/漏桶**，按阈值分档降速并带滞回防抖
    - 示例：堆积 > 10w → 50%；> 30w → 20%；< 6w 且持续 5 分钟 → 才逐级恢复
  - **降级策略**：只发关键消息、合并/采样、延迟发送、直接拒绝并让上游重试（按业务价值选择）

- **各 MQ 的常用“压力信号”长什么样**
  - **RabbitMQ**：`messages_ready`（待消费数）、`messages_unacknowledged`（未 ACK，在途压力）
  - **Kafka**：Consumer Group **lag**、端到端延迟（生产时间戳 vs 消费完成时间戳）
  - **RocketMQ**：堆积量（topic/group）、重试队列堆积、DLQ 增速

- **更强反馈（可选）：消费者下发“BUSY/配额/credits”**
  - 消费者（或调度器）定期发布控制信号（控制 topic/队列 或 Redis/配置中心）：
    - `state=BUSY` / `quota=5000 msg/s` / `credits=5000/s`
  - 生产端按配额调整发送速率；建议**短 TTL + 周期刷新**，避免控制信号丢失导致“一直限速/一直放开”

- **方式 A：消费者侧事件表（精细可控）**
  - 适合：希望自己掌控重试与状态，而不是依赖 MQ 默认 DLQ 行为
  - 记录：重试次数、ACK 状态、异常状态、最后错误、下次重试时间
  - 可选择禁止进入 DLQ，完全由“表 + 任务”驱动重试与人工介入

- **方式 B：MQ 自带重试队列 + DLQ（快速落地）**
  - 失败进入重试队列（TTL/延迟形成重试间隔）
  - 超过上限：推荐进入 **DLQ**（除非明确低价值可丢弃）

- **强兜底：MQ 挂了怎么办**
  - 对重要且高时效业务：消费端定时扫描业务表，处理“长时间处理中”的任务，触发补偿或告警

#### 7）最小可落地清单（照着做就能跑）

##### 7.1 Outbox 表字段建议

- `event_id`（唯一键/幂等键）
- `biz_key`（如 `order_id`）
- `event_type`
- `payload`（JSON）
- `status`（NEW/SENT/FAILED/DEAD）
- `retry_count`
- `next_retry_at`
- `last_error`
- `created_at` / `updated_at`

##### 7.2 消费幂等（必须）

- 用 `event_id` 或 `biz_key + event_type` 去重（落库或幂等表）
- 建议“处理成功再 ACK”，并确保重复消息不会重复扣款/重复加积分

##### 7.3 告警与人工介入（必须）

- 进入 `DEAD/NEED_MANUAL` 或 DLQ 即告警
- 后台能力：查询失败原因、手动重放、手动标记完成/忽略（带审计）

#### 8）选型快速结论

- **通用首选**：本地事务 + Outbox 事件表 + 异步投递（可控、好排障）
- **RocketMQ 场景**：可考虑事务消息，降低“发送失败补偿”复杂度
- **无论哪种**：消费者幂等 + 重试上限 + DLQ/人工介入 是必配“三件套”

### 13、落地案例（AT / TCC / Outbox）

> 目标：把“方案名”落到“你真的要写的表、接口、流程与兜底”。看完你就能在作业/方案/面试里讲清楚“怎么做”。

#### 13.1 案例 A：Seata AT（订单下单：落单 + 扣库存 + 扣余额）

- **业务背景**：用户下单必须同时满足：订单创建成功、库存扣减成功、余额扣减成功；任一步失败都要整体失败（不允许中间态）。
- **选择理由**：三步都是典型数据库写操作（CRUD），适合 **AT** 的“低侵入 + UndoLog 回滚”。
- **简化流程**：
  - 订单服务：`@GlobalTransactional` 开启全局事务
  - 三个服务各自 `@Transactional` 做本地写
  - 任一服务抛异常 → 全局回滚（AT 通过 UndoLog 反向补偿）
- **关键落地要点**：
  - **全局事务里别做慢调用**：支付外部渠道/短信/物流都不要放进来（放进来就是长事务，锁冲突会放大）
  - **热点冲突预案**：秒杀/热点 SKU 很容易全局锁冲突，AT 可能扛不住（这类更常用“预扣 + MQ”）
  - **排障要能定位 XID**：日志里打印 XID，方便把三方服务串起来查

#### 13.2 案例 B：Seata TCC（优惠券/库存冻结：Try 冻结 → Confirm 扣减 → Cancel 解冻）

- **业务背景**：下单时要“先占住资源”，避免并发下超卖/重复用券；但下单后可能支付失败，需要释放资源。
- **选择理由**：这是“先冻结、后确认/取消”的典型场景，适合 **TCC**。
- **表设计（示例）**：
  - **冻结表/事务表**：`freeze_id(tx_id)`, `biz_key(order_id)`, `resource_id(coupon_id/sku_id)`, `state(TRY/CONFIRMED/CANCELED)`, `expire_time`
  - **唯一约束**：`tx_id` 唯一，保证幂等（重复 Try/Confirm/Cancel 不会乱扣）
- **接口约定（示例）**：
  - `Try(orderId, couponId)`：写冻结记录（TRY），把券状态改为“冻结/占用”
  - `Confirm(orderId)`：把冻结记录改 CONFIRMED，并把券真正标记“已使用”
  - `Cancel(orderId)`：把冻结记录改 CANCELED，并把券恢复“可用”
- **必须兜住的 3 个坑**：
  - **幂等**：Try/Confirm/Cancel 都可能被重复调用
  - **空回滚**：Try 没成功也可能收到 Cancel，需要能安全返回成功
  - **防悬挂**：Cancel 先到、Try 后到时，Try 必须拒绝或识别已取消（避免“又冻结起来”）
- **超时释放**：定时任务扫 `expire_time < now and state=TRY` → 走 Cancel（避免资源长期冻结）

#### 13.3 案例 C：MQ 最终一致（Outbox：支付成功后通知“积分/发货/短信”）

- **业务背景**：支付成功后要做很多后置动作（加积分、发货单、短信、数据仓库），允许短暂延迟，但最终要做完。
- **选择理由**：这些都是“非核心后置动作”，用 **Outbox + MQ** 能抗高并发、解耦、可重试。
- **发送端（支付服务）流程**：
  - 本地事务里同时写：
    - 支付表：把 `pay_status` 改为 `SUCCESS`
    - Outbox 表：插入 `PAY_SUCCESS` 事件（`event_id`, `biz_key=order_id`, `payload`, `status=NEW`）
  - 事务提交后：后台发送器把 `status=NEW` 的事件投递到 MQ，成功后标记 `SENT`
- **消费端（积分/发货/短信）流程**：
  - 先用 `event_id` 或 `biz_key+event_type` **落库去重**（唯一索引）
  - 业务处理成功后再 ACK
  - 失败就重试；超过上限进 DLQ/DLT，并告警 + 支持人工重放
- **对账兜底（强烈建议）**：
  - 定时任务扫“支付成功但发货/积分未完成”的订单，触发补偿（因为 MQ/消费者不可能 100% 不出故障）
