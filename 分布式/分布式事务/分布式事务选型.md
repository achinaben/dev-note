## 一页决策树

1. **能否接受中间状态？**
   - **不能**（钱、库存、核心订单状态等）→ 优先 **Seata 强一致链路**（AT / TCC / XA）。
   - **能**（通知、积分、报表、对接外部系统）→ 优先 **MQ 最终一致**（Outbox + 重试 + 幂等 + 死信/人工兜底）。

2. **主要矛盾是「数据库写一致」还是「纯 API 也要一起成/一起败」？**
   - **以库表 CRUD 为主** → 优先 **AT**（需 JDBC、代理数据源、能接受全局锁）。
   - **要冻结/预留资源（库存、券、余额），或没有库写也要强一致** → **TCC**。
   - **流程很长、外部系统多、允许靠补偿收回** → **SAGA**；**非核心异步** → **MQ 最终一致**。

3. **三句话口诀（背）**
   - **能用 AT 就用 AT**：跨库写、链路不长、能接受全局锁与冲突成本。
   - **要冻结/预留就用 TCC**：库存/资金/券，或纯 API 强一致（必须自己做幂等、空回滚、防悬挂）。
   - **链路长/外部多/允许中间态** → **SAGA 或 MQ**：长流程可补偿用 SAGA；通知类异步用 MQ。

## Seata 与 MQ 怎么选

| 维度 | Seata（AT/TCC/XA 等） | MQ 最终一致性 |
| --- | --- | --- |
| 一致性 | 强一致（AT/TCC）或 SAGA 偏最终一致 | 最终一致 |
| 性能 | 较低（协调、锁、网络往返多） | 较高（异步、无全局锁） |
| 侵入性 | 中（AT 相对低，TCC/SAGA 要高） | 高（Outbox、补偿、重试、幂等都要设计） |
| 典型场景 | 核心流程、实时性高、不允许关键中间态 | 非核心、可短暂不一致、高并发削峰 |

**是否必须 Seata + MQ 一起上？**  
**不强制。** 多数情况是**二选一**（同一类一致性需求选一种主方案），或**各管一段**：同步核心用 Seata，异步周边用 MQ。

**常见组合**：同一订单里核心几步 Seata 一起提交/回滚；成功后发 MQ 做短信、积分、数仓等。

**易错点**：MQ **消费者**失败**不会**自动让生产者侧已提交的 Seata 全局事务回滚；异步链要靠 Outbox、消费幂等、重试/DLQ、对账补偿闭环。

## Seata 内部：AT / TCC / SAGA / XA

| 方案 | 一致性 | 侵入 | 前提/依赖 | 优点 | 典型代价 | 典型场景 |
| --- | --- | --- | --- | --- | --- | --- |
| **AT** | 强一致（对库写友好） | 低～中 | 代理数据源、RDB、ACID、UndoLog/全局锁 | 改造小，CRUD 好落地 | 全局锁冲突、长事务、慢 SQL/外部 HTTP 拖垮链路 | 订单-库存-账户跨库写 |
| **TCC** | 强一致（可覆盖纯 API） | 高 | Try/Confirm/Cancel，资源可预留 | 不绑数据源代理，可控 | 幂等/空回滚/防悬挂/超时补偿复杂，冻结降并发 | 冻结库存、冻结余额、原路退款 |
| **SAGA** | 最终一致（补偿） | 中～高 | 每步有补偿，允许中间态 | 不必长时间锁资源，适合长流程 | 补偿可能无法完全撤销（外部不可逆） | 旅行预订、审批流、跨系统长链路 |
| **XA** | 强一致（标准 2PC） | 中 | DB/驱动支持 XA，性能敏感时要谨慎 | 语义清晰 | 锁持有久、吞吐差、高并发慎用 | 与 AT 并列考虑时的「标准 2PC」路径 |
| **MQ 最终一致** | 最终一致 | 高 | Outbox/事务消息/CDC、重试、幂等、DLQ | 吞吐高、解耦 | 中间态与工程化成本高 | 通知、积分、报表、日志 |

**XA vs AT（工程直觉）**：都与「跨库原子提交」相关；XA 更偏数据库原生 2PC，AT 是 Seata 用 UndoLog + 全局锁的工程化实现，选型时一并评估锁与长事务风险即可。

## 决策 Checklist（逐条过）

- **是否允许中间状态？**  
  允许 → MQ；不允许 → Seata。

- **性能是否要求极高？**  
  极高 → 倾向 MQ（异步无全局锁）；否则可评估 Seata。

- **是否跨多个外部系统？**  
  是 → 倾向 MQ（避免外部不稳定拖死核心同步链路）；仅内部微服务 → 可评估 Seata。

- **业务复杂度？**  
  简单 CRUD 强一致 → **AT**；要自定义预留/强一致且复杂 → **TCC**（若可接受最终一致也可改 MQ + 状态机）。

## 混合使用（各管一段）

**订单类常见拆法**：

1. **核心（Seata AT）**：创建订单 + 扣库存 + 创建支付单 —— 任一步失败全局回滚。
2. **非核心（MQ）**：成功后发消息 → 短信、积分、数仓等，失败重试 + DLQ + 对账。

**秒杀/大促注意**：全链路 Seata AT 易被全局锁打爆，常改为「预扣库存 + 短链路 + MQ 异步确认」等，在性能与正确性之间再权衡。

## 写库与发消息：常见误区与分流

| 诉求 | 更贴近的做法 |
| --- | --- |
| 落库与发通知最终一致、进程重启后仍要继续发 | **Outbox**、**事务消息（如 RocketMQ）**、**CDC** |
| 只当场多试几次、可接受崩溃丢待发 | **SDK 重试**，不强制消息表 |
| 强绑定「本请求内库与通知全成或全败」、能接受同步绑 MQ | **COMMIT 前同步发 MQ，失败回滚本地库**（必须配幂等与超时；MQ 故障会拖 RT） |

**Outbox vs 事务消息（一句话）**：通用、好排障 → **本地事务 + Outbox**；已深度用 RocketMQ → 可评估**事务消息**降低发送补偿复杂度。**无论哪种**：消费端 **幂等 + 重试上限 + DLQ/人工** 是必配三件套。

## 落地最小清单

**强一致（AT/TCC/XA）**

- 避免长事务：慢 SQL、外部 HTTP、长逻辑不要放进全局事务。
- 可观测：日志能关联 **XID / BranchId** 与失败原因。
- 热点行/SKU：要有锁冲突降级或拆分预案。

**最终一致（MQ + Outbox）**

- 业务数据与 Outbox 事件在**同一本地事务**写入。
- **重试 + 上限 + DLQ**，失败可人工介入。
- 消费端 **event_id 或 biz_key+event_type** 落库去重（幂等）。
- 核心结果（支付、发货等）要有**对账/定时补齐**。

**TCC/SAGA 补充**

- TCC：Try/Confirm/Cancel **全幂等**，处理**空回滚**、**防悬挂**，冻结带 **expire_time**。
- SAGA：补偿 **幂等可重入**，补偿失败有人工/任务通道。
