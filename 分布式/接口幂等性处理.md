## CRUD 与幂等性（直观判断）

- **select（查询）**：天然幂等（查多少次结果一致，前提是数据本身不变）
- **delete（删除）**：通常幂等（同一条数据删除多次，最终效果一样：不存在）
- **update（直接更新为某个值）**：幂等（例如把状态改为“已完成”）
- **update（累加/扣减类更新）**：非幂等（例如余额 `+1`、库存 `-1`，重复执行会被多算）
- **insert（新增）**：非幂等（每次都会插入一条新记录）

## 引入幂等性后对系统的影响

幂等性主要是为了简化客户端处理（防重复提交、失败重试更安全），但会增加服务端复杂度与成本，典型影响：

- **并行变串行**：为了保证同一业务请求只处理一次，往往需要加锁/排队，降低吞吐与执行效率
- **额外业务控制逻辑**：需要设计幂等键、去重表、唯一索引、状态机等，使业务更复杂

使用时要评估是否真的需要：除非业务明确要求“重复请求不能产生额外副作用”，一般接口不必强行引入幂等。

## 接口幂等性的必要场景与处理方案

### 1）前端按钮重复点击（重复提交）

- **现象**：用户多次点击按钮，发起多个相同请求
- **处理**
  - **前端防重复提交**：按钮置灰、loading、防抖/节流等（降低发生概率）
  - **页面令牌（token）**：一次页面交互只允许提交一次
    - **实现思路（示例）**
      - 刷新页面时生成一个 `uuid` 作为页面令牌
      - 服务端校验并“只消费一次”
      - 可用 **Redis** 存放 `token -> 已使用/未使用`，消费后立刻标记或删除
      - 需要更强一致性时，可结合 **分布式锁** 或 Redis 原子操作（如 setnx）保证并发只成功一次

### 2）网络层重发（网络抖动/超时）

- **现象**：请求/响应丢失导致客户端重试，服务端可能收到多次相同请求
- **处理**
  - **服务端必须可幂等**（尤其是支付、下单、扣库存等有副作用的接口）
  - 客户端重试应携带同一个业务幂等键（同一次业务动作不变）

### 3）Nginx 重试导致的重复请求

- **现象**：Nginx upstream 失败时可能自动重试到其他节点，导致服务端收到重复请求
- **处理**
  - **关闭不必要的重试**（示例配置）

```nginx
proxy_next_upstream off;
```

> 注：是否关闭要结合业务与可用性权衡；对“有副作用的写请求”更应谨慎重试。

### 4）开放接口（外部系统接入/失败重试/反复调试）

- **现象**：外部系统会因为失败重试、调试反复调用，导致写接口重复执行
- **处理（常见组合拳）**
  - **事务**：保证一次处理内部一致
  - **先查再做（基于必然结果判断是否已执行）**
    - 例如：通过“业务主键 + 状态”判断是否已成功落库/已流转到某状态
    - 已完成则直接返回成功（不再重复执行）
  - **insertOrUpdate / exit if exist**
    - 通过唯一键实现“存在则不插入/更新”
  - **数据库唯一索引**：用 DB 的唯一约束做最终兜底
  - **分布式锁**：以“业务主键 + 状态”等作为锁 key，避免并发重复执行

#### 例：支付接口（银联类场景）

- 外部请求携带：
  - **source**：来源系统
  - **seq**：请求序列号（同一业务动作固定）
- 在数据库对 `source + seq` 建 **唯一索引**
  - 并发重复请求时，只有一个能成功落库，其他会因唯一冲突而失败/被识别为重复
  - 业务上可将“唯一冲突”转换为“已处理/重复请求”的正常响应

### 5）秒杀（恶意刷单/高并发）

- **现象**：用户/脚本高频请求，造成重复下单/资源被打爆
- **处理**
  - **限流**：按 **IP、userId** 做 QPS/并发限制（网关/Nginx/应用层均可）
  - 写接口仍建议具备幂等控制（避免重复扣库存/重复创建订单）

### 6）消息重复消费（至少一次投递）

- **现象**：消息中间件通常保证“至少一次”，同一消息可能被重复投递/重复消费
- **处理**
  - **消息事件表（去重表）**
    - 以 `messageId` 或 “业务唯一键” 做唯一索引
    - 消费前先落表/校验：已存在则直接跳过（或返回成功）

## 落地时常用的“幂等手段清单”

- **业务幂等键**：请求必须携带（客户端生成/服务端返回后复用）
- **数据库唯一索引**：最可靠的最终防线（并发下也有效）
- **去重表/幂等表**：记录已处理的请求（请求号、业务键、状态、时间）
- **分布式锁**：控制同一业务键并发只执行一次（注意锁超时与续租）
- **状态机**：用“合法状态流转”阻止重复执行副作用
- **限流**：减少重复请求与恶意请求的压力

## 幂等键（Idempotency Key）怎么设计（最重要）

你可以把“幂等键”理解成：**同一件事的唯一身份证**。同一件事重试 N 次，身份证不变；服务端看到同一身份证，就只做一次。

### 1）幂等键放哪里

常见 3 种方式，按推荐度排序：

- **HTTP Header**：`Idempotency-Key: <key>`（语义最清晰，推荐）
- **请求体字段**：`requestId` / `bizRequestNo`（最常见，易落地）
- **URL 参数**：不推荐（易被日志/缓存/代理影响，也容易被误改）

### 2）幂等键由谁生成

- **客户端生成（推荐）**：同一次业务动作生成一次（如 UUID / Snowflake），重试时复用同一个 key
- **服务端下发 token**：页面加载/预下单时返回 `token`，提交时带上（“一次性令牌”场景很合适）

### 3）幂等键的粒度（key 代表“哪一件事”）

必须写清楚“同一件事”的边界，否则会误伤正常请求：

- **下单**：同一次点击“提交订单”算一件事（key 与一次提交绑定）
- **支付扣款**：同一次支付意图算一件事（最严格，必须幂等）
- **扣库存**：通常跟“订单号/订单行”绑定（不要只用 userId）

### 4）服务端保存什么（推荐的最小集合）

至少要能回答：**这个 key 处理过没有？结果是什么？**

- **key**
- **业务唯一标识**：例如 `orderId` / `paymentId`
- **处理状态**：`PROCESSING / SUCCESS / FAIL`
- **响应快照（可选但强烈建议）**：上次的返回码+返回体（方便“重复请求返回一致结果”）
- **过期时间（TTL）**：避免无限增长

### 5）TTL（过期时间）怎么定

按业务后果来定，越关键 TTL 越长：

- **支付/扣款**：建议至少保留到对账结束（如 7～30 天）
- **下单**：通常 1～24 小时（视业务是否允许重复创建）
- **普通写接口**：30 分钟～2 小时（覆盖常见重试窗口）

### 6）返回规范（重复请求时怎么回）

两条原则：

- **重复请求要返回“同一个业务结果”**（最好连 `orderId` 都一致）
- **不要让客户端因为你返回不一致而继续疯狂重试**

常见做法：

- **第一次成功**：返回 `200/201` + 业务数据
- **重复请求**：返回 `200` + **与第一次一致的业务数据**（从“响应快照”取）
- **正在处理**：返回 `202 Accepted` 或业务码 `PROCESSING`（提示客户端稍后查询）

## 典型接口的幂等落地示例（可照抄的流程）

下面用“写接口最容易出事”的 3 类做模板：下单、支付、扣库存。

### 示例 A：创建订单（insert 非幂等 → 需要幂等）

**推荐方案：幂等键 + 订单唯一索引（或幂等表）**

- **请求**：`requestId`（或 `Idempotency-Key`） + 购物车内容 + 收货信息
- **服务端流程（简化版）**
  - 用 `requestId` 先查幂等记录
    - **已 SUCCESS**：直接返回同一个 `orderId`（以及同样的返回体）
    - **PROCESSING**：返回“处理中”，让客户端轮询查单
  - 写入幂等记录：状态 `PROCESSING`
  - 创建订单（建议给“订单业务号”加唯一约束）
  - 更新幂等记录：状态 `SUCCESS` + 保存 `orderId` + 保存响应快照

**关键点**

- **最终兜底必须靠 DB 唯一约束**：只靠 Redis 锁不够（锁丢了仍可能重复插入）
- **重复请求返回同一个订单**：用户体验最好，也避免二次扣款/二次发货链路

### 示例 B：支付扣款（最严格的幂等场景）

**推荐方案：外部支付单号唯一 + 状态机 + 幂等键**

- **请求**：`paymentRequestNo`（商户侧请求号，等价于你上面说的 `source+seq`）
- **服务端流程（简化版）**
  - 以 `paymentRequestNo` 查询支付单
    - **已 SUCCESS**：直接返回成功（带上同一个 `paymentId`/交易号）
    - **PROCESSING**：返回处理中（提示稍后查询）
    - **不存在**：创建支付单（`paymentRequestNo` 唯一索引），状态 `PROCESSING`
  - 调用支付渠道
  - 按回调/结果更新状态机：`PROCESSING -> SUCCESS/FAIL`

**关键点**

- **“支付单”本身就是幂等载体**：一切以支付单状态为准
- **回调也要幂等**：同一渠道回调可能多次到达，用“交易号/回调流水号”去重

### 示例 C：扣库存（update 扣减非幂等 → 必须防重复扣）

**推荐方案：用“订单号/订单行”做库存扣减唯一键**

- **请求**：`orderId`、`orderItemId`（或 `deductRequestNo`）
- **服务端流程（简化版）**
  - 先查“库存扣减记录”是否存在（`orderItemId` 唯一）
    - **存在**：直接返回已扣减（避免重复扣）
  - 执行扣减（乐观锁版本号或库存充足校验）
  - 插入扣减记录（唯一约束兜底）

**关键点**

- **扣减记录（事件表）很重要**：它让“扣过没”这件事可追溯、可对账

## 最小落地清单（你写接口时可以对照）

- **是否有副作用？**（扣钱/扣库存/发券/发货/写入新记录）→ 有就要认真做幂等
- **幂等键是什么？**（requestId / source+seq / orderId+action）
- **最终兜底在哪里？**（DB 唯一索引/约束）
- **重复请求返回什么？**（同结果/处理中/可查询）
- **TTL 多久？**（至少覆盖重试窗口，关键业务覆盖对账周期）

## 幂等表 / 去重表设计（SQL 示例）

这类表的目标是：**同一个幂等键只允许“第一次成功写入”，其余都能识别为重复**。

### 1）接口幂等表（适合 HTTP 写接口）

字段建议（最常用的一套）：

- `idem_key`：幂等键（唯一）
- `biz_type`：业务类型（例如 `CREATE_ORDER`、`PAY`）
- `biz_id`：业务 ID（例如 `orderId`、`paymentId`）
- `status`：`PROCESSING/SUCCESS/FAIL`
- `req_hash`：请求摘要（可选，用于检测“同 key 但参数不同”的异常）
- `resp_code`、`resp_body`：响应快照（建议保存，重复请求返回一致）
- `created_at`、`updated_at`、`expire_at`

MySQL 示例：

```sql
CREATE TABLE api_idempotency (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  idem_key VARCHAR(128) NOT NULL,
  biz_type VARCHAR(64) NOT NULL,
  biz_id VARCHAR(64) NULL,
  status VARCHAR(16) NOT NULL,
  req_hash CHAR(64) NULL,
  resp_code VARCHAR(32) NULL,
  resp_body JSON NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  expire_at DATETIME NULL,
  UNIQUE KEY uk_idem_key (idem_key),
  KEY idx_expire_at (expire_at),
  KEY idx_biz_type_biz_id (biz_type, biz_id)
);
```

写入方式（核心就是“先占坑”）：

- 第一次请求：插入一条 `PROCESSING`
- 处理完成：把该行更新成 `SUCCESS` 并写入 `biz_id/resp_body`
- 重复请求：查询到 `SUCCESS` 就直接返回快照；查到 `PROCESSING` 就返回处理中

> 注意：如果你担心 `resp_body` 太大，可以只存关键字段（如 `orderId`）或存一份压缩后的字符串。

### 2）消息去重表（适合 MQ 消费幂等）

用 `message_id` 或“业务唯一键”做唯一索引：

```sql
CREATE TABLE mq_consume_dedup (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  topic VARCHAR(128) NOT NULL,
  message_id VARCHAR(128) NOT NULL,
  biz_key VARCHAR(128) NULL,
  status VARCHAR(16) NOT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_topic_msg (topic, message_id),
  KEY idx_biz_key (biz_key)
);
```

消费时做法：

- 先尝试插入（占坑），插入成功才执行业务
- 插入失败（唯一冲突）说明重复消息 → 直接当成成功返回（不再执行业务）

## Redis 原子实现（setnx / Lua）示例

Redis 更适合做“短期幂等”（比如按钮重复点击、短时间重试窗口），优点是快、TTL 好控；缺点是**不能替代数据库唯一索引作为最终兜底**。

### 1）setnx + TTL（最常用）

语义：**第一次设置成功才允许执行**，并且要带 TTL 防止 key 永久占用。

```text
SET idem:{key} 1 NX EX 300
```

- 成功（返回 OK）：说明第一次请求 → 继续执行业务
- 失败（返回 nil）：说明重复请求 → 直接返回“已处理/处理中”

### 2）Lua：原子“只消费一次 token”（防并发）

适用：你用“页面 token”时，希望做到 **校验 + 标记已使用** 一次性完成。

思路：

- token 初始值：`0`（未使用）
- 消费成功后写成：`1`（已使用）

Lua 示例（原子判断并置位）：

```lua
-- KEYS[1] = token key
-- 返回：1=消费成功；0=已被消费/不存在
if redis.call('EXISTS', KEYS[1]) == 0 then
  return 0
end
local v = redis.call('GET', KEYS[1])
if v == '1' then
  return 0
end
redis.call('SET', KEYS[1], '1')
return 1
```

### 3）工程落地建议（避免踩坑）

- **永远设置 TTL**：避免宕机/异常导致“永远不能再提交”
- **写接口最终兜底靠 DB**：Redis 只能做“第一道门”，DB 唯一约束做“最后一道门”
- **处理中状态要可查询**：尤其是下单/支付，返回“处理中”时要给一个查询接口或订单号

## 同一个幂等键但请求参数不同，怎么办（req_hash 规范）

这是幂等实现里最容易被忽略、但非常关键的一点：**幂等键代表“同一件事”**，所以理论上“同一个幂等键”对应的请求参数必须一致；否则就属于“同号不同单”的风险请求。

### 1）为什么会出现

- 前端 bug：复用旧的 `requestId`，但用户修改了地址/优惠券/商品
- 外部系统 bug：重试时把部分字段改了
- 恶意请求：用同一个 key 试探系统

### 2）推荐处理策略（强烈建议采用）

服务端在首次受理时计算并保存 `req_hash`，后续重复请求带相同 `idem_key` 时做校验：

- **`req_hash` 一致**：当成正常重试 → 返回之前的结果（或处理中）
- **`req_hash` 不一致**：当成异常请求 → **拒绝**，避免把“另一件事”误当成重试

### 3）`req_hash` 怎么算（原则）

- 只对“决定业务结果”的字段做 hash（比如商品、数量、收货地址、优惠、金额）
- 过滤掉会变化但不影响业务的字段（比如时间戳、签名、客户端版本号）
- 推荐：对规范化后的 JSON（字段排序、去空）做 SHA-256 得到 `req_hash`

### 4）返回码/返回体约定（给客户端明确预期）

建议定义一个明确的业务错误码，例如：

- **`IDEMPOTENCY_KEY_CONFLICT`**：同一个幂等键，但请求参数不一致
- 返回信息要包含：`idem_key`、（可选）服务器记录的 `req_hash` 与本次的 `req_hash`（便于排查）

### 5）一条最重要的落地原则

**幂等键要么由客户端“每次动作生成一次”，要么由服务端“下发一次性 token”。**
不要让客户端随意缓存/复用旧 key，否则很容易产生“同 key 不同参数”的冲突。
