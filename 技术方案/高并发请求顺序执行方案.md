# 高并发请求：顺序执行方案（MQ 物理保障）

**本文目标**：把 **「请求顺序」** 的保障责任从分布式锁上挪开——**顺序性应由消息队列的队列语义（单分区 / 单队列 + 顺序消费）来物理保障**；分布式锁仍负责 **跨实例互斥、短临界区**，而不是全链路的 **FIFO**。读完能定方案：要强顺序时入口怎么收口、分区键怎么选、锁放在哪一层。文末 **第七节** 给出 **下单** 落地范例：**请求顺序执行 = 订单顺序生成 = 库存顺序扣除**（同一 SKU 竞争维度下由 MQ 单管道对齐）。

## 一、结论先行：谁保顺序、谁保互斥

| 目标 | 更合适的承载 | 分布式锁的角色 |
| --- | --- | --- |
| **同一串行域内请求按序被处理**（如 FIFO） | **MQ：同一队列 / 同一分区 + 分区内顺序消费**（必要时单消费线程） | **不作为主方案**；公平锁仅覆盖「仍在同步抢锁」的一小撮竞争者，且双路径时与 MQ 顺序不是同一套队列 |
| **同一时刻只有一个执行者改同一资源** | 可 **Consumer 内抢锁** 或 **DB 行锁/条件更新** | **主用场**：短临界区、多实例互斥 |
| **削峰、最终办成** | MQ + 幂等 + 可靠投递 | 锁不负责 |

**一句话**：**顺序性应由 MQ 的「单管道」语义来保障；分布式锁保障的是互斥与临界区安全，不是全链路请求顺序。**

---

## 二、为什么顺序不该主要靠分布式锁

1. **非公平锁**（最常见）：只保证互斥，**不保证**先到的请求先拿到锁。  
2. **公平锁**：只在 **同步、且所有竞争者都走这把锁** 时，改善「等待者之间的先后」；成本高，且 **一旦抢锁失败改走 MQ**，请求已离开锁的排队语义，**顺序改由 MQ 决定**。  
3. **双路径**（同步持锁 + 失败入队）：两条执行通道可并行，**全链路完成顺序** 不由锁单独决定，见《分布式锁功能》第七节 7.4。

因此：**要强顺序，应先把「排好序的流」放进一条 MQ 管道里，而不是指望锁在应用层「猜」顺序。**

## 三、用 MQ「物理」保障顺序的要件

「物理保障」指：**同一顺序域内的消息，在存储与投递维度上走同一条有序序列**（逻辑上的单队列或 Kafka 等模型的 **单分区**）。

| 要件 | 说明 |
| --- | --- |
| **顺序域 / 分区键** | 与业务一致，如 `userId`、`aggregateId`；**下单抢同一库存**时常用 **`skuId` / `活动+skuId`**（见第七节），**不宜用尚未生成的 `orderId`**；**同一域内所有要保序的消息必须进同一分区**（或同一经典队列） |
| **分区内顺序消费** | 消费端 **不要** 对同一分区多线程并发处理（除非框架明确支持且你理解其语义）；以所用 MQ 官方文档为准 |
| **单生产者顺序写入** | 同一分区内由 **单逻辑序列** 追加；多进程乱序写入同一分区会破坏顺序（设计入口聚合或分区规则时需注意） |
| **幂等与重试** | 重试、重复消费可能打乱「业务上」的观感，需 **幂等 + 业务序号/版本** 兜底，见《消息队列》 |

不同产品（Kafka、RocketMQ、RabbitMQ 等）对「顺序消息」的 API 与限制不同，**上线前以当前版本官方文档为准**。

## 四、推荐主路径：先入队，再顺序消费

高并发下若 **顺序优先**：

1. **接口侧**：校验 → **限流**（保护入口）→ **按顺序域路由** 投递到 MQ（保证同一域进同一分区/队列）。  
2. **消费侧**：**顺序拉取 / 单线程处理该分区** → 执行业务 → 必要时在 **Consumer 内** 对极短路径抢分布式锁或开事务写库。  
3. **查询**：异步结果通过 **查询接口 / 推送** 返回，不把「顺序」压在 HTTP 同步返回链路上。

这样，**从「进入 MQ 之后」的处理次序**由 **单管道** 约束，比「多实例抢公平锁」更贴近「物理顺序」。

---

## 五、与分布式锁如何配合（不抢活）

- **锁**：用于 **Consumer 内** 与 **其它入口**（如定时任务、补偿）对 **同一资源** 的 **互斥**，防止和别的通道并发写坏数据。  
- **顺序**：仍由 **MQ 分区 + 顺序消费** 保证；**不要用「全局一把锁」冒充顺序队列**。  
- 若存在 **同步接口 + MQ 兜底** 双路径：Consumer **应抢与同步路径相同的锁 key**，保证 **互斥**；**顺序** 仍依赖 **消息入队次序** 与 **分区策略**，不是锁的职责。

## 六、常见坑

- **分区键选错**：同一订单的消息进了不同分区 → **顺序丢失**。  
- **同一分区多线程消费**：框架若未保证顺序，**处理完成次序** 可能乱。  
- **以为公平锁 = 全站顺序**：公平锁只排序 **锁的等待者**，不是 **全系统请求顺序**。  
- **热点分区**：顺序域过粗（如全用一个 key）→ **单分区成为瓶颈**；需在 **顺序** 与 **并行度** 之间折中（按业务拆域）。

## 七、落地范例：下单（请求顺序 = 订单顺序 = 扣库顺序）

**场景**：高并发抢购、限量 SKU 等，产品上要满足 **先请求先得到结果链路上的一致顺序**，可抽象为同一等式（在**同一竞争资源**维度下）：

> **请求被处理的顺序 = 订单（或等价凭证）生成的先后顺序 = 库存扣减发生的先后顺序**

三者要对齐，靠的不是「多实例抢公平锁排 HTTP」，而是 **同一顺序域内只有一条 MQ 管道**，消息按投递先后被 **顺序消费**，在 Consumer 里 **按序** 完成「创单 + 扣库」。

### 7.1 顺序域怎么选（关键）

| 顺序域（分区键） | 典型取值 | 说明 |
| --- | --- | --- |
| **竞争同一库存池** | `skuId`，或 `活动ID + skuId` / `仓 + skuId` | 所有抢**同一库存**的请求必须进 **同一分区**，管道内才是「谁先入队谁先被处理」 |
| **不宜用** 尚未生成的 `orderId` 做分区键 | — | 下单前没有订单号；若用用户 ID 做主键，则**不同用户之间**本可并行，但**同一 SKU 下多用户抢**仍应对齐到 **SKU（或活动+SKU）** 维度的顺序管道 |

**结论**：要强绑「扣库顺序」，分区键应锚在 **库存竞争单元**（常见即 **SKU / 活动 SKU**），而不是订单号。

### 7.2 链路怎么接（推荐）

1. **接入**：参数校验 → **限流** → 生成 **幂等键**（如 `用户 + 活动 + 前端请求号`）→ **按分区键投递**「下单意图」消息（body：用户、SKU、数量、幂等键等）。  
2. **消费（单分区顺序）**：**同一分区内单线程（或框架保证的顺序消费）** 逐条处理：  
   - 先做 **幂等**（防重复消息）；  
   - 再 **事务内**：生成订单（或写订单流水）+ **`UPDATE … WHERE stock >= n`** 扣库存；  
   - 失败则按业务返回「售罄」等，**下一条消息仍按队列顺序处理**。  
3. **顺序为何成立**：同一 SKU 分区里，消息 **m1 先于 m2 被消费**，则 **m1 的创单/扣库先于 m2**（在单线程消费前提下），于是 **请求处理顺序 = 订单生成顺序 = 扣库顺序** 在**该 SKU 管道内**一致。

### 7.3 分布式锁在本场景中的位置

- **主顺序**：仍由 **MQ 单分区 + 顺序消费** 保障（本节主旨）。  
- **锁**：用于 **与其它入口互斥**（例如同步兜底、定时回补、运营后台改库）——Consumer 内可对 `lock:stock:skuId` 等与同步路径 **同 key** 抢短锁，避免双路径并发扣同一 SKU；**不要**指望用锁替代 MQ 做「全站谁先下单」的排序。  
- 若 **只有 MQ 一条路径** 写库存，且单分区单线程消费，部分实现里 **Consumer 内锁可省略**，仍以 **DB 条件更新 + 幂等** 为底线。

### 7.4 仍要注意的坑

- **分区键错用 `orderId`**：入队前尚无订单 → 无法用来把「抢同一 SKU 的人」排进一条队。  
- **同一分区多线程消费**：扣库顺序可能与消息顺序不一致 → **违背本节等式**。  
- **超卖防护**：顺序只解决「先后」；**库存不能扣成负数**仍靠 **`WHERE stock >= n`**、唯一约束、幂等，与《高并发接口串行顺序执行方案》中行锁/条件更新思路一致。  
- **热点 SKU**：单分区吞吐上限 = 单消费者处理能力；需 **压测** 与 **活动拆分**（多 SKU 天然多分区并行）。  
- **多实例同时接请求**：各节点**各自先入队**时，全局「谁先点到」与「谁先进入同一分区」未必完全一致；若产品要强绑 **全互联网 FIFO**，需在消息体带 **业务序号/统一时间线** 或由 **单一接入编排** 再写 MQ；多数活动场景只要求 **同一 SKU 库存池内** 的先后与扣库一致，本节等式即针对该维度。

- [ ] **双路径** 是否存在？若存在，Consumer 是否与同步路径 **同锁互斥**，且顺序仍 **只信 MQ 管道**？  
- [ ] 是否评估 **热点分区** 下的吞吐与延迟？
