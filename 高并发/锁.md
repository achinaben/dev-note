# 锁

## 乐观锁（Optimistic Lock）

### 核心思想
- **假设冲突很少**：认为共享资源每次被访问时大概率不会发生并发冲突，所以线程/请求可以先继续执行，**不需要先加锁等待**。
- **不“上锁”，而是在更新时校验**：更新数据时检查“在这段时间里数据有没有被别人改过”。
  - **没被改**：提交更新成功
  - **被改了**：更新失败，**重试**（通常是自旋重试）
- **本质**：不锁定资源，通过**版本记录与比较**来保证一致性，属于一种**业务逻辑层面的锁**（逻辑锁）。

---

### 常见实现方式

#### 1）CAS 机制 + 自旋重试（死循环自旋）
- **思路**：Compare-And-Swap（比较并交换）
  - 只有当“内存中的值 == 期望值”时才更新为新值
  - 否则说明被别人改过，失败后继续重试
- **关键点**：CAS 的原子性通常由 CPU 指令保证（如 `cmpxchg`）。

##### CAS 一句话理解（给初学者）
- **CAS 是一种“先假设没人改，再提交时检查”的更新方式**：提交那一刻，如果发现数据还和我当初看到的一样，就更新；不一样就失败并重试。

##### CAS 的 3 个操作数（记住 V / A / B）
- **V**：要更新的内存位置（变量所在的地址）
- **A**：旧的期望值（expected）
- **B**：要写入的新值（update）

执行逻辑：
- 如果此刻内存里的值 **== A**，就把它改成 **B**，并返回成功
- 否则什么都不改，返回失败

> 关键点：这个“比较 + 更新”在 CPU 层面是原子的（不可被拆开插队）。

##### Java 里最常见的 CAS：原子类（`java.util.concurrent.atomic`）
- **`AtomicInteger/AtomicLong`**：对整数做 CAS 更新（例如自增）
- **`AtomicReference`**：对“对象引用”做 CAS 更新

典型结构（理解即可）：很多“原子自增”内部都类似这样做事：
- 先读 current
- 计算 next
- CAS(current, next) 成功就结束
- 失败就说明被别人抢先改了：重新读、重新算、再 CAS（这就是自旋）

#### 2）版本号/时间戳机制 + 自旋重试
- **做法**：数据表增加 `version`（或 `update_time`）字段
- **更新时带条件**：只有版本一致才允许更新，同时把版本 +1（或更新时间刷新）

---

### 案例：版本号机制（SQL）

#### 表结构示例
- 字段：`id`, `stock`, `version`

#### 读取
- `SELECT stock, version FROM product WHERE id = ?;`

#### 更新（乐观锁）
- `UPDATE product`
  `SET stock = stock - 1, version = version + 1`
  `WHERE id = ? AND version = ?;`

#### 重试（伪代码）
- 读出 `stock, version`
- 尝试按 `id + version` 更新
- 若影响行数为 0（更新失败），则说明版本冲突，**重新读取并重试**
- 建议加入**退出机制**：限次/限时/退避（避免一直自旋）

---

### 缺点 / 注意点

#### CAS + 自旋
- **只能保证单个变量**更新的原子性；当涉及**多个变量**需要一起保持一致时，单纯 CAS 往往不够（需要更复杂的设计或借助事务/锁）。
- **ABA 问题**：值“看起来没变”，但其实变过
  - 场景：线程 1 读到 A；线程 2 把 A 改成 B 又改回 A；线程 1 再 CAS 发现还是 A，于是误以为“没人动过”，CAS 成功
  - 解决思路：给值配一个“版本号/时间戳”一起比较（值变一次，版本号 +1）
  - Java 常用：`AtomicStampedReference`（stamp 版本号）、`AtomicMarkableReference`（mark 标记，更轻但表达能力弱）
- **自旋开销**：并发冲突激烈时，CAS 会频繁失败导致 CPU 空转
  - 经验结论：热点计数场景下，`LongAdder` 往往比 `AtomicLong` 更快（把冲突分散到多个槽位，最后汇总）

#### 版本号 + 自旋
- **跨表/跨资源困难**：如果 `query` 针对表 A，但 `update` 需要改表 B（或多个表），仅靠单表 `version` 很难覆盖完整一致性需求（通常要靠事务、全局版本、或更高层协议）。
- **高冲突下浪费 CPU**：并发冲突频繁时更新会反复失败，需要不断重试，带来 CPU 消耗；即便有限次/限时，**在高并发下仍可能有失败率**。

---

### 优点
- **竞争不激烈时更有优势**：并发冲突概率小 → 失败重试少 → 吞吐更高、等待更少。

---

### 使用场景（经验结论）
- **更适合**：读多写少、冲突概率低的场景（如配置更新、资料编辑、后台管理类更新等）。
- **不太适合**：强一致、强约束、冲突很高的热点写场景（例如“下单扣库存”这种高并发热点写，通常需要更强的并发控制策略，单靠乐观锁重试容易被打满/失败）。

## 悲观锁（Pessimistic Lock）

### 核心思想
- **假设冲突很多**：认为共享资源每次被访问都“可能出问题”（例如共享数据会被别人修改），所以**访问前先加锁**。
- **真实的锁/排他锁**：操作数据时直接把数据锁住，直到操作完成才释放。
- **效果**：强制**串行化**（同一份资源同一时刻只允许一个线程/请求修改）。

---

### 常见实现方式

#### 1）语言层面的锁
- `synchronized`
- `Lock`（如 `ReentrantLock`）

#### 2）数据库锁
- **行锁**：事务中使用 `SELECT ... FOR UPDATE`（在事务提交/回滚前一直持有锁）

#### 3）分布式锁（跨进程/跨机器）
- **Redis**：基于 key 的分布式锁（如 `SET key value NX PX ...`）
- **ZooKeeper**：基于临时节点/顺序节点实现互斥

> 说明：分布式锁的实现细节很多（续租、超时、可重入、故障恢复等），这里先把它当作“悲观锁的一种实现形态”来理解即可。

---

### 优点
- **竞争激烈时更有优势**：并发冲突概率高时，悲观锁能把冲突变成等待，通常更稳定、更可控。
- **更直观**：先锁再做事，逻辑上更接近“只有我能改”的直觉。

---

### 缺点 / 风险点
- **排他影响并发**：容易降低吞吐量，等待时间变长。
- **有额外开销**：加锁/释放锁需要消耗资源（上下文切换、锁管理、网络往返等）。
- **死锁风险**：多个锁交叉获取时可能产生死锁（需要约定锁顺序/超时/死锁检测）。
- **长事务风险（数据库）**：`FOR UPDATE` + 长事务会持锁很久，导致大量阻塞，甚至拖垮库。
- **分布式锁更复杂**：需要处理超时、误删锁、服务崩溃、时钟/网络抖动等问题。

---

### 什么时候用悲观锁（经验结论）
- **更适合**：写多读少、热点数据、冲突非常频繁、且对一致性要求强的场景。
  - 例：热点库存扣减、同一账号余额扣减、同一订单状态流转（强一致且不允许并发写穿）。
- **不太适合**：读多写少、冲突概率低的场景（这类通常用乐观锁更划算）。

---

### 小例子：数据库 `FOR UPDATE`（概念理解）
- **场景**：扣减库存（同一商品 id 的库存是热点数据）。
- **做法**（伪流程）：
  - 开启事务
  - `SELECT stock FROM product WHERE id = ? FOR UPDATE;`（先把这一行锁住）
  - 判断 `stock > 0` 再 `UPDATE product SET stock = stock - 1 WHERE id = ?;`
  - 提交事务（提交时释放锁）

> 关键点：`FOR UPDATE` 必须在事务里才有意义；事务越短越好，否则会造成大面积阻塞。

## 双重检查（DCL）什么时候用？（以及分布式锁里的“二次校验”）

> 一句话：**只想让“初始化那一下”加锁**，初始化完成后绝大多数访问都不加锁，就用 DCL；而在分布式场景里，对应的是“**拿到锁后再做一次业务状态/幂等检查**”。

### 1）Java 单例（最典型的 DCL 用法）

#### 什么时候需要
- **懒加载**（第一次用到才创建对象），且并发下可能同时第一次访问
- 获取单例的方法会被**高频调用**，不想每次都 `synchronized`

#### 正确写法（必须 `volatile`）
```java
public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                 // 第一次检查：不加锁快
            synchronized (Singleton.class) {
                if (instance == null) {         // 第二次检查：防止重复创建
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

#### 更推荐（更简单，不容易写错）：静态内部类 Holder
```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

---

### 2）缓存 Map 初始化：别急着手写 DCL，优先用并发容器原子方法

#### 场景 A：整个 Map 只初始化一次
- 本质和“单例初始化”一样：可以用 DCL，但**更推荐**启动时初始化，或用 Holder/框架管理生命周期。

#### 场景 B：按 key 懒加载 value（更常见）
- 目标通常是：**同一个 key 在并发下只加载/计算一次**
- 推荐用 `ConcurrentHashMap.computeIfAbsent`，比“containsKey + 加锁 + 再 containsKey”更稳更清晰

```java
private final java.util.concurrent.ConcurrentHashMap<String, Object> cache =
        new java.util.concurrent.ConcurrentHashMap<>();

public Object get(String key) {
    return cache.computeIfAbsent(key, k -> load(k));
}
```

---

### 3）数据库/Redis 分布式锁：用“锁 + 幂等/状态二次校验”替代传统 DCL 思维

#### 为什么要二次校验
- **拿到锁 ≠ 业务一定没做过**：可能遇到超时重试、锁过期、服务重启、网络抖动等
- 所以正确做法是：**加锁成功后，再检查一次“业务是否已完成/是否已处理”**

#### 典型流程（思路）
- 尝试加锁成功（如 Redis：`SET lock:key token NX PX ...`）
- **锁内二次校验**：
  - 订单/任务是否已处理（状态字段、幂等表、唯一约束等）
  - 已处理：直接返回“已完成/已处理”
  - 未处理：执行业务 + 落库标记已处理（幂等）
- 释放锁时要防误删：只允许持有者释放（用 token 校验，常见是 Lua 脚本）

> 记忆：**锁解决“同一时刻谁能做”**，幂等/状态校验解决“这件事到底只做一次”。两者配合才稳。

---

### 4）对照案例：什么时候“查 + 锁住”就够了？什么时候仍要二次校验？

> 你提的思路“把查询 + 上锁这两个动作用方法锁包起来”在**单机同进程**里通常可行；但在**分布式/可重试**场景，仍然要靠幂等/状态校验兜底。

#### 案例 1：Java 单例（不做 DCL 的二次判空也能安全，但会慢）
- **做法**：把“判空（查询）+ 初始化（写入）”都放进 `synchronized` 方法
- **是否需要二次判空**：不需要（因为每次都串行进来）
- **代价**：每次 `getInstance()` 都要加锁

```java
public class Singleton {
    private static Singleton instance;
    private Singleton() {}

    public static synchronized Singleton getInstance() {
        if (instance == null) {           // 查询
            instance = new Singleton();   // 初始化
        }
        return instance;
    }
}
```

#### 案例 2：缓存 Map（单机同 JVM：锁住“查 + 放”就够；更推荐用原子 API）

**写法 A：一把锁包住“查 + 放”（不需要二次校验）**
- **前提**：所有访问都走同一把锁（同一 JVM、同一锁对象）
- **缺点**：并发能力差（大锁）

```java
class Cache {
    private final java.util.Map<String, Object> map = new java.util.HashMap<>();

    public synchronized Object get(String key) {
        Object v = map.get(key);          // 查询
        if (v == null) {
            v = load(key);
            map.put(key, v);              // 写入
        }
        return v;
    }
}
```

**写法 B：并发容器原子方法（不手写二次校验，语义更清晰）**
```java
private final java.util.concurrent.ConcurrentHashMap<String, Object> cache =
        new java.util.concurrent.ConcurrentHashMap<>();

public Object get(String key) {
    return cache.computeIfAbsent(key, k -> load(k));
}
```

> 注意（很容易踩坑）：
> - `computeIfAbsent` 的 `load(k)` **尽量保持“无副作用”**，不要在里面再去操作同一个 `cache`，否则容易引发复杂的递归/锁问题。
> - `load(k)` **不要返回 `null`**：返回 `null` 表示“不缓存”，下次还会再次触发加载。
> - `load(k)` 抛异常时，这次不会写入缓存；下次请求还会再触发加载（需要考虑降级/限流，避免被打爆）。

#### 案例 3：数据库行锁 `FOR UPDATE`（事务内“锁住后再查再改”，一般不需要“再查一次”）
- **目标**：防止并发写穿（比如余额/库存被并发扣错）
- **做法**：事务里先 `SELECT ... FOR UPDATE` 把行锁住，再校验并更新
- **是否需要“二次校验”**：
  - 对“并发写穿”问题：一般不需要（行被锁住了，别人改不了）
  - 但对“重复请求/重复消息”问题：仍需要幂等（见下一条）

伪流程（概念理解）：
- 开启事务
- `SELECT ... FOR UPDATE`（先锁住）
- 校验条件（余额是否够、库存是否够）
- `UPDATE ...`
- 提交事务释放锁

**仍要幂等的典型：支付回调/消息重复投递**
- 即使串行执行两次，也会“重复记账/重复发货”
- 解决：用订单状态机/幂等表/唯一约束，让“同一业务只能成功一次”

**落地 SQL 示例 1：状态条件更新（推荐，简单硬核）**
- 场景：支付回调把订单从 `UNPAID` 改为 `PAID`
- 结果：第一次更新影响行数为 1 表示成功；第二次影响行数为 0，直接当作“已处理”

```sql
UPDATE orders
SET status = 'PAID', paid_time = NOW()
WHERE order_id = ? AND status = 'UNPAID';
```

**落地 SQL 示例 2：幂等表 + 唯一约束（更通用）**
- 场景：每个请求带 `request_id`（或网关生成），保证同一请求只处理一次

```sql
-- request_id 上有 UNIQUE 约束/唯一索引
INSERT INTO idempotent_requests(request_id, created_time)
VALUES(?, NOW());
-- 若唯一冲突插入失败 => 说明已经处理过，直接返回“已处理”
```

#### 案例 4：Redis 分布式锁（跨机器：方法锁无效，必须“锁 + 幂等/状态二次校验”）
- **反例**：你在 A 机器用 `synchronized` 包住“查 + SETNX 加锁”，B 机器照样能同时进来（因为不是同一进程）
- **正确做法**：拿到 Redis 锁后，仍要检查业务是否已处理（幂等），避免：
  - 锁过期/续租失败导致并发执行
  - 请求重试/重复消息导致重复执行

典型流程（思路）：
- `SET lock:biz:{id} token NX PX 30000` 成功 → 进入
- **锁内二次校验**：查订单状态/幂等表（已处理就直接返回）
- 未处理 → 执行业务 + 把状态更新为已处理（最好用 DB 条件更新/唯一约束保证原子幂等）
- 释放锁：用 token 校验（Lua 脚本）防误删

**安全解锁 Lua（只允许“锁的持有者”释放）**
```lua
-- KEYS[1] = lock_key
-- ARGV[1] = token
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
else
  return 0
end
```

> 补充：分布式锁再“强”，也建议用 **DB 状态条件更新/唯一约束**做最终幂等兜底；这样即使锁过期、重复消息，也不会把业务做两次。

### 5）背诵版速记：一眼判断要不要二次校验
| 场景 | “查 + 锁住”是否足够？ | 还需要什么兜底？ |
|---|---|---|
| Java 单例（全方法加锁） | 足够（不需要 DCL 二次判空） | 性能差；更推荐 Holder/DCL |
| 缓存 Map（同 JVM 同锁） | 足够（锁住查+写） | 更推荐 `computeIfAbsent` |
| 数据库 `FOR UPDATE` | 对并发写穿通常足够 | 对重复请求/重复消息仍要幂等 |
| Redis 分布式锁 | 不够 | 必须“锁 + 幂等/状态二次校验” |

### 6）锁的“颗粒度”速记（你刚才问到的关键点）

> 颗粒度不是“物理锁/方法锁”谁更高级，而是看：**锁住的临界区有多大**、以及**这把锁覆盖了多大范围的并发主体**。

#### 颗粒度 = 临界区大小 + 覆盖范围
- **临界区大小**：锁住的代码/操作越多（例如整个方法），颗粒度越粗
- **覆盖范围**：锁影响的对象/主体越多（线程/连接/进程/机器），范围越广

#### 从“细”到“粗”的直觉排序（帮助建立感觉）
- **CAS / 原子变量**：通常很细（单个变量的原子更新）
- **并发容器的原子 API**（如 `computeIfAbsent`）：对单个 key 往往更细、更贴近业务语义
- **代码块锁**（只锁关键几行）
- **方法锁**（锁整个方法的临界区）
- **数据库行锁**（对象很细=一行，但覆盖范围广=跨连接/跨线程）
- **数据库表锁 / 长事务长时间持锁**：非常粗
- **分布式锁**：覆盖范围最广（跨机器）；“细不细”取决于 key 设计（锁 `order:{id}` 细，锁 `global` 很粗）

#### 一句话把“二次校验”和颗粒度串起来
- **锁越粗**：越不容易出现“检查后被插队”的竞态，但并发越差
- **锁越细**：并发越好，但更容易出现竞态，所以更需要 DCL/原子 API/状态条件更新
- **分布式场景**：就算锁很“粗”（跨机器互斥），也仍要幂等/状态校验，因为还存在**重试、重复消息、锁过期、网络抖动**等问题

## 背诵版速记（考试/面试 1 分钟说清）

### 1）一句话定义（最常考）
- **乐观锁**：先不锁，提交更新时校验“期间有没有人改过”，**失败就重试**。
- **悲观锁**：先加锁再干活，别人想改只能**等待**。

### 2）三句话讲清 CAS（背这三句就够）
- **CAS 做的事**：如果“值还是我期望的那个”，就**原子更新**；否则失败。
- **CAS 为啥要自旋**：失败说明被别人抢先改了，只能**重新读、重新算、再 CAS**。
- **CAS 两大坑**：**ABA**（值看着没变其实变过）+ **自旋耗 CPU**（竞争激烈时空转）。

### 3）对比表（背表 = 秒懂）
| 维度 | 乐观锁（CAS / version） | 悲观锁（synchronized / Lock / FOR UPDATE） |
|---|---|---|
| 核心动作 | 先做事，提交时校验 | 先上锁，再做事 |
| 冲突处理 | 冲突 → 失败 → 重试（自旋/退避） | 冲突 → 阻塞等待 |
| 适合场景 | **读多写少**、冲突低 | **写多/热点写**、冲突高、强一致 |
| 风险/代价 | 自旋耗 CPU、失败率、ABA | 阻塞、吞吐下降、死锁/长事务风险 |

### 4）3 个超短例子（直接套用）
- **例 1：CAS（内存里自增）**：多个线程抢着把计数 +1，用 `AtomicInteger` 反复 CAS，失败就重试。
- **例 2：version（数据库更新）**：`UPDATE ... WHERE id=? AND version=?`，影响行数为 0 就重试（最好限次/退避）。
- **例 3：FOR UPDATE（数据库强互斥）**：扣库存先 `SELECT ... FOR UPDATE` 把行锁住，再更新，事务提交后释放锁。

### 5）一句口诀（背下来不容易忘）
- **“乐观先跑，提交再考；考不过就重跑。”**
- **“悲观先锁，别人等我。”**

## 乐观锁 vs 悲观锁（面试对比表）

| 维度 | 乐观锁（Optimistic） | 悲观锁（Pessimistic） |
| --- | --- | --- |
| **核心假设** | 冲突少，“大多数时候不会撞车” | 冲突多，“随时可能撞车” |
| **做事顺序** | 先执行，**提交时校验**（版本/CAS） | 先加锁，**再执行**（互斥排他） |
| **冲突时表现** | 更新失败 → **重试/退避/报错** | 其他请求/线程 **阻塞等待**（或超时失败） |
| **是否会阻塞** | 通常不阻塞（但会自旋重试） | 经常阻塞（等待锁释放） |
| **一致性控制** | 依赖校验 + 重试策略；高冲突下可能失败率高 | 通过互斥把并发写“串行化”，一致性更直观 |
| **性能特征** | 冲突低时吞吐高；冲突高时 CPU 可能被重试打满 | 冲突高时更稳定；但锁竞争会拉低吞吐、增加等待 |
| **常见实现** | CAS；版本号/时间戳字段 | `synchronized`/`Lock`；DB 行锁 `FOR UPDATE`；分布式锁（Redis/ZK） |
| **典型风险点** | 自旋开销；ABA；多变量一致性难；重试风暴 | 死锁；长事务/长持锁；锁超时；分布式锁复杂度高 |
| **更适合** | 读多写少、冲突低（配置更新、资料编辑等） | 热点写、强一致（扣库存/扣余额/订单状态流转等） |
| **一句话面试总结** | “不先锁，靠版本/CAS在提交时判断是否被改过” | “先锁住资源，别人来了先等，避免并发写冲突” |

## 面试案例（会讲就加分）

### 案例 1：乐观锁（版本号）——“商品库存扣减（非极端热点）”
- **适用前提**：冲突不是特别高（例如同一商品并发没那么夸张），允许少量重试。
- **做法**：
  - 表里加 `version` 字段
  - 先查：`SELECT stock, version FROM product WHERE id = ?;`
  - 再更：`UPDATE product SET stock = stock - 1, version = version + 1 WHERE id = ? AND version = ? AND stock > 0;`
  - 如果更新影响行数为 0：说明**版本冲突或库存不足** → 重新读取并重试（建议限次/退避）
- **面试一句话**：不先锁，提交时用 `version` 保证“我更新的是我看到的那份数据”。
- **常见坑**：
  - 不做限次/退避 → 冲突高时容易“重试风暴”
  - 忘了 `stock > 0` → 可能扣成负数

### 案例 2：悲观锁（数据库行锁）——“账号余额扣减（强一致，不能错）”
- **目标**：保证同一账号的余额不会被并发扣成负数/重复扣。
- **做法（典型事务流程）**：
  - 开启事务
  - `SELECT balance FROM account WHERE id = ? FOR UPDATE;`（锁住这一行）
  - 校验余额足够，再 `UPDATE account SET balance = balance - ? WHERE id = ?;`
  - 写流水（同一事务）
  - 提交事务释放锁
- **面试一句话**：对“钱/状态”这类强一致数据，我宁可让别人等，也不允许并发写穿。
- **常见坑**：
  - 事务太长（包含远程调用/大查询）→ 持锁时间长，导致大量阻塞
  - 锁顺序不一致 → 死锁（多行/多表时尤其常见）

### 案例 2.1：事务 + 分布式锁 + 行锁（跨进程串行，推荐套路）

> 适用：**多实例部署**下的“热点写”（扣库存/扣余额/处理同一任务），你既要跨机器互斥，又要数据库层面强一致。

#### 推荐流程（先分布式锁，再事务）
- **加分布式锁**（如 Redis `SET lock:biz:{id} token NX PX ...`）
- **开启事务**
- `SELECT ... FOR UPDATE`（把需要的行锁住）
- 校验业务条件并 `UPDATE`（必要时写流水/写状态）
- **提交事务**（提交后行锁释放）
- **释放分布式锁**（token 校验释放）

#### 关键点（你这段话的“核心意思”）
- **行锁**：只有在事务里才有意义，且一般在**事务提交/回滚后**才释放
- **分布式锁**：要覆盖“跨机器”的并发入口，通常在**事务开启前就要拿到**，在**事务提交后再释放**
- **事务要短**：不要把远程调用/慢查询放进事务，否则会“长时间持锁”导致阻塞

#### 隔离级别建议（并发插入/范围锁时尤其重要）
- 在 MySQL/InnoDB 下，`REPEATABLE READ` 可能因为**间隙锁/Next-Key Lock**在某些范围条件下放大锁冲突
- 经验上，如果你的场景里有较多“并发插入/范围条件更新”，可以优先考虑：
  - **`READ COMMITTED`** + **正确索引/唯一约束** + **状态条件更新（幂等）**

#### Java 伪代码（结构参考）
```java
public void buy(String productName, int buyQuantity) {
    Lock lock = lockService.getLock("lock:product:" + productName); // 分布式锁抽象
    lock.lock();
    try {
        // @Transactional(rollbackFor = Exception.class, isolation = Isolation.READ_COMMITTED)
        // 1) SELECT ... FOR UPDATE
        // 2) 校验库存 >= buyQuantity
        // 3) UPDATE 扣减库存 + 记录流水/状态（建议用状态条件更新做幂等）
        // 4) 提交事务（提交后行锁释放）
    } finally {
        lock.unlock(); // 释放分布式锁（实现上要 token 校验）
    }
}
```

#### `SELECT ... FOR UPDATE` + 索引 + 死锁（把你这段话说严谨）

> 结论先背：**优先用“主键/唯一索引等值条件”做 `FOR UPDATE`**，锁范围最小、并发最好；非主键也能用，但更要注意“锁范围”和“锁顺序”，否则死锁概率更高。

- **事务与释放时机**：
  - `FOR UPDATE` 必须在事务里才有意义
  - 行锁通常在**事务提交/回滚后**释放

- **索引命中（重点）**：
  - `WHERE id = ?`（主键等值）通常是最稳的写法：精确锁一行
  - `WHERE unique_key = ?`（唯一索引等值）也同样常用
  - 走**二级索引（非主键索引）**时，InnoDB 通常会：
    - 先锁住二级索引上的相关记录（用于定位）
    - 再回表锁住聚簇索引（主键）对应的那一行
    - 这属于正常机制，并不代表“必死锁”，但**锁链更长**，并发下更容易出现交叉等待

- **为什么并发下更容易死锁（常见原因）**：
  - 多行/多表更新时，两个事务**锁的顺序不一致**（例如 T1 先锁 A 再锁 B，T2 先锁 B 再锁 A）
  - `REPEATABLE READ` 下遇到范围条件/并发插入时，可能出现**间隙锁/next-key 锁/插入意向锁**，等待关系变复杂
  - 条件没走好索引导致锁范围变大（扫行、锁间隙）

- **实战规避建议（直接照做）**：
  - 能用主键/唯一索引**等值点查**就别用范围条件
  - 需要锁多行时，先把 id 排序，保证所有事务按**同一顺序**加锁
  - 控制事务时长（短事务），减少持锁时间
  - 对“重复请求/重复消息”用状态条件更新/唯一约束做幂等兜底（锁解决互斥，幂等解决只做一次）

### 案例 3：分布式场景（Redis 锁）——“同一订单只能被处理一次”
- **场景**：多实例部署的消费者并发处理同一订单（或定时任务多节点跑同一批数据）。
- **做法（思路级）**：
  - 尝试加锁：`SET lock:order:{id} <token> NX PX 30000`
  - 成功：执行业务；失败：说明有人在处理 → 直接跳过/稍后重试
  - 释放锁：只允许“持有者”释放（用 `<token>` 校验，避免误删别人的锁）
- **面试一句话**：单机锁管不了多台机器，用分布式锁把“同一资源”在集群里也变成互斥。
- **常见坑**：
  - 锁过期但业务没做完 → 可能出现并发执行（需要合理超时/续租策略）
  - 直接 `DEL key` 释放 → 可能误删别人的锁（必须校验 token）

### 案例 4：高频追问——“防重复下单 / 接口幂等（别只靠锁）”
- **为什么会问这个**：很多人看到“重复请求”就上锁，但面试官更想听你知道：**幂等优先**，锁只是手段之一。
- **目标**：用户点了两次“提交订单”，或者网络重试/超时重发，最终只能成功一次。
- **常见做法（任选其一或组合）**：
  - **幂等 token**：
    - 下单前发一个 `token` 给前端（一次性）
    - 提交时带上 token，服务端用 Redis/DB 校验“token 是否已使用”
    - 成功创建订单后把 token 标记为已用（或删除），重复请求直接返回“已处理”
  - **数据库唯一约束（最硬核）**：
    - 设计一个“幂等键”字段，例如 `request_id` / `biz_id`（前端或网关生成）
    - 在订单表或去重表上加 **UNIQUE** 约束
    - 重复请求第二次插入会触发唯一冲突 → 直接当作“已成功/已处理”
  - **去重表（dedup table）**：
    - 先插入去重表（`biz_id` 唯一），插入成功才执行业务；失败说明重复
- **面试一句话**：防重复提交优先靠“幂等键 + 唯一约束/去重记录”，锁更多是用来保护热点资源的并发写。
- **常见坑**：
  - 只靠分布式锁：锁超时/异常释放时仍可能重复执行
  - 没有“幂等键”：无法判断两次请求是不是同一笔业务重试
