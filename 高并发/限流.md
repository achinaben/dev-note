## 限流（Rate Limiting）笔记

### 你要先记住的一句话
- **限流**：控制“单位时间内允许通过的请求量/速率”（QPS、每秒请求数）。
- **并发控制**：控制“同一时刻正在执行的请求数”（比如最多同时 5 个线程占用资源）。

---

## 常见落地位置（从外到内）

### 1）Nginx：`ngx_http_limit_req_module`（常用）
- **作用**：限制访问频率（按 IP、按指定 key）。
- **原理本质**：**漏桶算法**思想（核心是“平滑输出”，让请求以稳定速率被处理/放行）。
- **特点**
  - **优点**：离用户最近，能在入口就挡住恶意/突发流量，保护后端。
  - **缺点**：更偏网关层策略，难做复杂的“按用户/业务维度”的精细规则（但可以通过 key 设计增强）。

---

### 2）Gateway：通过 Filter（网关层）
- **针对 URL**：按接口路径、路由维度做限流（例如 `/api/order/create`）。
- **典型做法**：在网关 Filter 中生成限流 key（例如：`ip + path`、`userId + path`），然后在网关处拦截或降级。

---

### 3）业务服务：通过注解（Controller/方法级）
- **针对 Controller/方法**：更贴近业务，适合“某个接口单独限流”“某类用户限流”。
- **典型做法**：自定义注解 + AOP 拦截，在进入方法前做限流判断，失败直接返回“请求过于频繁”。

---

## Guava `RateLimiter`（令牌桶模型）

### 模型一句话
- **令牌桶（Token Bucket）**：系统按固定速率往桶里放令牌；请求来了先拿令牌，**拿到就放行**；桶满则多余令牌丢弃。
- **好处**：既能限制平均速率，又允许**短时间突发**（桶里攒的令牌可以被“突发消费”）。

### `create`
```java
public static RateLimiter create(double permitsPerSecond)
```
- **含义**：按指定稳定吞吐率创建 `RateLimiter`，通常把 `permitsPerSecond` 当作 **QPS**。
- **保证**：平均情况下，每秒发放许可数不超过 `permitsPerSecond`，持续请求会被“平滑”分摊到每一秒中。
- **超速时**：当请求速率超过 `permitsPerSecond`，限流器大约每隔 \(1.0 / permitsPerSecond\) 秒释放 1 个许可。
- **空闲时允许突发**：当限流器空闲，会允许最多累计到 `permitsPerSecond` 的突发（之后再按稳定速率平滑限制）。
- **参数**
  - `permitsPerSecond`：每秒产生多少个许可（必须 > 0）。
- **异常**
  - `IllegalArgumentException`：`permitsPerSecond <= 0`。

### `acquire`
```java
public double acquire(int permits)
```
- **含义**：获取指定数量许可；**会阻塞**直到拿到足够许可。
- **返回**：为了遵守限速，需要等待（sleep）的时间，单位秒；如果无需等待则返回 0。
- **异常**
  - `IllegalArgumentException`：`permits <= 0`。
- **版本提示**
  - `Since: 16.0`（早期版本可能没有返回值）。

### `tryAcquire`
```java
public boolean tryAcquire(int permits, long timeout, TimeUnit unit)
```
- **含义**：在不超过 `timeout` 的时间内尝试获取许可；拿不到就直接返回 `false`（不无限等待）。
- **参数**
  - `permits`：需要的许可数（必须 > 0）。
  - `timeout`：最大等待时间（负数按 0 处理）。
  - `unit`：时间单位。
- **返回**
  - `true`：获取到许可
  - `false`：超时前拿不到许可
- **异常**
  - `IllegalArgumentException`：`permits <= 0`。

## Redis 限流（常见方案）

### 方案 A：计数器 + 过期时间（固定窗口/滑动窗口的简化版）
- **思路**：对某个 key（例如某接口、某用户）在单位时间内计数：
  - 如果 key **未过期**：直接 `+1`
  - 如果 key **已过期/不存在**：`+1` 并设置过期时间（重新开始计时）
  - 取出计数值判断是否达到阈值，达到则拒绝（抛异常/返回 429）
- **缺点（非常关键）**
  - **不够平滑**：窗口边界会出现“瞬时冲击”，可能导致短时间流量把下游打穿（你原文的“击穿缓存/冲击”就是这个意思）。

### 方案 B：令牌桶（主动控制）
- **特点**
  - **可控**：桶容量、放令牌速度可控
  - **满了丢弃**：桶满则新令牌丢弃
  - **不回收令牌**：令牌被请求消费后就没了（靠后续补充）
- **适用**
  - **控制有限资源的派发**（比如给请求发“执行资格”）

### 方案 C：漏桶（被动控制）
- **特点**
  - **核心是控“流出速率”**：输出（处理/放行）速率固定、稳定
  - 常用于“把突发流量抹平”
- **适用**
  - **控制流量**（更强调平滑）

---

## 限流 key 的维度设计（决定你能按谁限）

### 常见维度（可组合）
- **userId**：按用户限流（最常见）
- **ip**：按 IP 限流（防刷）
- **接口/URL**：按接口限流（保护热点接口）
- **参数 hash**：把 `Request IP + 参数 + 接口` 等做 hash 作为 key（你在“幂等性请求”里提到的这种思路，也能用于更细粒度的限流）

> 经验：实际工程里通常是 **`(用户 or IP) + 接口`** 的组合最常用。

## 信号量 Semaphore（并发控制，不等于限流）

### 一句话理解
- **Semaphore（信号量）**：维护一组可回收的“许可证”，用来控制**同时访问资源的线程数量**（并发数），不是按“每秒多少次”。

### 典型场景
- **数据库连接池/外部服务连接**：比如最多允许 5 个请求同时占用连接，其余等待。

### 示例（限制同时最多 5 个线程做耗时任务）
```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;
import java.util.concurrent.TimeUnit;

public class SemaphoreDemo {
    public static void main(String[] args) {
        final ExecutorService service = Executors.newFixedThreadPool(100);
        Semaphore s = new Semaphore(5);

        for (int i = 0; i < 100; i++) {
            service.submit(() -> {
                try {
                    s.acquire();
                    System.out.println("数据库耗时操作 " + Thread.currentThread().getName());
                    TimeUnit.MILLISECONDS.sleep(3000);
                    System.out.println(Thread.currentThread().getName() + " 正在执行....");
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    s.release();
                }
            });
        }
    }
}
```

### 常用方法速记
- `acquire()`：获取许可证，拿不到就阻塞等待
- `release()`：释放许可证归还
- `tryAcquire()`：尝试获取，获取不到立刻返回 `false`
- `tryAcquire(timeout, unit)`：最多等一段时间
- `availablePermits()`：当前剩余许可证数
- `getQueueLength()`：等待许可证的线程数（估计值）

## 漏桶 vs 令牌桶（对比表：背下来就够用）

| 维度 | 漏桶（Leaky Bucket） | 令牌桶（Token Bucket） |
|---|---|---|
| **控制目标** | **控制输出速率**（流出恒定） | **控制平均速率**（按放令牌速率） |
| **对突发流量** | **不友好**：突发会排队/溢出（更“平滑”） | **友好**：可用积攒令牌吸收突发 |
| **是否平滑** | **更平滑**（常用于削峰填谷） | **较平滑**（但允许短时突发） |
| **实现直觉** | “水以固定速度漏出去” | “固定速度发门票，拿到票才能进” |
| **常见使用场景** | 网关/Nginx 出入口限流、保护下游稳定 | 业务侧限流、API 配额、需要允许突发的场景 |

### 一句话记忆法
- **漏桶**：强调“**匀速出**”，突发会被抹平。
- **令牌桶**：强调“**凭票进**”，空闲可攒票，所以能突发。

## MQ 背压（消费者扛不住时，别把系统打爆）

> **一句话**：当下游处理不过来时，要让上游“少推/少拉/慢拉”，把压力控制在可承受范围内（背压 = 反向施压）。

### 1）通用套路（任何 MQ 都适用）

- **控并发**：消费者线程池固定大小（例如 10/20/50），不要无限开线程
- **控在途**：限制“未完成处理、还没确认成功”的消息数量
- **成功后再确认**：处理成功再 ACK/提交 offset（否则背压失效且更容易丢数据）
- **失败要退避**：失败重试要有退避（间隔越来越长）与上限；超过上限进 DLQ/人工兜底
- **先幂等再重试**：没有幂等就重试，会把业务越压越乱（重复扣款/重复加库存）

### 2）RabbitMQ 怎么做背压（prefetch + 手动 ACK）

- **核心开关：prefetch（在途上限）**
  - 含义：每个消费者连接最多同时拿 \(N\) 条“未 ACK”的消息
  - 建议起步：`prefetch ≈ 消费并发 × (1~3)`（例如并发 20 → prefetch 20~60）
- **手动 ACK**
  - 处理成功再 `ack`
  - 处理失败：按你的策略 `nack/requeue`（重试）或进入 DLQ
- **为什么能背压**
  - 你处理慢 ⇒ ACK 变慢 ⇒ 未 ACK 达到 prefetch 上限 ⇒ Broker 暂停继续投递 ⇒ 自动背压成立

### 3）Kafka 怎么做背压（少 poll + pause/resume）

- **核心：控制拉取量 + 必要时暂停拉取**
  - `max.poll.records`：控制每次 poll 拉取条数（先从 100/200/500 这种量级起步）
  - 本地有界队列：poll 到的数据先放入有界队列；队列快满时触发背压
  - `pause(partitions)`：处理不过来就暂停分区拉取；处理恢复后再 `resume(partitions)`
- **提交 offset 的底线**
  - 处理成功后再提交（避免“没处理完就提交导致丢处理”）
- **注意超时配置（否则会抖动）**
  - 单批处理可能很慢时要关注 `max.poll.interval.ms`（处理太久可能触发 rebalance，导致重复消费与抖动）

### 4）RocketMQ 怎么做背压（线程数 + 批量 + 重试退避）

- **控并发**：限制消费线程数（不要无限开）
- **控批量**：限制一次拉取/一次消费处理的批量（避免单次塞太多把内存/DB 打爆）
- **失败退避/重试**：消费失败让消息进入延迟重试（逐步拉长间隔），并设置上限与 DLQ/人工兜底

### 5）让生产端“感知消费压力”并自动降速（闭环）

> **要点**：MQ 的消费者变慢后，生产端想自动降速，通常靠两种信号：**堆积指标（间接反馈）** 或 **配额/控制消息（直接反馈）**。

#### 5.1 最推荐（通用）：生产端根据“堆积/延迟”动态限流

- **做法**：生产端发送前经过一个**动态令牌桶/漏桶**；限流器的速率由“堆积指标”实时调整（超阈值就降速，恢复后再升速）。
- **为什么可行**：堆积（lag/ready/unacked）是消费能力不足的客观结果，用它做闭环最稳，且不依赖消费者改协议。
- **防抖（必做）**：用“**分档 + 滞回 + 稳定窗口**”避免频繁升降速。
  - 示例：堆积 > 10w → 降到 50%；> 30w → 降到 20%；< 6w 且持续 5 分钟 → 才恢复到 50%/100%

#### 5.2 RabbitMQ：看 queue depth / unacked

- **信号**
  - `messages_ready`（待消费数）
  - `messages_unacknowledged`（未 ACK：消费者忙的直观信号）
- **生产端动作**
  - `unacked/ready` 持续上升 ⇒ 降低发送 QPS、缩小批量、只发送关键消息（非关键延迟/丢弃）

#### 5.3 Kafka：看 Consumer Group Lag（最标准）

- **信号**
  - `consumer group lag`（按 topic/partition/group）
  - 端到端延迟（生产时间戳 vs 消费完成时间戳）
- **生产端动作**
  - lag 持续上升 ⇒ 动态降速（必要时降级：合并消息、采样、只发关键）

#### 5.4 RocketMQ：看堆积 + 重试/死信态势

- **信号**
  - 消息堆积量（topic/group 维度）
  - 重试队列堆积、DLQ 增速（说明下游长期扛不住或存在“毒消息”）
- **生产端动作**
  - 堆积或重试显著上升 ⇒ 降速 + 降级（延迟发送、只发关键）

#### 5.5 更强反馈（可选）：消费者发“BUSY/配额”控制生产速率

- **控制通道**：消费者（或调度器）定期发布控制消息/写入共享存储（Redis/配置中心）
  - 例：`state=BUSY` 或 `quota=5000 msg/s` 或 `credits=5000/s`
- **生产端行为**：按配额调整令牌桶速率；控制信号要**短 TTL + 周期刷新**（避免丢失导致一直限速/一直放开）

## Redis + Lua 原子限流（模板）

> 目的：把“读-判断-写”合并成一次 Lua 执行，避免并发下不一致（例如多个请求同时通过判断）。

### 模板 1：计数器 + 过期时间（固定窗口）

**含义**：在 `windowMs` 时间窗口内最多允许 `limit` 次请求；超过返回 0，否则返回剩余可用次数（或返回 1 表示通过）。

```lua
-- KEYS[1] = 限流 key（例如 "rl:{userId}:{path}"）
-- ARGV[1] = windowMs（毫秒，例如 1000）
-- ARGV[2] = limit（阈值，例如 100）

local key = KEYS[1]
local windowMs = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
  redis.call('PEXPIRE', key, windowMs)
end

if current > limit then
  return 0
end

return limit - current
```

### 模板 2：滑动窗口（用 ZSET，理解用，生产要注意成本）

**含义**：把每次请求的时间戳写进 ZSET，保留窗口内的记录，`ZCARD` 超过阈值则拒绝。

```lua
-- KEYS[1] = 限流 key（例如 "rl:{userId}:{path}"）
-- ARGV[1] = nowMs（当前时间戳毫秒）
-- ARGV[2] = windowMs（窗口毫秒）
-- ARGV[3] = limit（阈值）

local key = KEYS[1]
local nowMs = tonumber(ARGV[1])
local windowMs = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

-- 1) 清理窗口外数据
redis.call('ZREMRANGEBYSCORE', key, 0, nowMs - windowMs)

-- 2) 统计窗口内请求数
local count = redis.call('ZCARD', key)
if count >= limit then
  return 0
end

-- 3) 记录本次请求（member 用 nowMs+随机数避免冲突）
redis.call('ZADD', key, nowMs, tostring(nowMs) .. '-' .. tostring(math.random(100000, 999999)))

-- 4) 设置过期时间（避免 key 永久存在）
redis.call('PEXPIRE', key, windowMs)

return limit - count - 1
```

### key 怎么设计（直接套用）
- **按用户 + 接口**：`rl:{userId}:{path}`
- **按 IP + 接口**：`rl:{ip}:{path}`
- **按参数哈希 + 接口**：`rl:{hash}:{path}`（更细粒度，但 key 数会更多）
