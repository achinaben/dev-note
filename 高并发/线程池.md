## 线程池（ThreadPool）笔记：预热 & 线程数怎么估

### 池的预热（Warm-up）
线程池的 **corePoolSize（核心线程数）** 设计初衷是“常驻线程”，用来承载日常稳定流量，所以它应该尽快初始化。

- **ThreadPoolExecutor 的常见行为**：当线程数还没达到 `corePoolSize` 时，来了新任务通常会 **优先创建新线程** 来执行（而不是先排队），从而把核心线程尽快“拉起来”。
- **为什么要预热**：很多高并发服务做预热，并不是主要指望 JVM 对热点代码做 JIT 优化；更关键的往往是：
  - **线程池预热**：提前创建核心线程，避免第一波流量到来时“边建线程边处理”造成抖动
  - **连接池预热**：比如 DB 连接池提前建立连接
  - **本地缓存预热**：把热点数据提前加载到内存缓存

#### 怎么做线程池预热（常见手段）
- **方式 A：启动时提交一些空任务/轻任务**，让线程池尽快创建到 `corePoolSize`
- **方式 B：配置并调用 `prestartAllCoreThreads()`**（如果你们工程里有直接拿到 `ThreadPoolExecutor`）

> 预热的目标是：**降低冷启动抖动、减少首波请求延迟、让资源提前就绪**。

---

### 单核 CPU 与“同时执行”这句话怎么理解
**单核 CPU 在同一时刻只能真正执行一个线程**（从“正在用 CPU 跑指令”的角度）。

但你会看到“好像多个线程同时在跑”，原因是：
- 操作系统在很短时间片内 **频繁切换** 线程（上下文切换），让人感觉并行
- 大量线程会因为 **IO 等待**（网络、磁盘、DB）而阻塞，这时 CPU 可以去执行别的线程

---

### 多线程的开销（为什么线程不是越多越好）
- **上下文切换开销**：线程越多，切换越频繁，CPU 时间花在“切来切去”上就越多
- **锁竞争开销**：并发访问共享资源需要锁（或 CAS/同步手段），竞争会造成等待、阻塞，吞吐下降

结论：线程数要和业务类型匹配（CPU 密集 vs IO 密集），并结合外部资源上限（比如 DB 连接池）。

---

### Java 获取 CPU 核数

```java
System.out.println(Runtime.getRuntime().availableProcessors()); // 输出本机CPU的数量
```

### 线程数不要超过数据库连接池连接数（常见约束）
如果你的任务“每个请求都会占用一个数据库连接”，那么线程数一旦远大于 DB 连接池最大连接数：
- 多出来的线程会卡在等连接上（等待时间变长）
- 反而可能造成 **线程堆积、队列堆积、响应变慢**

所以常见经验是：
- **线程池并发上限 ≈ DB 连接池上限（或略小）**，并结合实际 SQL 耗时/事务占用时间调优

---

### 最大线程数（maximumPoolSize）：什么时候才会用到
一个容易记住的规律：
- **没到 corePoolSize**：来任务 → 倾向于 **新建线程**（尽快把核心线程拉满）
- **到 corePoolSize 之后**：来任务 → 倾向于 **先入队**
- **队列满了之后**：才会考虑 **继续新建线程**，一直涨到 `maximumPoolSize`
- **队列也满 + 线程也到 maximumPoolSize**：触发 **拒绝策略**（RejectedExecutionHandler）

所以你写的这句话可以记成：
- **队列溢出后才会进入最大线程数的扩容逻辑**

> 实战里还要注意：如果你用的是无界队列（例如 `new LinkedBlockingQueue()` 不传容量），队列通常“永远不满”，那么 `maximumPoolSize` 基本就形同虚设，风险是任务无限堆积导致 OOM（这也是前面强调“用有界队列”的原因）。

---

### 常见配置：corePoolSize = maximumPoolSize（固定大小线程池）
很多线上业务会把 **最大线程数 = 核心线程数**，并让线程几乎不回收：
- **maximumPoolSize = corePoolSize**
- **keepAliveTime 设为 0 或很大**（通常配合 TimeUnit.DAYS 这种“几乎不回收”的效果）
- **allowCoreThreadTimeOut = false（默认）**：核心线程空闲也不回收

这样做的“核心目的”：
- 让线程池 **启动后尽快达到固定线程数**
- 避免业务有波峰/波谷时频繁 **创建/回收线程** 带来的抖动和开销

一个更推荐的示例写法（强调“有界队列”）：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 10,
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>(1000) // 有界，避免任务无限堆积
);
```

你描述的现象（在 core=max 的前提下）可以整理成：
- 请求数 < coreSize：新增线程
- 请求数 ≥ coreSize 且队列不满：任务入队
- 队列满：因为 coreSize = maxSize，无法再加线程 → **直接拒绝**

这种写法成立的前提/代价：
- **前提**：机器资源足够、这个线程池的并发上限你能“承受得住”（CPU、内存、下游）
- **代价**：如果线程数/队列配置不合理，可能更容易在高峰期触发拒绝；因此一定要配合压测和监控（队列长度、拒绝次数、TP99、CPU、GC、下游耗时）

---

### allowCoreThreadTimeOut：核心线程要不要“按空闲超时回收”
`allowCoreThreadTimeOut` 决定了 **keepAliveTime 的回收规则是否也作用于核心线程**。

- **FALSE（默认）**
  - **keepAliveTime 只对“非核心线程”生效**：也就是线程数 \(>\) `corePoolSize` 的那部分
  - **核心线程会一直保活**：没有任务时通常阻塞等待（不占用 CPU）
  - **适用场景**：大多数常驻业务线程池（线程复用的意义就在于减少创建/销毁开销）

- **TRUE**
  - **keepAliveTime 对“所有线程”生效**：包括核心线程
  - **适用场景**：使用频率低、长期空闲的线程池
  - **目的**：避免空闲的核心线程长期占用内存等资源（尤其是线程栈）

一般情况下我们不会回收核心线程：线程池就是做线程复用的，而且核心线程空闲时处于阻塞状态，不占 CPU；真正需要关注的是它占用的 **线程栈内存**、以及线程数量过多带来的调度开销。

---

### ThreadPoolExecutor 怎么回收工作线程：getTask() 返回 null 就退出
你可以把它记成一句话：
- **工作线程会循环取任务（内部类似 `getTask()`），当取不到任务且符合回收条件时，`getTask()` 返回 `null`，线程就会退出并被回收。**

常见两类场景（按线程池状态区分）：

#### 场景 1：未调用 shutdown()（RUNNING 状态），任务都执行完/队列空了
- 当线程数 \(>\) `corePoolSize` 时，多出来的线程会在 `getTask()` 里 **按 keepAliveTime 超时等待**
- 超时醒来后，会尝试 **CAS 减少工作线程数**
  - CAS 成功 → `getTask()` 返回 `null` → 线程退出回收
  - CAS 失败 → 进入下一次循环重试
- 当工作线程数 \(\le\) `corePoolSize`：
  - 如果 `allowCoreThreadTimeOut=false`（默认），核心线程可以一直阻塞等待，不会因为超时被回收

#### 场景 2：调用 shutdown() 后，所有任务执行完成
`shutdown()` 会让线程池进入“要收尾”的状态，并 **对空闲的工作线程发出中断信号**（帮助它们从“阻塞等任务”里醒来，去检查退出条件）。

两种常见情况：
- **所有线程都在阻塞等待任务**
  - 被中断唤醒 → 进入循环 → 满足退出条件 → `getTask()` 返回 `null` → 线程逐步回收
- **还有任务没执行完**
  - 至少会有线程在收尾过程中退出
  - 在线程退出的收尾逻辑里会触发终止检查，并对空闲线程发出中断信号
  - 最终阻塞的线程会被一个个唤醒并完成回收

> 你不用死记内部方法名；记住“**阻塞等待 + 超时/中断唤醒 + 满足条件返回 null + 线程退出**”这条主线就够了。

## 线程池按业务分类（按业务语义来拆）

> 目的：**别用一个“大杂烩线程池”跑所有任务**。不同任务的耗时、阻塞点、优先级、上下文要求不同，混在一起最容易互相拖垮（排队、饥饿、延迟飙升）。

### 1）定时线程池（Scheduled）
- **适合**：定时任务/周期任务（心跳、对账、定时刷新缓存）。
- **典型实现**：`ScheduledExecutorService`（常用 `Executors.newScheduledThreadPool(n)`）。
- **关键点**：
  - 定时任务要有**超时/退出**，避免一次任务卡死导致后续堆积。
  - 周期任务尽量做到**幂等**（失败可重试、重复执行不出错）。

### 2）业务线程池（Biz）
- **适合**：纯业务计算/CPU 占比高的同步处理（规则计算、加密验签、数据转换）。
- **建议**：**有界队列 + 明确拒绝策略**（常见 `CallerRunsPolicy` 做背压，或快速失败+降级）。
- **关键点**：
  - 线程数更贴近 CPU：常用起步值 \(P+1\)（见你后面的估算公式）。
  - **不要**把 IO 密集任务塞进这个池，否则把 CPU 池拖成“等 IO 的池”。

### 3）IO 线程池（IO）
- **适合**：大量阻塞 IO 的任务（RPC/HTTP、DB、磁盘等）。
- **建议**：
  - 线程数通常会比 CPU 核数大（因为大量时间在等 IO）。
  - 但上限要受**下游资源**约束：尤其 DB 连接池、下游限流、网络连接数。
- **关键点**：
  - 仍然要**有界队列**，避免 IO 抖动时任务无限堆积导致 OOM。
  - 能用异步 IO/批处理/缓存就尽量减少“线程阻塞等 IO”的时间。

### 4）上下文线程池（Context）：跨线程传递上下文（TransmittableThreadLocal）
- **适合**：你需要在子线程里拿到父线程上下文的场景，比如：
  - **TraceId / 日志 MDC**
  - **用户信息/租户信息**
  - **灰度标记/请求链路标签**
- **典型问题**：用普通 `ThreadLocal` 时，线程池会复用线程，导致：
  - 子线程拿不到父线程值（或拿到“上一次请求残留”的脏值）
- **常见方案**：使用 **`TransmittableThreadLocal (TTL)`**，并对线程池做包装，让提交任务时把上下文“拷贝过去”。
- **提醒**：上下文传递是“能力”，不是“免费午餐”，要注意：
  - 传递的数据别太大（否则复制成本高）
  - 任务结束要确保清理（避免上下文泄漏/串号）

### 5）异步线程池（Async）：`@Async`
- **适合**：把“主流程非关键路径”异步化（发消息、写审计日志、异步通知、统计上报）。
- **典型实现**：Spring 的 `@Async`（注意不是 `@Asyc`）。
- **关键点**：
  - 需要**显式指定线程池**（不建议用默认配置“黑盒线程池”），并配合监控与拒绝策略。
  - 如果异步任务也需要 TraceId/MDC 等上下文，通常要和上面的“上下文线程池/TTL”配套，否则日志链路容易断。

## 线程池定义：Executors vs ThreadPoolExecutor

### 先记一句话
- **`Executors`**：线程池的“工厂/快捷创建工具类”（帮你拼好参数）。
- **`ThreadPoolExecutor`**：真正干活的“核心实现”（你能完全掌控线程数、队列、拒绝策略、线程工厂等）。

### 为什么线上更推荐直接 new ThreadPoolExecutor
- `Executors` 的一些快捷方法默认使用 **无界队列** 或 **无限扩容**，在高并发下可能导致：
  - **任务无限堆积 → OOM**
  - **线程无限增长 → 资源被打爆**
- 你在这篇笔记里反复强调“有界队列、拒绝策略、背压”，这些都更适合用 `ThreadPoolExecutor` 明确表达出来。

### 实战推荐的“最小必选项”（创建线程池时你至少要想清楚）
- **线程数**：`corePoolSize / maximumPoolSize`
- **队列**：优先有界（例如 `LinkedBlockingQueue<>(capacity)`）
- **拒绝策略**：根据“能否丢、能否慢、能否落地”选择（你后面已经写得很完整）
- **线程工厂**：线程命名（便于排查）、是否 daemon、异常处理策略

## 线程池关闭：shutdown() vs shutdownNow()（以及如何等待真正关闭）

### 线程池的 5 种状态（RUNNING/SHUTDOWN/STOP/TIDYING/TERMINATED）
你可以用一句话概括它们的区别：**越往后越“拒绝新任务”且越接近完全退出**。

- **RUNNING（运行）**
  - **能接收新任务**（`execute/submit`）
  - **也会处理队列里的任务**

- **SHUTDOWN（优雅关闭中）**：调用 `shutdown()` 进入
  - **不再接收新任务**
  - **会继续执行队列里已提交的任务**，以及正在执行的任务

- **STOP（强制关闭中）**：调用 `shutdownNow()` 进入
  - **不再接收新任务**
  - **会尝试中断正在执行的线程**（`interrupt`）
  - **会清空队列并返回“未开始的任务列表”**（`shutdownNow()` 的返回值）
  - 注意：这是“尽力而为”。任务是否立刻停止，取决于任务是否**响应中断**。

- **TIDYING（收尾中）**
  - 条件：**任务都结束了** 且 **工作线程数为 0**
  - 进入后会执行 `terminated()` 钩子（收尾动作）

- **TERMINATED（已终止）**
  - `terminated()` 执行完之后的最终状态

#### 状态迁移（背一条线就够用）
- **`shutdown()` 路线**：RUNNING → SHUTDOWN → TIDYING → TERMINATED
- **`shutdownNow()` 路线**：RUNNING/SHUTDOWN → STOP → TIDYING → TERMINATED

> 补充：如果你一直不调用 `shutdown()/shutdownNow()`，线程池状态通常会一直是 RUNNING；但线程数量可能因为空闲超时（`keepAliveTime`、`allowCoreThreadTimeOut`）而变化。

### 先给你一句话结论（背下来就够用）
- **`shutdown()`**：**拒绝新任务**，但会让**已提交的任务（队列里 + 正在执行）继续执行完**，然后线程池退出（优雅关闭）。
- **`shutdownNow()`**：**拒绝新任务**，并**尝试中断正在执行的线程**，同时把**队列里尚未开始的任务移出并返回**（强制关闭，尽力而为）。

### shutdownNow() 的几个你必须知道的点
- **它不保证“立刻停”**：只是对工作线程发出 `interrupt`。任务是否停止取决于：
  - 任务是否在可中断阻塞点（如 `sleep/wait/join`、部分 `BlockingQueue` 操作等）
  - 任务是否正确响应中断（捕获 `InterruptedException` 后退出、或定期检查 `Thread.currentThread().isInterrupted()`）
- **它会返回“没来得及执行的任务”**：`List<Runnable> dropped = shutdownNow()` 里的任务默认不会再被执行（除非你自己拿出来重跑）。

### shutdown() 为什么可能“关不掉”
`shutdown()` 的目标是“收尾执行完”，所以如果任务里存在**永久阻塞/死循环**，线程池就可能一直无法终止。
- **结论**：任务代码必须有**超时/退出条件/可取消**，否则关闭会卡住。

### awaitTermination：等“真正关闭完成”
调用 `shutdown()` / `shutdownNow()` **不代表已经关闭完成**，它只是“通知开始关闭”（异步）。
如果你要同步等待线程池彻底关闭，再继续往下执行，用：

```java
executor.shutdown();
try {
    // 先优雅等一段时间
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        // 超时则强制关，并拿到队列里未开始的任务
        List<Runnable> dropped = executor.shutdownNow();
        // 再等一段时间
        if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
            // 仍未结束：说明任务不响应中断/卡死，需要告警与排查
        }
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

## 判断“线程池的这一批任务是否都执行完”：4 种常见方法

> 先说最稳妥的选择：**CountDownLatch/Phaser** 这种“由任务自己上报完成”的方式，最准。

### 方法 1：`isTerminated()`（需要先 shutdown；不推荐空转）
思路：先 `shutdown()`，再判断是否已终止。

不推荐写法（空转吃 CPU）：

```java
executor.shutdown();
while (!executor.isTerminated()) {
}
```

推荐写法（用 awaitTermination，或至少带超时）：

```java
executor.shutdown();
executor.awaitTermination(30, TimeUnit.SECONDS);
```

### 方法 2：`getTaskCount()` vs `getCompletedTaskCount()`（近似值，有坑）
思路：如果“计划任务数 == 已完成任务数”，就认为执行完。

```java
while (executor.getTaskCount() != executor.getCompletedTaskCount()) {
    // 注意：可能需要 sleep，避免空转
}
```

注意点（一定要记住）：
- `getTaskCount()` / `getCompletedTaskCount()` 都是**近似值**（并发变化时不精确）
- 如果还有线程在持续提交新任务，这个条件可能永远等不到

### 方法 3：`CountDownLatch`（推荐：语义清晰、最准）
适合：你知道“这一批任务有多少个”，等它们全部完成后再继续。

```java
int taskCount = 5;
CountDownLatch latch = new CountDownLatch(taskCount);

for (int i = 0; i < taskCount; i++) {
    executor.execute(() -> {
        try {
            // do work
        } finally {
            // 关键：放 finally，避免异常导致永远等不到
            latch.countDown();
        }
    });
}

latch.await();
```

### 方法 4：`CyclicBarrier`（更像“会合点”，可重复使用）
它适合“大家都跑到某个点再一起做下一步”，而不是最常见的“等待所有任务结束”。
如果你要“分阶段、可重复的批次等待”，建议你额外认识一个更灵活的工具：**`Phaser`**。

## 提交任务：execute vs submit（以及异常为什么“看不见”）

### 先说你最关心的结论
- **`execute()`**：没有返回值；任务抛出的异常如果没有在任务内部捕获，通常会走线程的 **`UncaughtExceptionHandler`**（所以控制台/日志更容易看到堆栈）。
- **`submit()`**：会返回 `Future`；任务异常会被封装进 `Future`，**不会作为“未捕获异常”直接打印**。你必须 **`future.get()`** 才能拿到异常（否则可能“悄悄丢失”）。

### 参数与返回值对比（背这一段就够用）
- **参数**：
  - `execute(Runnable)`
  - `submit(Runnable)` / `submit(Runnable, T result)` / `submit(Callable<T>)`
- **返回值**：
  - `execute`：无返回值（不代表不能失败，只是你拿不到 `Future`）
  - `submit`：返回 `Future`（可拿结果、可取消、可 `get()` 拿异常）
- **接口归属**：
  - `execute`：定义在 `Executor`
  - `submit`：定义在 `ExecutorService`（继承了 `Executor`）

### 最常见的坑：submit 不 get，就等于“异常没人管”
你可以把 `submit()` 理解成：线程池把任务包装成 `FutureTask`，异常会被“装进 Future 盒子里”。  
你不打开盒子（`get()`），外面就看不到异常堆栈。

### 例子：lambda 提交任务

```java
threadPool.execute(() -> {
    System.out.println("任务被执行,线程:" + Thread.currentThread().getName());
});
```

### 例子：批量 submit + get（你的代码思路整理版）
`get()` 会阻塞等待结果；同时也是把任务异常“捞出来”的关键动作。

```java
private static List<String> findPriceFutureAsync(String product) {
    ExecutorService es = Executors.newCachedThreadPool();
    List<Future<String>> futureList = shopList.stream()
            .map(shop -> es.submit(() -> String.format("%s price is %.2f",
                    shop.getName(), shop.getPrice(product))))
            .collect(Collectors.toList());

    return futureList.stream()
            .map(f -> {
                try {
                    return f.get(); // 不 get：异常可能就“丢”了
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException(e);
                } catch (ExecutionException e) {
                    throw new RuntimeException(e.getCause());
                }
            })
            .collect(Collectors.toList());
}
```

### 拒绝策略（RejectedExecutionHandler）：什么时候触发、有哪些选择
你这句总结非常关键：
- **队列溢出后进入最大线程数，最大线程数也满了后，才会执行拒绝策略**

更完整的触发链路可以记成：
- 请求进来
  - 线程数 < `corePoolSize` → 建线程执行
  - 否则 → 入队
  - 队列满 → 尝试建线程（直到 `maximumPoolSize`）
  - 线程也到 `maximumPoolSize` 且队列仍满 → **触发拒绝策略**

#### 默认拒绝策略
`ThreadPoolExecutor` 默认使用 **`AbortPolicy`**（直接抛异常）：

```java
private static final RejectedExecutionHandler defaultHandler =
        new AbortPolicy();
```

并且线程池会通过 `reject()` 来调用策略：

```java
final void reject(Runnable command) {
    handler.rejectedExecution(command, this);
}
```

#### 4 种内置拒绝策略（怎么选）
- **`AbortPolicy`（默认）**：直接抛 `RejectedExecutionException`
  - **适合**：你希望“宁可失败也别悄悄丢请求”，并且上层能捕获异常做降级/告警
- **`DiscardPolicy`**：直接丢弃任务（啥也不做）
  - **适合**：允许丢弃、且丢弃不会产生严重后果的场景（例如极低价值的统计上报）
  - **风险**：最容易“无声失败”，排查困难
- **`CallerRunsPolicy`**：线程池满时，让**提交任务的线程**自己执行 `r.run()`
  - **效果**：用“让调用方变慢”来反向限流（背压）
  - **适合**：希望系统在高峰期自动降速、并尽量不丢任务的场景
  - **注意**：如果提交线程是业务关键线程（例如 Web 请求线程），可能导致请求线程被拖慢、RT 变大
- **`DiscardOldestPolicy`**：丢弃队列里**最旧**的一个任务，然后把当前任务重新入队
  - **适合**：更希望处理“新请求”，旧请求价值更低的场景（仍要谨慎）
  - **风险**：被丢的是“最早排队的任务”，可能造成某些任务长期饥饿

下面是它们的核心行为（你给的源码思路整理版）：

```java
// DiscardPolicy：丢弃任务
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) { }

// AbortPolicy：抛异常
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    throw new RejectedExecutionException(...);
}

// CallerRunsPolicy：调用线程执行（线程池未关闭时）
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    if (!e.isShutdown()) r.run();
}

// DiscardOldestPolicy：丢弃最旧任务，再提交当前任务（线程池未关闭时）
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    if (!e.isShutdown()) {
        e.getQueue().poll();
        e.execute(r);
    }
}
```

#### 总结一句话（你写得很对）
线程池通过“策略接口” `RejectedExecutionHandler` 提供多种实现，本质是 **模板/策略模式**：你换不同实现类，就得到不同拒绝行为。

### 自定义拒绝策略（示例）+ 风险提醒
你给的自定义策略是：被拒绝时不丢不报错，而是 **阻塞等待队列腾出位置**（`put()` 会一直等）。

```java
ThreadPoolExecutor threadPoolExecutor = new ThreadPoolExecutor(
    availableProcessors,
    availableProcessorsMultipleResult,
    availableProcessorsAwaitTerminationSeconds, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(availableProcessorsQueueSize),
    new RejectedExecutionHandler() {
        @Override
        public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
            try {
                executor.getQueue().put(r); // 阻塞直到队列有空位
            } catch (InterruptedException ex) {
                Thread.currentThread().interrupt();
                throw new RejectedExecutionException(ex);
            }
        }
    }
);
```

这种做法的优缺点：
- **优点**：不丢任务，形成强背压（上游会被“卡住”）
- **风险**：如果“提交任务的线程”不能被阻塞（例如 Web 容器线程、Netty 事件循环线程、持锁线程），可能造成整体卡死/雪崩；因此一定要确认调用方线程模型，并做好超时与降级设计。

### 队列溢出时如何“保证不丢请求”（背压/落地/限流）
先说结论：**线程池本身不能凭空“保证不丢”**，因为资源（线程、内存、CPU、下游）是有限的；你能做的是在溢出时选择：
- **背压（让上游变慢）**
- **落地（把请求先存起来，后面慢慢处理）**
- **限流/降级（保护系统）**

#### 方式 1：CallerRunsPolicy（最常用的不丢方案）
队列满 + 线程满时，让“提交任务的线程”执行任务：
- **好处**：不丢任务，并且提交方会变慢 → 自动降速形成背压
- **注意**：如果提交方线程很关键（例如 Web 请求线程/事件循环线程），RT 会变大，要结合业务实时性评估

#### 方式 2：自定义拒绝策略，阻塞等待入队（强背压）
被拒绝时用 `queue.put()` 或 `offer(timeout)`：
- **好处**：尽量不丢任务
- **风险**：提交线程可能被长期阻塞，造成连锁阻塞；更建议用 **`offer + 超时`**，超时后走降级/快速失败/告警

#### 方式 3：可靠队列（MQ/DB/日志）先落地（严格不丢的正确姿势）
如果“不丢”是硬要求（例如订单/支付/扣库存）：
- **做法**：先把请求写入 **MQ/数据库/本地日志** 等可持久化介质，写成功就算“接收成功”，再异步消费处理
- **优点**：线程池再满也不等于丢请求（因为请求已落地）
- **配套**：需要幂等、重试、补偿等机制（你的笔记里已有“接口幂等性”可联动）

#### 方式 4：入口限流/降级（避免溢出才是根本）
与其在队列满了才处理，不如在进入线程池之前就控制：
- 限流（令牌桶/漏桶）
- 熔断降级（返回“系统繁忙”或兜底结果）
- 分级处理（高优先级任务进线程池，低优先级直接丢/延后）

> 实战建议：对“尽量不丢”的在线请求，优先用 **CallerRunsPolicy** 做背压；对“必须不丢”的关键请求，用 **可靠队列落地** 才真正稳。

### 自定义“严格不丢”的拒绝策略（推荐：先落地，再异步处理）
如果你说的“一定不丢”是 **订单/支付/扣库存** 这种级别，那么**不要只靠阻塞 put**，更稳妥的做法是：在拒绝时把任务内容 **可靠落地**（MQ / DB / 本地日志），落地成功就算“接收成功”，后续再补偿处理。

下面给一个“模板级”写法：拒绝时尝试把任务写进一个可靠存储（你们项目里可以对接 MQ/DB/本地文件），写成功就返回；写失败则抛异常并告警（因为这已经是“兜底也失败”）。

```java
import java.util.concurrent.RejectedExecutionException;
import java.util.concurrent.RejectedExecutionHandler;
import java.util.concurrent.ThreadPoolExecutor;

public final class PersistThenRejectPolicy implements RejectedExecutionHandler {

    public interface ReliableStore {
        /**
         * 把任务“可靠保存”起来（MQ/DB/本地日志等），成功返回 true。
         * 注意：这里保存的应该是“可重放的数据”，而不是直接保存 Runnable 对象本身。
         */
        boolean save(Object taskPayload);
    }

    private final ReliableStore store;
    private final TaskPayloadExtractor extractor;

    public interface TaskPayloadExtractor {
        Object toPayload(Runnable task);
    }

    public PersistThenRejectPolicy(ReliableStore store, TaskPayloadExtractor extractor) {
        this.store = store;
        this.extractor = extractor;
    }

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (executor.isShutdown()) {
            throw new RejectedExecutionException("executor is shutdown");
        }

        Object payload = extractor.toPayload(r);
        boolean ok = store.save(payload);
        if (!ok) {
            throw new RejectedExecutionException("task rejected and persist failed");
        }
        // 落地成功：不丢。后续由“补偿消费者/定时任务”把 payload 再投递回业务处理。
    }
}
```

你需要记住一个关键点：
- **不要试图把 `Runnable` 直接落地**（大概率不可序列化/不可重放）
- 应该落地的是：**订单号/请求ID/用户ID/业务参数** 这种“可重放的请求数据”
- 落地后再由“消费者/补偿任务”重新投递执行，并配合 **幂等** 保证重复执行不出错

> 如果只是“尽量不丢”（比如搜索/推荐/统计），通常 `CallerRunsPolicy` 或 `offer(timeout)` 背压就够了；只有“必须不丢”才需要上升到可靠落地。

---

### 误区澄清：只能靠持久化吗？拒绝策略会不会“把线程耗尽”？
#### 结论先说清楚
- **不是只能持久化**：要做到“不丢”，你还有 **背压**（让上游变慢/停）和 **失败重试**（端到端语义不丢）两条路。
- 但如果你要的是“**高峰期也要持续接收，并且单点就保证绝不丢**”，那就必须把压力转移到 **可持久化介质**（MQ/DB/日志），否则迟早会在 **线程/内存/下游** 任意一处被打爆。

#### 自带拒绝策略是不是都会阻塞线程？
不是。它们分三类：
- **直接失败（不阻塞）**：`AbortPolicy`（抛异常，让上游捕获后重试/降级/告警）
- **直接丢弃（不阻塞）**：`DiscardPolicy` / `DiscardOldestPolicy`（风险是“无声丢”或丢旧任务）
- **背压（占用提交线程执行任务）**：`CallerRunsPolicy`
  - 它不是“阻塞等待队列空位”，而是让提交方去跑任务，从而让提交方变慢 → 自动降速

真正容易把线程“长期占住”的，通常是你自己写的：
- `executor.getQueue().put(r)` 这种 **无限期阻塞等待入队**
- 或者用无界队列导致任务无限堆积（最终 OOM/GC 抖动）

#### 一个现实且稳妥的组合（建议你背下来）
- **在线请求（尽量不丢）**：`CallerRunsPolicy` 或 `offer(timeout)` 等一小会儿；超时就快速失败并提示“系统繁忙”
- **关键请求（必须不丢）**：拒绝时 **可靠落地（MQ/DB/日志）**，再异步消费；配合 **幂等**

---

### 队列怎么选：尽量使用有界队列（避免 OOM）
**注意：队列的选择上应该使用有界队列**，无界队列可能会导致任务无限堆积，最终把堆内存吃光引发 **OOM**。

常见选择：
- `LinkedBlockingQueue(capacity)`：**有界**链表队列（推荐显式传 `capacity`）
- `ArrayBlockingQueue(capacity)`：**有界**数组队列

#### “最大队列长度 = 虚拟机栈大小 / 线程栈大小”这句怎么理解（更准确的说法）
这句更像是在估算 **最大线程数**，而不是“最大队列长度”：
- **线程栈（Thread Stack）** 属于每个线程私有内存，线程越多，栈内存占用越大
- 近似理解：如果其它内存开销不考虑，最大线程数大概会受限于
  - **进程可用内存 / 每个线程栈大小（`-Xss`）**

而 **队列长度** 主要影响的是 **堆内存（Heap）**：
- 队列里堆的是 `Runnable` / 任务对象（以及它引用的参数、上下文数据）
- 队列越长，堆上对象越多 → 更容易内存飙升、GC 压力变大，甚至 OOM

所以更实用的结论是：
- **线程数**：受 CPU、外部资源（DB/下游）、线程栈、上下文切换影响
- **队列长度**：受堆内存、任务对象大小、可接受排队时延影响（实时性要求高时，队列不宜太大）

---

### 用“超时窗口”倒推线程数 + 队列大小（你给的例子整理版）
假设：
- 业务实时性较高
- 平均 RT = 200ms
- 请求超时时间 = 2000ms（也就是最多允许排队 + 执行不超过 2s）
- 机器 4 核 CPU、内存 16G
- 单机 QPS = 100

一种非常直观的估算思路是：用“**2 秒超时窗口内最多能处理多少请求**”来约束线程池的“在途请求上限”（线程数 + 队列可排队数）。

你的推算过程可以整理成：
- 4 核 CPU，先做一个很保守的近似：**每 200ms 能处理 4 个请求**
- 那么在 2000ms 内，最多能处理：
  - \(2000/200 \times 4 = 40\) 条请求
- 考虑到机器真实性能可能更强，可以“拍脑袋”多加一些余量（比如 +10）：
  - 2000ms 内预估能处理 **50** 条请求

对照流量：
- 单机 QPS=100 → 2 秒内会到达约 **200** 个请求
- 但你估算的 2 秒内最多处理 **50** 条 → 这说明 **不优化 RT 的话，需要扩容**（至少再加机器），否则必然有大量请求超时/排队爆炸

线程池设置示例（你给的配置）：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    15, 15, 365L, TimeUnit.DAYS,
    new LinkedBlockingQueue<>(35)
);
```

解释这个配置背后的“控制目标”：
- 最大线程数 = 15
- 队列最大 = 35
- **线程数 + 队列数 = 50**

这样在“2 秒超时窗口”内，系统最多允许 50 个请求处于“执行中或排队中”，尽量避免把请求无限堆积导致：
- 排队时间变长 → 超时更多
- 队列无限膨胀 → OOM 风险上升

> 这只是很粗糙的起点。真正工作中一定要结合压测：观察队列堆积、拒绝次数、TP99、CPU/GC、以及下游（尤其 DB）是否成为瓶颈，然后不断调参或改架构。

## 核心线程数（corePoolSize）怎么估算

> 前提：下面是“估算/经验公式”，不是死规定。最终要以压测数据为准。

### 方式 1：按业务类型的经验值（最常用）
假设：一台服务器只部署这一个应用，并且你只关心这一个线程池。

- **高并发、任务执行时间短（偏 CPU 计算）**
  - 最佳线程数 ≈ **CPU 总核数 + 1**
- **并发不高、任务执行时间长（通常是 IO 密集）**
  - 因为 IO 不吃满 CPU，为了不让 CPU 闲着，可以多开一些线程
  - 最佳线程数 ≈ **2 * CPU 总核数 + 1**（常见经验）
- **CPU/计算密集型应用**
  - 最佳线程数 ≈ **CPU 总核数 + 1**
- **并发高、业务执行时间长**
  - 关键往往不在“把线程池调大”，而在 **架构和链路优化**：
    - **缩短业务时间**：缓存、减少外部依赖耗时、拆分/解耦（消息队列等）
    - **提升整体并发能力**：水平扩容（加机器）
  - 线程池设置仍可参考上面 IO/CPU 密集的思路

### 方式 2：等待时间/计算时间公式（更贴近本质）
最佳线程数目：
\[
N \approx (\frac{W}{C} + 1)\times P \times U
\]

- \(W\)：线程等待时间（IO、锁等待、RPC 等）
- \(C\)：线程 CPU 运行时间（真正用 CPU 算的时间）
- \(P\)：CPU 核数（`availableProcessors()`）
- \(U\)：目标 CPU 利用率（0~1）

理解：
- **等待占比越高（W 越大）** → 需要更多线程来“填满等待空档”
- **计算占比越高（C 越大）** → 线程数应该更接近 CPU 核数


---

### 一句“最实用”的落地建议
- **先按业务类型选一个初值**（CPU 密集：\(P+1\)，IO 密集：\(2P+1\)）
- **再用压测把参数定下来**：重点观察 CPU 利用率、平均/TP99 延迟、队列堆积、拒绝策略触发次数
- **别忘了外部资源上限**：DB 连接池、下游限流、线程栈内存、GC 等

## 如何确认“服务器扛得住多少线程池配置”（别拍脑袋）

> 一句话：**“能创建多少线程”不等于“应该开多少线程”**。线上配置要同时满足：机器硬约束、下游约束、以及压测结果。

### 先看机器硬约束（上限感）
- **CPU 核数（P）**：`Runtime.getRuntime().availableProcessors()`
  - CPU 密集型线程池的线程数通常不会远大于 P（否则上下文切换浪费 CPU）。
- **内存 + 线程栈（-Xss）**：
  - 每个线程至少要占一份线程栈内存，线程越多，栈内存越大。
  - 粗略理解：最大线程数会受限于 \(可用内存 / Xss\)（这只是“上限感”，因为你还要给堆、元空间、缓存等留空间）。
- **OS/容器限制**：
  - 容器 CPU/内存限制会直接影响线程池有效能力（特别是 CPU 密集任务）。
  - 进程线程数/句柄数也可能成为上限（不同环境不一样，记住“要看限制”即可）。

### 再看下游硬约束（最常见的真实瓶颈）
- **数据库连接池**：
  - 如果任务执行期间会占用 DB 连接，那么线程池并发上限不要远大于 DB 连接池上限，否则多出来的线程只会卡在“等连接”上。
- **下游接口/网关限流**：
  - RPC/HTTP 下游能承受的并发往往更小；线程数开太大只会排队、超时更多。
- **网络/磁盘带宽**：
  - IO 等待越多，越不能靠“无脑加线程”解决，应该优先减少阻塞（异步 IO、批处理、缓存、降级）。

### 最可靠的确认方式：压测 + 监控信号（定配置的“证据”）
重点看 4 类信号（你后面监控章节都能采到）：
- **队列是否持续增长**：增长说明处理能力跟不上（要么优化任务，要么扩容/限流/降级）。
- **activeCount 是否长期接近 maximumPoolSize**：长期打满说明池饱和。
- **拒绝次数**：频繁拒绝说明容量不足/队列太小/拒绝策略不匹配。
- **TP95/TP99 延迟**：吞吐上去但尾延迟炸了，也是不合格配置。

## 多个线程池如何分配资源（线程数/队列/拒绝策略）

> 目标：**隔离**（别互相拖垮）+ **可控**（有界队列/明确拒绝）+ **可观测**（每个池单独监控）。

### 第 1 步：先按任务类型拆池（最常见 2～5 个）
- **biz（CPU 池）**：计算型、短任务、核心链路
- **io（IO 池）**：DB/RPC/文件等阻塞 IO
- **async（异步池）**：`@Async` 的非关键链路（通知、审计、统计）
- **schedule（定时池）**：周期任务/延迟任务
- （可选）**context（上下文池）**：强依赖 TraceId/MDC/用户上下文跨线程时，配 TTL/任务包装

### 第 2 步：给每个池做“线程预算”（先经验值，再压测修正）
- **CPU 池（biz）**：
  - 起步：\(P\) 或 \(P+1\)（P=CPU 核数）
  - 原因：CPU 计算越多，线程越接近 P 越有效，过多只会切换浪费。
- **IO 池（io）**：
  - 起步：通常会比 P 大（因为大量时间在等待 IO）。
  - 但必须受 **下游并发能力** 约束（尤其 DB 连接池、下游限流）。
- **异步池（async）**：
  - 建议“小而稳”：给非核心任务有限额度，避免它把机器资源吃光影响主流程。
- **定时池（schedule）**：
  - 通常不大，重点是每个任务要有超时/幂等，避免一次卡死带来堆积。

### 第 3 步：队列大小怎么分（比线程数更容易踩坑）
- **统一原则：有界队列**（无界队列最大风险是 OOM + 排队延迟无限变大）。
- **实时性强的任务**：
  - 队列不宜过大（宁可早拒绝/早降级，也不要排队排到超时再失败）。
- **允许延后处理的任务**：
  - 可以适当放大队列，但要结合任务对象大小与堆内存预算，避免堆积导致 GC/内存问题。

### 第 4 步：拒绝策略怎么分（看“能不能丢、能不能慢、能不能落地”）
- **核心链路（尽量不丢）**：
  - 常用 `CallerRunsPolicy` 做背压，或者快速失败 + 上层降级/重试/告警。
- **低价值任务（可丢/可降级）**：
  - 可以用丢弃类策略（或在业务层直接跳过），避免拖垮系统。
- **必须不丢（订单/支付等关键）**：
  - 正确姿势是“可靠落地（MQ/DB/日志）+ 异步消费 + 幂等”，而不是无限阻塞 put 把线程卡死。

### 第 5 步：防止“多个池互相抢资源”
- **隔离**：核心池与低价值池分开，避免排队饥饿。
- **背压**：让上游变慢（CallerRuns 或短超时入队），而不是无限堆积。
- **入口限流/降级**：比“队列满了才拒绝”更健康。
- **单池单指标**：每个线程池都单独采集队列长度、活跃线程、拒绝次数、耗时分位。

## 线程池监控（你真正需要盯的指标）

### 先说结论：监控的目标是什么
- **能提前发现“排队变长”**：队列堆积（`queueSize`）持续上升
- **能确认“线程是否打满”**：活跃线程数（`activeCount`）是否接近 `maximumPoolSize`
- **能判断“是不是处理变慢了”**：任务完成速率下降（`completedTaskCount` 增长变慢）
- **能定位“线程池是不是曾经满过”**：`largestPoolSize` 是否达到过 `maximumPoolSize`

### 线程池常用指标（直接可用）

```java
// 线程池中正在执行任务的线程数量
int activeCount = executor.getActiveCount();
// 线程池已完成的任务数量，该值小于等于 taskCount
long completedTaskCount = executor.getCompletedTaskCount();
// 线程池的核心线程数量
int corePoolSize = executor.getCorePoolSize();
// 线程池曾经创建过的最大线程数量（判断是否“满过”）
int largestPoolSize = executor.getLargestPoolSize();
// 线程池的最大线程数量
int maximumPoolSize = executor.getMaximumPoolSize();
// 线程池当前的线程数量
int poolSize = executor.getPoolSize();
// 线程池已经执行的和未执行的任务总数
long taskCount = executor.getTaskCount();
// 还剩多少个任务未执行（队列当前积压）
int queueSize = executor.getQueue().size();
```

你可以把它们理解成三类信号：
- **压力信号**：`queueSize`、`activeCount`
- **容量信号**：`corePoolSize`、`maximumPoolSize`、`poolSize`、`largestPoolSize`
- **产出信号**：`completedTaskCount`、`taskCount`

### 方式 1：定时采样日志（最简单、最常用）
适合：你只想快速看到“是否堆积/是否打满/吞吐是否下降”，不想改业务代码结构。

```java
import java.util.concurrent.ScheduledExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

public final class ThreadPoolMonitor {
    public static void start(String name, ThreadPoolExecutor executor, long period, TimeUnit unit) {
        ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "tp-monitor-" + name);
            t.setDaemon(true);
            return t;
        });

        scheduler.scheduleAtFixedRate(() -> {
            int activeCount = executor.getActiveCount();
            long completedTaskCount = executor.getCompletedTaskCount();
            int corePoolSize = executor.getCorePoolSize();
            int largestPoolSize = executor.getLargestPoolSize();
            int maximumPoolSize = executor.getMaximumPoolSize();
            int poolSize = executor.getPoolSize();
            long taskCount = executor.getTaskCount();
            int queueSize = executor.getQueue().size();

            System.out.printf(
                "tp=%s core=%d max=%d pool=%d active=%d largest=%d queue=%d task=%d done=%d%n",
                name, corePoolSize, maximumPoolSize, poolSize, activeCount, largestPoolSize,
                queueSize, taskCount, completedTaskCount
            );
        }, period, period, unit);
    }
}
```

常见用法（示例）：

```java
ThreadPoolExecutor executor = /* 你的线程池 */;
ThreadPoolMonitor.start("biz", executor, 5, TimeUnit.SECONDS);
```

### 方式 2：继承线程池 + 重写钩子（更精确：统计耗时/异常）
通过继承 `ThreadPoolExecutor` 并重写 `beforeExecute`、`afterExecute`、`terminated`，你可以在：
- **任务执行前**：记录开始时间、打点“当前队列长度/活跃线程”
- **任务执行后**：统计耗时、异常数量
- **线程池关闭前**：输出最终汇总

示例（统计任务耗时 + 异常数）：

```java
import java.util.concurrent.BlockingQueue;
import java.util.concurrent.RejectedExecutionHandler;
import java.util.concurrent.ThreadFactory;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.LongAdder;

public class MonitorableThreadPoolExecutor extends ThreadPoolExecutor {
    private final ThreadLocal<Long> startNs = new ThreadLocal<>();
    private final LongAdder errorCount = new LongAdder();
    private final LongAdder totalCostNs = new LongAdder();
    private final LongAdder finishedCount = new LongAdder();

    public MonitorableThreadPoolExecutor(
            int corePoolSize,
            int maximumPoolSize,
            long keepAliveTime,
            TimeUnit unit,
            BlockingQueue<Runnable> workQueue,
            ThreadFactory threadFactory,
            RejectedExecutionHandler handler
    ) {
        super(corePoolSize, maximumPoolSize, keepAliveTime, unit, workQueue, threadFactory, handler);
    }

    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        startNs.set(System.nanoTime());
        super.beforeExecute(t, r);
    }

    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        try {
            Long s = startNs.get();
            if (s != null) {
                totalCostNs.add(System.nanoTime() - s);
                finishedCount.increment();
            }
            if (t != null) {
                errorCount.increment();
            }
        } finally {
            startNs.remove();
            super.afterExecute(r, t);
        }
    }

    @Override
    protected void terminated() {
        long finished = finishedCount.sum();
        long avgMs = finished == 0 ? 0 : TimeUnit.NANOSECONDS.toMillis(totalCostNs.sum() / finished);
        System.out.printf("tp terminated: finished=%d avgMs=%d errors=%d%n",
                finished, avgMs, errorCount.sum());
        super.terminated();
    }
}
```

> 注意：`afterExecute` 里拿到的 `Throwable t` **不一定能覆盖所有异常场景**（比如任务内部捕获了异常）。如果你想做到“业务异常也统计”，需要统一在任务包装器里记录，或者规范任务必须把异常上抛。

### 怎么判断“告警阈值”（给你一套易懂的规则）
- **队列持续增长**：`queueSize` 连续多个采样周期上升（比如 1 分钟都在涨）→ 说明处理速度跟不上
- **线程打满**：`activeCount` 长期接近 `maximumPoolSize` 且 `queueSize` 也在涨 → 线程池已饱和
- **曾经打满过**：`largestPoolSize == maximumPoolSize` 说明历史上到达过上限（结合队列/拒绝次数看是否需要扩容或降级）
