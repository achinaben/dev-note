## 线程不安全的原因（从 JMM 角度理解）

### 一句话结论
- **线程不安全**本质上是：多个线程并发访问同一份数据时，**发生数据竞争（data race）**或**破坏了原子性/可见性/有序性**，导致结果不可预测。

### JMM 对“数据竞争”的定义
Java 内存模型规范对数据竞争的定义可以理解为同时满足：
- **一个线程写**某个变量
- **另一个线程读**同一个变量
- **读写没有通过同步来排序**（没有建立 happens-before 关系，比如没用 `synchronized` / `volatile` / `Lock` / 线程启动与 join 等规则）

换句话说：**读到的值不保证是“应该看到的值”**。

---

## JMM 的两层内存视角：主内存 vs 工作内存

### 主内存（线程共享）
- **堆（Heap）**
- **元空间（Metaspace）**

### 工作内存（线程私有）
每个线程都有自己的“工作内存/本地内存”副本，常见对应到 JVM 运行时区域可理解为：
- **虚拟机栈**
- **本地方法栈**
- **程序计数器**

### 为什么会出问题
共享变量在**主内存**中，但线程执行时会把它**复制到自己的工作内存**里使用/修改；修改后再**刷新回主内存**。  
这个“复制—修改—回写”的过程，在没有同步约束时，容易出现下面三类问题：**原子性、可见性、有序性**。

---

## 1）原子性问题：操作被拆成多步

### 基本类型的原子性边界（容易误解的点）
- 在 Java 中，**大多数基本类型的单次读/写**是原子操作：
  - `byte / short / int / float / boolean / char` 的读写是原子操作
- 但要注意历史/实现差异：
  - 在 **32 位 JVM** 中，`long / double` 是 **64 位**，可能被拆成 **两次 32 位**读写，从而出现“半个值被写入/读出”的风险（**非原子**）。
  - 在现代主流 JVM/平台上该问题很少成为实际坑点，但作为“原子性边界”的经典例子需要理解。

### 典型复合操作：`a++` 不是原子操作
`a++` 可以拆成三步（复合操作）：
- step1：从主内存读取最新的 `a`，在 CPU/工作内存保存副本  
- step2：对副本执行 `+1`
- step3：把结果写回（回写到工作内存/主内存）

#### 为什么会丢失更新（两个线程同时 `a++`）
假设 `a=0`，线程 A 和线程 B 同时执行 `a++`：
- A 读取到 0，B 也读取到 0
- A 计算得到 1 并写回
- B 也计算得到 1 并写回（覆盖 A 的结果）

最终 `a` 可能还是 1，而不是 2。  
这里的问题不是“看不见”，而是：**读-改-写这组动作不是一个不可分割的整体**。

### “先读再根据读到的结果修改”都不是原子
只要逻辑形如：
- 读取同一个共享值
- 基于这个值做判断/计算
- 再写回

在并发时就可能出现：多个线程基于**同一个旧值**计算，最后写回相互覆盖，导致结果错误。

---

## 2）可见性问题：一个线程改了，另一个线程看不到

### 问题来源
一个线程更新了共享变量，但更新可能还停留在它的工作内存/CPU 缓存里，**没有及时刷新到主内存**；其他线程就可能一直读到旧值。

### 直觉理解
- **你以为你改的是“共享变量”**  
- 实际上你先改的是**你这条线程的副本**，别人不一定立刻能看到。

---

## 3）有序性问题：你看到的执行顺序不一定是代码顺序

### 未同步程序的现象
- 在 JMM 中，**没有同步的程序**不仅“整体执行顺序”可能无序，甚至**不同线程看到的操作顺序也可能不一致**。

### 为什么会这样
一个线程把写过的数据缓存在本地（工作内存/缓存）里，尚未刷新到主内存之前：
- 这个写操作对当前线程可见
- 从其他线程角度看，可能认为这个写操作“还没发生”

只有当前线程把本地内存中写过的数据刷新到主内存之后，这个写操作才更可能对其他线程可见。  
因此：**线程 A 和线程 B 对同一段代码的“发生顺序”的观察可能不同**。

### 常见根因：指令重排序 + 流水线
- CPU 为了提高性能，会使用**流水线技术**并进行**指令重排序**（在不改变单线程语义的前提下调整执行顺序）。
- 编译器/JIT 也可能做重排序优化。
- 在没有同步约束时，这些优化会让并发下的“可见顺序”变得不可靠。

---

## 小结：为什么会线程不安全
- **原子性**：一个看似“1 行代码”的操作，可能被拆成多步并被其他线程插入执行（例如 `a++`）。
- **可见性**：线程对共享变量的更新可能只停留在本地副本，别的线程看不到最新值。
- **有序性**：编译器/CPU 重排序让并发下的观察顺序不一致，导致“先后关系”被打乱。

## 线程安全的要求（要同时满足什么）

### 1）原子性（Atomicity）
- **定义**：一个操作（或一组操作）必须是**不可中断**的。多个线程并发执行时，一旦开始，就不会被其他线程“插进来打断”，从而把中间状态暴露出去。

#### 原子操作 vs 非原子操作（例子）
- **原子操作示例**：对大多数基本类型做**单次赋值**通常是原子的，例如 `a = 0`（这里的 `a` 指 `int` 等，**不讨论 32 位 JVM 下 `long/double` 的历史差异**）。
- **非原子操作示例**：`a++` 看起来是一行，但本质是 `a = a + 1`（读→改→写），可分割，所以**不是原子操作**，并发下会“丢失更新”。

### 2）可见性（Visibility）
- **定义**：一个线程对共享变量做了修改，其他线程要能在“应该看到”的时机看到最新值。  
- **常见表述**：必须确保**释放锁之前**对共享数据做出的更改，对于**随后获得同一把锁**的另一个线程是可见的。

#### 常见实现手段（你需要记住的三个关键词）
- **`volatile`**：保证对该变量的读写具有一定的可见性（以及一定的有序性约束），但**不保证复合操作的原子性**（比如 `volatile int a; a++` 仍然不安全）。
- **`synchronized` / `Lock`**：通过互斥与 happens-before 关系，同时解决**可见性 + 原子性**（临界区内的复合操作也能安全）。
- **`final`**：对正确构造后的对象，`final` 字段在发布后对其他线程具有可见性保证（常用于“初始化安全发布”）。

### 3）有序性（Ordering）
- **定义**：程序执行要满足你依赖的“先后顺序”。在并发下，必须避免因为**指令重排序**导致其他线程看到的顺序与预期不一致。
- **直观要求**：对关键共享状态的读/写，需要用同步手段建立 happens-before，**从效果上禁止（或约束）重排序**对并发语义造成破坏。

## 正确同步与顺序一致性（JMM 的强保证）

### JMM 对“正确同步”的保证
- 如果程序是**正确同步**的，那么程序的执行将具有**顺序一致性**（sequentially consistent, **SC**）。
- 也就是说：程序的执行结果，与它在“顺序一致性内存模型”中的执行结果**相同**。  
  这对程序员来说是一个**非常强**的保证：你可以按“大家看到的是同一个顺序、而且按代码顺序发生”的方式来理解程序行为。

### 这里的“同步”是广义的
不仅仅是 `synchronized`，还包括对常用同步原语的**正确使用**：
- **锁（`synchronized` / `Lock`）**
- **`volatile`**
- **`final`**

> 关键点是“正确使用”：要让它们在你依赖的读写之间建立 happens-before 关系。

### 顺序一致性（SC）内存模型的两大特性
- **线程内有序**：一个线程中的所有操作，必须按照**程序顺序**来执行。
- **全局单一顺序**：所有线程只能看到**同一个**操作执行顺序。  
  在 SC 内存模型中，每个操作都必须**原子执行**，并且**立刻对所有线程可见**。

### 直觉图解（“开关把并发串行化”）
可以把 SC 想象成一个“开关/闸门”：
- 在任意时间点，只有**一个线程**能“连接到内存”进行读/写。
- 当多个线程并发执行时，这个开关会把所有线程的所有内存读/写操作**串行化**，因此所有线程看到的都是**同一个顺序**。
